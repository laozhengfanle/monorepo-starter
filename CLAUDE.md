# CLAUDE.md

## 项目性质：企业级基座（Enterprise Foundation）

这是一个 **企业级基座项目**，目标是把重复的基础设施一次写好，后续项目直接在其上构建。

核心理念：**不重复造轮子，只造一次底座。**

- 认证、鉴权、RBAC、安全防护、错误码、日志、限流、缓存、文件上传 — 这些每个项目都要的东西，在基座里一次到位
- 新建项目时，fork 这个基座，直接开始写业务代码，而不是从零搭架子
- 因此代码追求**通用性、可复用性、文档完备性**——写的时候想清楚别人怎么用

同时这是一个 **0→1 的绿色项目（Greenfield）**，没有历史代码包袱，没有存量数据迁移负担。所有技术选型追求"当前最新"而非"历史兼容"。

## 核心原则：尝鲜优先（Latest-First）

**所有库都用最新稳定版，所有写法都用最新 API。**

因为你没有历史负担，不需要考虑：

- 旧版 API 兼容性
- 存量数据迁移
- 遗留代码重构成本

所以每次见到一个库，**先去了解它的最新版本和最新写法，再写代码**。不要：

- 凭记忆写旧版 API（比如 `@nestjs/throttler` v4 的 `forRoot({ ttl, limit })` 单对象写法）
- 用被废弃的装饰器或配置格式
- 复制粘贴过时的教程代码

**要求**：在引入或使用任何 npm 包之前，先查它的最新版本、changelog、当前推荐 API。如果版本号你不确定，问一句或搜一下。

## 临时文件管理

**谁产生，谁清理；统一收纳，做完就删。**

### 核心规则

1. **统一目录**：所有临时文件（测试脚本、py 工具、SQL 片段、临时截图、curl/fetch 调试产物等）**必须**放在仓库根的 `.tmp/` 目录下，**禁止**散落到 `/tmp/`、项目子目录、`apps/*/src/` 等任何位置
    - 目录名固定为 `.tmp/`（不要用 `.temp/`、`.cache/`、`tmp/` 等其他名字——散落各处的临时文件是最难找的债务）
    - `.tmp/` 已在 `.gitignore` 里，不会被 commit
2. **谁产生，谁清理**：产生临时文件的当次任务**结束前必须删除**该目录里的全部内容
3. **清理时机**：清理动作必须在告知用户"完成"之前执行——「完成」消息附带"已清理 .tmp/" 才算齐活
4. **没清理 = 没完成**：如果还有临时文件残留，整个任务**不算完成**

### 常见临时文件类型

- 截图文件（`.png`、`.jpg`）
- 测试产物（`test-results/`、`screenshots/`、`coverage/`）
- **临时脚本**（`.py`、临时 `.ts`/`.js` 调试脚本、`.sql`、`curl` 抓包片段）—— 全部放 `.tmp/`
- 编译中间文件（`.map`、`.d.ts` 等，除非是项目正常构建产物）
- 日志文件（`.log`）

### 错误示范（被骂过）

- ❌ 把 `test_crud.py` 放到 `/tmp/`、`.verify_icons.ts` 放到 monorepo 根、Prisma 中间产物放到 `apps/server/` —— **散落 6+ 个位置**让清理变成体力活
- ❌ 完成后用 `ls` 一条条找 + 删，浪费时间还容易漏
- ❌ 「我以为不归我管」—— 所有临时文件都归当前任务管，**不归环境管**

### 正确做法

- ✅ 一开始就 `mkdir -p .tmp/` 把临时文件全部往里放
- ✅ 任务结束前 `rm -rf .tmp/*`（保留目录本身，方便下次复用）
- ✅ 在告诉用户"完成"前，先 `ls .tmp/` 确认是空的

**理由**（用户原话，2026-06-17）：

> "所有的测试文件，包括你们写的 py 工具什么的，不要到处乱放，都统一放到 .tmp 中，做完就删除。这个思想同步到 CLAUDE.md 中。"

把临时文件管好，是 0→1 项目最容易忽略、最容易累积债务的卫生习惯——比写代码还重要。

## 不做国际化（i18n）

**这个项目不做 i18n，永远不要引入 vue-i18n / i18next / 任何多语言库。**

- 业务定位：内部管理后台 + 国内 B 端产品，**唯一目标语言是简体中文**
- i18n 带来的成本（字典维护、键补全、占位符、复数处理、回退语言）与本项目**完全不成比例**
- 任何"未来可能需要多语言"的想法都是 YAGNI——等真正有海外业务再说，**到那时再重构也来得及**（且届时 i18n 库本身的 API 会更稳定）

**禁止事项**：

- ❌ 不要 `pnpm add vue-i18n` / `i18next` / `react-intl` / 任何 i18n 库
- ❌ 不要在 shared 包里建 `i18n/` 目录
- ❌ 不要写 `$t()` / `useI18n()` / `t()` / `<i18n-t>` / 任何翻译调用
- ❌ 不要做语言切换器（AppSettings 里不要语言卡片、Header 里不要语言下拉、登录页不要语言选择）
- ❌ 不要把组件文案抽到 `zh-CN.json` / `en-US.ts` / 任何字典文件
- ❌ 不要在 main.ts 注册 vue-i18n（不要再 `app.use(i18n)`）
- ❌ 不要做"键对齐"测试（"中英文必须 key 完全对齐"这种回归守卫也不要）

**正确做法**：

- ✅ 组件文案直接写中文：`{{ '保存' }}` / `<h1>企业级开发底座</h1>`
- ✅ 用户提示、按钮 label、菜单项、表单 placeholder 一律硬编码中文
- ✅ 任何时候用户要求"加个英文版"——先反问业务场景，**默认拒绝**
- ✅ 看到 PR / Issue / 旧代码里有 `$t()` / `useI18n()` / `messages: { 'zh-CN': ... }` —— **直接删掉**，改回硬编码

**理由**（用户原话，2026-06-16）：

> "整个网站不需要国际化，把国际化的东西都去掉，为了避免后面的ai再添加国家化，请把我的想法放到 CLAUDE.md 中。"

如果未来真的有海外业务，**那时候**再重新评估 i18n 方案（可能是 next-intl、可能是 LinguiJS、可能直接用 ChatGPT 翻译后端模板），不要在当下做 YAGNI 的"提前投资"。

## 项目结构

pnpm workspace monorepo：

- `apps/server/` — NestJS 统一 API（GraphQL + REST）
- `apps/admin/` — 管理后台 Vue 3 SPA
- `apps/web/` — C端/B端 Vue 3 前端
- `packages/` — 共享配置和工具
- `docs/` — 消费者技术文档（fork 后阅读）
- `开发文档/` — 内部开发文档（设计决策依据，详见 [开发文档/项目总览.md](./开发文档/项目总览.md)）
