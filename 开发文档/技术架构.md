## 后端分层架构

```text
请求
  │
  ▼
┌─────────────────┐
│  Middleware      │  CORS、日志、限流
├─────────────────┤
│  Guard           │  JWT 验证 → 权限校验（按 userType 分流）
├─────────────────┤
│  Pipe            │  Zod 校验 + 类型转换（ZodValidationPipe）
├─────────────────┤
│  Controller /    │  路由 + 参数解析（REST）
│  Resolver        │  Query / Mutation（GraphQL）
├─────────────────┤
│  Service         │  业务逻辑 + 事务管理
├─────────────────┤
│  Prisma          │  数据库访问
└─────────────────┘
```

## 权限校验链路

```text
JwtAuthGuard       → 解析 token，注入 request.user（accountId + userType）
    │
    ▼
AdminPermissionGuard   → 校验 @Permission('iam:admin:delete')
MemberPermissionGuard  → 校验 C端等级权限
GuestPermissionGuard   → 游客无需 JWT，直接读 guest 角色缓存
```

Guard 使用 NestJS 的 `APP_GUARD` 全局注册，通过 `Reflector` 读取 `@Permission()` 装饰器元数据。

## 缓存架构

两级缓存，角色级吸收重复查询，账户级保证热路径只需 1 次 Redis GET。

```text
mono:role:permission:{userType}:{roleCode}   → SET    角色权限列表
mono:role:menus:{userType}:{roleCode}  → JSON   角色菜单（扁平，不拼树）
mono:auth:{accountId}                  → JSON   计算结果（roles + permissions + 拼好的菜单树）
```

角色变更删角色级 key + pipeline 批量删相关账户 key。菜单结构变更删角色级全量 + 缩短账户 TTL 防雪崩。详细设计见 [权限控制.md](./权限控制.md)。

## 数据库

PostgreSQL 16 + Prisma，自动迁移。

### 主键策略：UUID v7

所有表的 `id` 统一使用 **UUID v7**（`UUID DEFAULT uuidv7()`）。

**为什么选 UUID v7：**

| 维度       | 说明                                                                                     |
| ---------- | ---------------------------------------------------------------------------------------- |
| 时序性     | 前 48 位是毫秒时间戳，`ORDER BY id DESC` 等价于 `ORDER BY created_at DESC`，不需额外索引 |
| 分布式安全 | 不会像自增 ID 冲突，多实例部署零改动                                                     |
| 不可推测   | API 返回 `id` 不会暴露数据量（`/user/42` → 不存在 → `/user/43`），防爬虫/遍历            |
| 标准化     | RFC 9562，跨语言兼容（PostgreSQL、JS、Go 都有原生支持）                                  |
| 存储紧凑   | PostgreSQL `UUID` 类型只占 16 bytes（CUID2 存成 TEXT 要 ~24 bytes）                      |

**不选其他方案的原因：**

| 方案        | 为什么不选                                           |
| ----------- | ---------------------------------------------------- |
| BIGINT 自增 | 可推测、分布式冲突、依赖数据库                       |
| UUID v4     | 随机无序，索引写入退化（页分裂）                     |
| CUID2       | 非标准、PostgreSQL 只能存 TEXT、字符串索引比二进制慢 |

**应用层生成，不依赖数据库扩展：**

```ts
// lib/uuid.ts
import { uuidv7 } from 'uuidv7';
export const newId = () => uuidv7();

// Prisma Client Extensions — create 时自动注入
const prisma = new PrismaClient().$extends({
    query: {
        $allModels: {
            async create({ args, query }) {
                args.data.id ??= newId();
                return query(args);
            },
        },
    },
});
```

好处：迁移干净，测试/CI 环境同样可用，不依赖 PostgreSQL 扩展（`pg_uuidv7`）。

### 表分组

核心表 18 张，分六组：

- **认证层**（4 张）：account、account_identity、admin_profile、member_profile
- **管理端 RBAC**（5 张）：admin_role、admin_menu、admin_account_role、admin_role_menu、admin_account_menu

## 双协议运维规范（REST + GraphQL）

项目同时使用 REST 和 GraphQL 两种协议，以下是运维时需要注意的规范。

### 新增端点决策树

```text
新增功能时，按以下决策树选择协议：

该功能是认证/文件上传？
  ├── 是 → REST（/auth/*, /upload/*）
  └── 否 → 该功能是数据 CRUD？
            ├── 是 → GraphQL（/graphql）
            └── 否 → 该功能是健康检查/监控？
                      ├── 是 → REST（/health, /metrics）
                      └── 否 → 默认走 GraphQL
```

### 双协议注意事项

| 关注点         | REST                                           | GraphQL                              | 说明                                       |
| -------------- | ---------------------------------------------- | ------------------------------------ | ------------------------------------------ |
| 认证           | `@UseGuards(JwtAuthGuard)`                     | `@UseGuards(JwtAuthGuard)`           | Guard 代码共享，context 从 `req.user` 获取 |
| 权限           | `@Permission('iam:admin:view')`                | `@Permission('iam:admin:view')`      | 装饰器完全一致                             |
| 参数验证       | `ZodValidationPipe` + `@Body()`                | `ZodArgsPipe` (PipeTransform)        | 底层同一套 Zod schema                      |
| 错误处理       | `GlobalExceptionFilter`（统一 REST + GraphQL） | 同一过滤器                           | `host.getType()` 区分上下文，错误码统一    |
| 限流           | `ThrottlerGuard`                               | `ThrottlerGuard` + 深度/复杂度/超时  | GraphQL 需额外三重限制                     |
| CORS           | `app.enableCors()`                             | 同一实例，共享 CORS 配置             | 无需单独配置                               |
| 原型链污染防护 | `sanitizeObject` 中间件                        | `bodyParserConfig: false` 共享中间件 | GraphQL 必须禁用 Apollo 自带 parser        |

> **关键原则**：优先走 GraphQL。REST 仅用于认证、文件上传、健康检查等非数据操作场景。新增 REST 端点需要团队 review 确认必要性。

## 实时通信扩展说明

当前基座不包含 WebSocket / SSE 实时通信能力。如需扩展（如管理端通知推送、C端消息提醒），推荐方案：

- **SSE（Server-Sent Events）**：服务端单向推送（通知、告警），实现简单，NestJS 原生支持 `@Sse()` 装饰器
- **WebSocket（Socket.IO）**：双向通信（聊天、协作），需要额外安装 `@nestjs/websockets` 和 `socket.io`

实时通信不在当前基座范围内，但架构已预留扩展点：

- `apps/server/src/modules/` 可新增 `notification` 模块
- JWT 认证可复用现有 `JwtAuthGuard` 的 handshake 验证
- Redis Pub/Sub 可用于多实例间消息广播
