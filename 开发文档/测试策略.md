# 测试策略

> 企业级基座对可靠性要求高——基础设施的 bug 会被所有 fork 项目继承。测试分三层：单元测试（Vitest）、集成测试（supertest）、端到端测试（Playwright）。Vitest 是 NestJS v12 的默认测试框架，也是本项目唯一使用的测试框架。

## 测试分层

```text
           ┌─────────┐
           │   E2E   │  Playwright（前端关键流程）
           │   10%    │
           ├─────────┤
           │Integration│  supertest（API 端点 + 缓存失效链路）
           │   25%    │
           ├─────────┤
           │  Cache  │  Vitest（缓存命中/未命中/击穿/雪崩）
           │   15%    │
           ├─────────┤
           │   Unit  │  Vitest（Service / Util / Guard / Pipe）
           │   50%    │
           └─────────┘
```

| 层           | 框架                           | 范围                                            |     覆盖率目标     |
| ------------ | ------------------------------ | ----------------------------------------------- | :----------------: |
| **单元测试** | Vitest（后端）/ Vitest（前端） | Service、Util、Guard、Pipe、Composable          |       ≥ 80%        |
| **缓存测试** | Vitest                         | 缓存读写、懒构建、失效逻辑、穿透/击穿/雪崩防护  | 关键路径 100% 覆盖 |
| **集成测试** | supertest                      | API 端点：REST 和 GraphQL，含缓存失效端到端验证 | 关键端点 100% 覆盖 |
| **E2E**      | Playwright                     | 核心用户流程（登录→操作→登出）                  |    核心流程全覆    |

## 二、集成测试（supertest）

```ts
// apps/server/test/auth.e2e-spec.ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { Test, TestingModule } from '@nestjs/testing';
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';
import { AppModule } from '../src/app.module';

describe('Auth (e2e)', () => {
  let app: INestApplication;

  beforeAll(async () => {
    const moduleFixture: TestingModule = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();

    app = moduleFixture.createNestApplication();
    await app.init();
  });

  afterAll(async () => {
    await app.close();
  });

  describe('POST /admin/auth/login', () => {
    it('should return tokens on valid credentials', async () => {
      const res = await request(app.getHttpServer())
        .post('/admin/auth/login')
        .send({ username: 'root', password: 'Root!123' })
        .expect(200);

      expect(res.body.code).toBe(0);
      expect(res.body.data.accessToken).toBeDefined();
      expect(res.body.data.refreshToken).toBeDefined();
    });

    it('should return 401 on wrong password', async () => {
      const res = await request(app.getHttpServer())
        .post('/admin/auth/login')
        .send({ username: 'root', password: 'WrongPass1' })
        .expect(200); // 业务错误仍返回 200，通过 code 区分

      expect(res.body.code).toBe(20003); // 认证失败
    });

    it('should reject missing fields', async () => {
      const res = await request(app.getHttpServer())
        .post('/admin/auth/login')
        .send({ username: 'root' }) // 缺少 password
        .expect(400); // Zod 校验失败

      expect(res.body.code).toBe(10001);
    });
  });

  describe('POST /admin/auth/refresh', () => {
    it('should issue new tokens with valid refresh token', async () => {
      const loginRes = await request(app.getHttpServer())
        .post('/admin/auth/login')
        .send({ username: 'root', password: 'Root!123' });

      const cookies = loginRes.headers['set-cookie'];

      const res = await request(app.getHttpServer())
        .post('/admin/auth/refresh')
        .set('Cookie', cookies)
        .expect(200);

      expect(res.body.data.accessToken).toBeDefined();
    });

    it('should detect token reuse and revoke all tokens', async () => {
      // 第一次 refresh
      const cookies1 = /* ... */;
      await request(app.getHttpServer())
        .post('/admin/auth/refresh')
        .set('Cookie', cookies1)
        .expect(200);

      // 用同一个 token 再次 refresh → 触发 reuse detection
      await request(app.getHttpServer())
        .post('/admin/auth/refresh')
        .set('Cookie', cookies1)
        .expect(401); // Token compromised
    });
  });
});
```

## 四、缓存层测试

权限缓存是本项目最复杂的组件之一（两级缓存 + 懒构建 + 雪崩防护），必须有独立的测试策略。

### 测试范围

| 场景                                               | 测试类型 | 说明                                                |
| -------------------------------------------------- | :------: | --------------------------------------------------- |
| 缓存命中：`mono:auth:{accountId}` 存在             |   单元   | 直接返回缓存，不查 DB                               |
| 缓存未命中 → 懒构建                                |   单元   | DB 查角色/菜单 → 写角色级缓存 → 合并 → 写账户级缓存 |
| 角色级缓存命中：`mono:role:permission:{code}` 已有 |   单元   | 不重复查 DB，直接用                                 |
| 角色级缓存未命中 → 回源 DB                         |   单元   | 查 DB → 写角色级缓存                                |
| 账户额外权限（grant/deny）覆盖                     |   单元   | `aggregatePermissions()` 纯函数的测试               |
| 角色变更 → 缓存失效                                |   集成   | 改角色菜单 → auth key 被删，下次请求查最新 DB       |
| 菜单结构变更 → 角色级缓存全删 + auth TTL 缩短      |   集成   | `invalidateMenuCache()` 逻辑                        |
| 缓存穿透：不存在 ID → `__NULL__`                   |   单元   | 空值缓存 TTL=60s                                    |
| 缓存击穿：热点 key 过期 → 互斥锁                   |   单元   | `getCachedWithLock()` 双重检查                      |
| 缓存雪崩：批量过期 → TTL jitter                    |   单元   | `ttlWithJitter(300)` 分布在 270-330s                |
| Refresh token reuse detection                      |   集成   | 同一 token 两次 refresh → 第二次触发全家失效        |

### 纯函数测试示例

```ts
// apps/server/src/modules/auth/__tests__/aggregate-permissions.spec.ts
import { describe, it, expect } from 'vitest';
import { aggregatePermissions } from '../rbac.utils';

describe('aggregatePermissions', () => {
    const mockRoles = [
        {
            roleMenus: [
                { menu: { permissionCode: 'iam:admin:view' } },
                { menu: { permissionCode: 'iam:admin:create' } },
            ],
        },
    ];

    it('should collect permissions from roles', () => {
        const result = aggregatePermissions(mockRoles, []);
        expect(result).toEqual(['iam:admin:view', 'iam:admin:create']);
    });

    it('should add grant overrides', () => {
        const overrides = [{ menu: { permissionCode: 'iam:admin:delete' }, type: 'grant' as const }];
        const result = aggregatePermissions(mockRoles, overrides);
        expect(result).toContain('iam:admin:delete');
    });

    it('should remove deny overrides', () => {
        const overrides = [{ menu: { permissionCode: 'iam:admin:create' }, type: 'deny' as const }];
        const result = aggregatePermissions(mockRoles, overrides);
        expect(result).not.toContain('iam:admin:create');
    });

    it('should deduplicate permissions', () => {
        const duplicateRoles = [
            { roleMenus: [{ menu: { permissionCode: 'iam:admin:view' } }] },
            { roleMenus: [{ menu: { permissionCode: 'iam:admin:view' } }] },
        ];
        const result = aggregatePermissions(duplicateRoles, []);
        expect(result).toEqual(['iam:admin:view']);
    });
});
```

### E2E 场景详细清单

以下为基座必须覆盖的 E2E 场景，fork 后按业务扩展。

#### 认证流程

| #   | 场景                 | 验证点                                                                     |
| --- | -------------------- | -------------------------------------------------------------------------- |
| A1  | 管理员用户名密码登录 | 登录成功 → 跳转 dashboard → localStorage 无 token → cookie 有 refreshToken |
| A2  | C端手机验证码登录    | 输入手机号 → 收到验证码 → 登录成功 → 跳转首页                              |

## 八、测试数据管理

### 测试数据库

- 每个测试 suite 开始时用 `prisma migrate deploy`（不是 `migrate dev`）
- 测试用独立数据库 `mono_test`，不与开发数据库混用
- 集成测试前 seed 基础数据（超管账号、角色、权限），测试后清理
