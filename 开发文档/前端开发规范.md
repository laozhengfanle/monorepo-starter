# 前端开发规范

> admin 和 web 两个 Vue 3 前端（≥ 3.5，`defineModel` 在 3.5 正式稳定），共享 packages/shared（类型定义、Zod schemas）。本文定义统一的目录结构、组件规范、状态管理和 API 调用约定。

## 目录结构（Feature-First）

> 完整目录树 + 目录规则见 [目录结构.md](./目录结构.md)（单一权威源）。此处只列核心原则。

### 核心原则

| 原则                              | 说明                                                                                                                   |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **按业务域分组**                  | `features/{domain}/` 包含该域的所有代码（组件、store、API、类型）                                                      |
| **Feature 之间禁止互引**          | `features/users/` 不能 import `features/roles/` 的东西。通过 ESLint `no-restricted-imports` 规则强制执行（见下方配置） |
| **共享代码放到 `shared/`**        | 两个以上 feature 用到的组件、composable、工具函数                                                                      |
| **每个 feature 有清晰的公共接口** | 通过 `index.ts` 只导出 `useXxx()` composable 和 types                                                                  |

### Feature 互引检查（ESLint Flat Config）


### admin 和 web 的差异

| 维度           | admin                                 | web                         |
| -------------- | ------------------------------------- | --------------------------- |
| GraphQL 客户端 | urql（`@urql/vue`）                   | graphql-request             |
| 布局           | DashboardLayout（侧边栏+顶栏）        | DefaultLayout               |
| pages          | 表格/表单型（管理员管理、角色管理）   | 展示型（首页、产品列表）    |
| 权限控制       | 按钮级（`v-if="hasPermission(...)"`） | 等级级（VIP/SVIP 内容区别） |

### 组件拆分原则

| 规则                  | 说明                      |
| --------------------- | ------------------------- |
| `<script>` ≤ 150 行   | 超过 → 拆 composable      |

## 状态管理（Pinia）

### Store 规范

### Store 组织策略：Feature-First

Store 按 Feature 维度组织，而非全局按类型组织。每个 Feature 目录下有自己的 store 文件：

```text
src/features/
├── auth/
│   └── store.ts          # useAuthStore — 登录/登出/刷新 token
├── admins/
│   └── store.ts          # useAccountStore — 管理员列表/搜索/分页
└── roles/
    └── store.ts          # useRoleStore — 角色列表/权限分配
```

**命名规范**：`use{Feature}Store`，如 `useAuthStore`、`useUserStore`

**跨 Feature 调用**：Feature Store 之间可以相互调用（如 `useUserStore` 中调用 `useAuthStore`），但应保持单向依赖，避免循环引用。

**与全局 Store 的区别**：传统方案将所有 store 放在 `src/stores/` 全局目录下。Feature-First 方案将 store 放在对应的 Feature 目录中，好处是：

1. 删除 Feature 时，相关 store 一并删除，无残留
2. 修改 Feature 时，所有相关文件在同一个目录下
3. 与后端模块化结构对应（auth → auth, admins → admin-account）

```ts
// features/auth/store.ts
import { defineStore } from 'pinia';
import { ref, computed } from 'vue';
import type { UserInfo } from './types';

export const useAuthStore = defineStore('auth', () => {
    // ——— State ———
    const accessToken = ref<string | null>(null);
    const userInfo = ref<UserInfo | null>(null);
    const permissions = ref<Set<string>>(new Set());

    // ——— Getters ———
    const isLoggedIn = computed(() => !!accessToken.value);
    const hasPermission = (code: string) => permissions.value.has(code);

    // ——— Actions ———
    async function login(credentials: LoginInput) {
        const res = await loginApi(credentials);
        accessToken.value = res.accessToken;
        userInfo.value = res.user;
        permissions.value = new Set(res.permissions);
    }

    function logout() {
        accessToken.value = null;
        userInfo.value = null;
        permissions.value = new Set();
    }

    return { accessToken, userInfo, permissions, isLoggedIn, hasPermission, login, logout };
});
```

### Store 设计规则

| 规则                           | 说明                                               |
| ------------------------------ | -------------------------------------------------- |
| Store 名与文件名一致           | `useAuthStore` ← `auth/store.ts`                   |
| **不把 UI 逻辑放 Store**       | Store 只有 state + getters + actions，不含模板逻辑 |
| Feature store 放 feature 目录  | `features/auth/store.ts`                           |
| 全局 store 放 `shared/stores/` | 如 `theme`、`appConfig`                            |
| 大数据用 `shallowRef`          | 避免深度响应式的性能开销                           |

### Store 持久化

```ts
// pinia-plugin-persistedstate v4+ 配置
import { createPersistedState } from 'pinia-plugin-persistedstate';

const pinia = createPinia();
pinia.use(
    createPersistedState({
        storage: sessionStorage,
        // 只持久化用户偏好，不持久化任何 token
        // pick 指定要持久化的 store 中的 key
        pick: ['theme', 'locale', 'sidebarCollapsed'],
    }),
);
```

> **Token 存储策略**（与 [安全防护.md](./安全防护.md) 保持一致）：
>
> | Token         | 存储位置                                                  | 生命周期    | XSS 可窃取？      | 刷新丢？ |
> | ------------- | --------------------------------------------------------- | ----------- | ----------------- | -------- |
> | Access Token  | `httpOnly + Secure + SameSite=Lax` cookie（`path=/`）     | 15 分钟过期 | 不能（JS 不可读） | 不丢     |
> | Refresh Token | `httpOnly + Secure + SameSite=Lax` cookie（`path=/auth`） | 7 天过期    | 不能（JS 不可读） | 不丢     |
>
> **方案对比**：
>
> | 方案                        | 优点                       | 缺点                                      |
> | --------------------------- | -------------------------- | ----------------------------------------- |
> | **localStorage**            | 永久持久化，关浏览器也不丢 | XSS 可窃取，关闭浏览器后 token 残留       |
> | **sessionStorage**          | 刷新不丢，关标签页自动清   | XSS 可读取，窃取后有效期 15 分钟          |
> | **纯内存（Pinia ref）**     | XSS 窃取难度最高           | 页面刷新即丢失，必须依赖 refresh 接口恢复 |
> | **httpOnly cookie（当前）** | XSS 完全免疫               | 需 CSRF 防护（待加），前端无法手动读取    |
>
> **关于 httpOnly 与持久化的关系**：
>
> **httpOnly 只控制 JS 能不能读，不控制存多久。** Cookie 持久化由 `Max-Age` / `Expires` 决定：
>
> - `accessToken` cookie：`Max-Age=900`（15 分钟），在有期限内刷新页面不丢
> - `refreshToken` cookie：`Max-Age=604800`（7 天）
> - 纯内存变量（Pinia ref）才刷新即丢——跟 httpOnly 无关
>
> **当前选择全 httpOnly cookie 的理由**：
>
> - 双 cookie 均由后端 `Set-Cookie` 下发，前端不接触任何 token
> - urql 客户端设 `credentials: 'include'`，浏览器自动在每次请求中携带 cookie
> - `accessToken` cookie `path=/`，覆盖所有 API 请求；`refreshToken` cookie `path=/auth`，仅 refresh 端点可用
> - 前端 authStore 不再存 token，仅存 `userInfo` + `permissions`，以 `userInfo` 是否存在判断登录态
> - **页面刷新流程**：路由守卫先直接调 `me` 查询（cookie 仍有效）→ 只有 `me` 返回 20003（token 过期）时才调 `/auth/refresh` → 重试 `me`。正常刷新只发 1 次 GraphQL 请求，不触发 refresh 限流
> - **CSRF 防护**：`SameSite=Lax` 阻止跨站 POST 携带 cookie；GraphQL 只接受 POST（`preferGetMethod: false`），GET 无法执行 mutation
> - 登出 → 后端 `clearCookie` 清除双 cookie + Redis 清理

### 路由守卫

```ts
// app/router/guards.ts
router.beforeEach(async (to, from, next) => {
    const authStore = useAuthStore();

    // 需要登录但未登录 → 跳登录页
    if (to.meta.requiresAuth && !authStore.isLoggedIn) {
        next({ name: 'Login', query: { redirect: to.fullPath } });
        return;
    }

    // 已登录去登录页 → 跳首页
    if (to.name === 'Login' && authStore.isLoggedIn) {
        next({ name: 'Dashboard' });
        return;
    }

    // 权限检查
    if (to.meta.permissions && !to.meta.permissions.every(authStore.hasPermission)) {
        next({ name: '403' });
        return;
    }

    next();
});
```

```vue
<!-- features/users/composables/useUsers.ts -->
<script setup lang="ts">
import { useQuery } from '@urql/vue';

const USERS_QUERY = `
  query Users($page: Int!, $pageSize: Int!, $filter: UserFilter) {
    users(page: $page, pageSize: $pageSize, filter: $filter) {
      items { id realName email status createdAt }
      total page pageSize hasMore
    }
  }
`;

export function useUsers(filter: Ref<UserFilter>) {
    const page = ref(1);
    const variables = computed(() => ({
        page: page.value,
        pageSize: 20,
        filter: filter.value,
    }));

    const { data, error, isFetching } = useQuery({
        query: USERS_QUERY,
        variables,
    });

    const users = computed(() => data.value?.users?.items ?? []);
    const total = computed(() => data.value?.users?.total ?? 0);

    return { users, total, page, error, isFetching };
}
</script>
```

## GraphQL 错误处理

### 全局错误拦截

urql 和 graphql-request 都需要统一的全局错误处理，避免每个页面重复写 try-catch。

```ts
// api/error-handler.ts — admin 端（urql）
import { CombinedError } from '@urql/core';

/** 解析 GraphQL errors，返回业务错误码和字段级错误 */
export function parseGraphQLError(error: CombinedError) {
    const graphqlErrors = error.graphQLErrors;
    if (graphqlErrors.length > 0) {
        const gqlErr = graphqlErrors[0];
        return {
            code: (gqlErr.extensions?.code as string) || '10999',
            message: gqlErr.message,
            fields: gqlErr.extensions?.fields as Array<{ field: string; message: string }> | null,
        };
    }
    // 网络错误
    if (error.networkError) {
        return { code: 'NETWORK_ERROR', message: '网络请求失败，请检查连接' };
    }
    return { code: '10999', message: '未知错误' };
}
```

```ts
// api/error-handler.ts — web 端（graphql-request）
import { ClientError } from 'graphql-request';

export function handleRequestError(error: unknown): { code: string; message: string } {
    if (error instanceof ClientError) {
        const gqlErr = error.response.errors?.[0];
        return {
            code: (gqlErr?.extensions?.code as string) || '10999',
            message: gqlErr?.message || '请求失败',
        };
    }
    return { code: 'NETWORK_ERROR', message: '网络请求失败，请检查连接' };
}
```

### 错误码 → 用户行为映射

```ts
// api/error-actions.ts
import { useRouter } from 'vue-router';
import { useAuthStore } from '@/features/auth/store';

/** 根据业务错误码执行全局行为（跳转登录、显示无权限等） */
export function handleErrorByCode(code: string, router: ReturnType<typeof useRouter>) {
    const authStore = useAuthStore();

    switch (code) {
        case '20003': // token 过期 / 未认证
            authStore.logout();
            router.push({ name: 'Login', query: { redirect: router.currentRoute.value.fullPath } });
            break;
        case '22001': // 无权限
            router.push({ name: '403' });
            break;
        case '10001': // 参数验证失败 — 表单已自己处理，这里只 toast
            break;
        default:
            // 其他错误由调用方自行处理
            break;
    }
}
```

### urql errorExchange（admin 端推荐）

利用 urql 的 `errorExchange` 在 pipe 层统一拦截：

```ts
import { errorExchange } from '@urql/core';
import { parseGraphQLError } from './error-handler';

// 在 createClient 的 exchanges 中使用
export const urqlClient = createClient({
    url: import.meta.env.VITE_GRAPHQL_ENDPOINT || '/graphql',
    exchanges: [
        cacheExchange,
        errorExchange({
            onError: (error) => {
                const { code } = parseGraphQLError(error);
                if (code === '20003' || code === '22001') {
                    // 全局处理：跳登录 or 跳403
                    // 注意：errorExchange 中无法直接访问 router，可通过 event bus 或 window dispatch
                    window.dispatchEvent(new CustomEvent('auth:error', { detail: { code } }));
                }
            },
        }),
        fetchExchange,
    ],
});
```

### 使用示例

```vue
<script setup lang="ts">
import { useGqlMutation } from '@/composables/useGraphql';
import { CREATE_USER } from '@/graphql/mutations/user';
import { useMessage } from 'naive-ui';

const message = useMessage();

// ⚠️ Vue composable 必须在 <script setup> 顶层调用，不能在事件处理函数中调用
const { execute, error } = useGqlMutation(CREATE_USER);

async function handleCreate(input: CreateUserInput) {
    await execute({ input });

    if (error.value) {
        const { code, message: errMsg, fields } = parseGraphQLError(error.value);

        if (code === '10001' && fields) {
            // 字段级错误 → 显示在表单对应字段下方
            fields.forEach((f) => {
                formErrors.value[f.field] = f.message;
            });
        } else {
            // 通用错误 → toast
            message.error(errMsg);
        }
    } else {
        message.success('创建成功');
    }
}
</script>
```

> **原则**：20003（未认证）和 22001（无权限）由全局拦截器统一处理，其他业务错误码由各页面/组件自行决定 UI 反馈方式。

## 样式规范

### Tailwind CSS（主要样式方案）

布局、间距、颜色、响应式断点等优先使用 Tailwind CSS utility classes。Naive UI 负责复杂交互组件（Table、Modal、Form 等），Tailwind 负责所有布局和样式细节。

```vue
<template>
    <!-- ✅ Tailwind 做布局和样式 -->
    <div class="max-w-4xl mx-auto px-4 py-8">
        <h1 class="text-2xl font-semibold text-gray-900 mb-6">管理员管理</h1>
        <n-data-table :columns="columns" :data="users" :loading="loading" />
    </div>
</template>
```

### Tailwind Config 主题色

```js
// tailwind.config.js
export default {
    theme: {
        extend: {
            colors: {
                primary: '#18a058',
                danger: '#d03050',
                warning: '#f0a020',
            },
        },
    },
};
```

### 规则

| 规则              | 说明                                                           |
| ----------------- | -------------------------------------------------------------- |
| Tailwind 优先     | 布局、间距、颜色、响应式用 Tailwind utility classes            |
| Naive UI 优先     | 复杂交互组件用 Naive UI，不自己写 Button/Modal/Table           |
| `scoped` 必须     | `<style scoped>` 避免样式泄露（仅在 Tailwind 无法覆盖时使用）  |
| 不用内联样式      | 除非动态计算（`<div :style="{ height: `${rows \* 40}px` }">`） |
| 不用 `!important` | 优先通过选择器特异性解决                                       |
| 不手写 CSS        | 除非 Tailwind 无法表达的场景                                   |

### web 端权限控制：等级级

与 admin 的按钮级权限（`v-if="hasPermission('iam:admin:delete')"`）不同，web 端使用**等级级权限**：根据用户会员等级（普通/VIP/SVIP）控制内容可见性。

```vue
<!-- shared/components/VipGuard.vue — VIP 内容守卫 -->
<script setup lang="ts">
import { useAuth } from '@/shared/composables/useAuth';

const props = defineProps<{
    /** 需要的最低等级：normal / vip / svip */
    requiredLevel?: 'normal' | 'vip' | 'svip';
}>();

const { userInfo } = useAuth();

/** 等级优先级映射 */
const levelPriority = { normal: 0, vip: 1, svip: 2 } as const;

/** 当前用户等级是否满足要求 */
const hasAccess = computed(() => {
    if (!props.requiredLevel) return true;
    const userLevel = userInfo.value?.level || 'normal';
    return levelPriority[userLevel] >= levelPriority[props.requiredLevel];
});
</script>

<template>
    <slot v-if="hasAccess" />
    <slot v-else name="fallback">
        <div class="text-center py-8 text-gray-400">
            <p>该内容仅限 {{ requiredLevel?.toUpperCase() }} 会员查看</p>
            <n-button type="primary" size="small" class="mt-2">升级会员</n-button>
        </div>
    </slot>
</template>
```

### web 端 graphql-request 封装

web 端使用 `graphql-request`（轻量，无内置缓存），需自行封装错误拦截：

```ts
// shared/composables/useGql.ts
import { GraphQLClient } from 'graphql-request';
import { handleRequestError } from '@/api/error-handler';
import { handleErrorByCode } from '@/api/error-actions';

/** 创建带错误拦截的 GraphQL 客户端 */
const client = new GraphQLClient(import.meta.env.VITE_GRAPHQL_ENDPOINT || '/graphql', {
    credentials: 'include',
    headers: () => ({
        'X-Request-ID': crypto.randomUUID(),
    }),
});

/**
 * 封装 graphql-request 的请求方法
 * - 自动拦截全局错误（401 → 跳登录，403 → 跳无权限）
 * - 业务错误返回给调用方自行处理
 */
export function useGql() {
    const router = useRouter();

    async function query<T>(document: string, variables?: Record<string, unknown>): Promise<T> {
        try {
            return await client.request<T>(document, variables);
        } catch (error) {
            const { code, message } = handleRequestError(error);

            // 全局错误码：跳转登录 / 403
            if (code === '20003' || code === '22001') {
                handleErrorByCode(code, router);
                throw error; // 仍然抛出，让调用方知道请求失败了
            }

            // 其他业务错误：抛出结构化错误，由调用方决定 UI 反馈
            throw { code, message };
        }
    }

    return { query };
}
```

```text
用户访问 web 端
  │
  ├─ 点击"手机号登录"
  │   ├─ POST /member/auth/sms/send  → 发送验证码
  │   ├─ POST /member/auth/sms/login → 验证码登录

```

## 前端错误处理闭环

> 后端定义了完整的错误码体系（[错误码.md](./错误码.md)），前端必须建立对应的消费机制，才能形成"后端抛码 → 前端拦截 → 用户感知"的完整闭环。

### 错误处理架构

```text
后端返回错误
  │
  ▼
前端 API 层（graphql-request / urql）
  │
  ├─ 网络错误（无响应 / 超时 / CORS）
  │   └─ → toast "网络请求失败，请检查连接"
  │
  ├─ GraphQL errors（有响应，但包含 errors 数组）
  │   │
  │   ├─ 全局错误码（20003 / 22001 / 22002）
  │   │   ├─ 20003（未认证）→ 清除本地状态 → 跳转登录页
  │   │   ├─ 22001（无权限）→ 跳转 403 页面
  │   │   └─ 22002（账号被禁用）→ 清除本地状态 → 跳转登录页 + 提示
  │   │
  │   ├─ 参数验证错误（10001）
  │   │   └─ 提取 fields → 显示在表单对应字段下方
  │   │
  │   └─ 其他业务错误码
  │       └─ 返回给调用方 → 由页面/组件决定 UI 反馈（toast / dialog / inline）
  │
  └─ REST 错误（HTTP 状态码非 200）
      ├─ 400 → Zod 校验失败
      ├─ 429 → 限流 → toast "操作过于频繁，请稍后再试"
      ├─ 413 → 请求体过大
      └─ 5xx → 服务端异常 → toast "服务器开小差了"
```

### 错误码 → 前端行为映射表

| 错误码 | 含义         | 前端全局行为                                   | 调用方处理 |
| ------ | ------------ | ---------------------------------------------- | ---------- |
| 10001  | 参数验证失败 | 无（表单字段级处理）                           | 可选       |
| 10002  | 资源不存在   | 无                                             | 可选       |
| 20003  | 未认证/过期  | 清除 authStore → 跳登录页（带 redirect）       | 不需要     |
| 22001  | 无权限       | 跳 403 页面                                    | 不需要     |
| 22002  | 账号被禁用   | 清除 authStore → 跳登录页 + 提示"账号已被禁用" | 不需要     |
| 20001  | 用户名密码错 | 无                                             | toast      |
| 30001  | 短信发送频繁 | 无                                             | toast      |
| 30002  | 每日发送超限 | 无                                             | toast      |
| 40001  | 限流触发     | 无                                             | toast      |
| 10999  | 未知业务错误 | 无                                             | toast      |

### 全局错误拦截器实现

#### admin 端（urql errorExchange）

```ts
// api/client.ts
import { createClient, cacheExchange, fetchExchange, errorExchange } from '@urql/vue';
import { parseGraphQLError } from './error-handler';

export const urqlClient = createClient({
    url: import.meta.env.VITE_GRAPHQL_ENDPOINT || '/graphql',
    fetchOptions: () => ({
        credentials: 'include',
        headers: { 'X-Request-ID': crypto.randomUUID() },
    }),
    exchanges: [
        errorExchange({
            onError: (error) => {
                const { code } = parseGraphQLError(error);
                // 通过自定义事件通知全局处理器
                // （errorExchange 中无法直接访问 router / store）
                window.dispatchEvent(new CustomEvent('app:error', { detail: { code } }));
            },
        }),
        cacheExchange,
        fetchExchange,
    ],
});
```

#### 全局错误事件监听

```ts
// App.vue — 监听全局错误事件（admin 和 web 通用）
<script setup lang="ts">
import { useRouter } from 'vue-router';
import { useAuthStore } from '@/features/auth/store';

const router = useRouter();
const authStore = useAuthStore();

window.addEventListener('app:error', ((event: CustomEvent) => {
    const { code } = event.detail;

    switch (code) {
        case '20003': // 未认证 / Token 过期
            authStore.logout();
            router.push({ name: 'Login', query: { redirect: router.currentRoute.value.fullPath } });
            break;
        case '22001': // 无权限
            router.push({ name: '403' });
            break;
        case '22002': // 账号被禁用
            authStore.logout();
            router.push({ name: 'Login' });
            // 使用 nextTick 确保 DOM 更新后再显示提示
            nextTick(() => {
                window.dispatchEvent(new CustomEvent('app:toast', {
                    detail: { type: 'error', message: '账号已被禁用，请联系管理员' },
                }));
            });
            break;
    }
}) as EventListener);
</script>
```

### 表单字段级错误处理

```vue
<script setup lang="ts">
import { parseGraphQLError } from '@/api/error-handler';

const formErrors = ref<Record<string, string>>({});

async function handleSubmit() {
    formErrors.value = {}; // 清除旧错误

    try {
        await execute({ input: formValue });
        message.success('操作成功');
    } catch (error) {
        const { code, fields } = parseGraphQLError(error as any);

        if (code === '10001' && fields) {
            // 参数验证失败 → 显示在表单对应字段下方
            fields.forEach((f) => {
                formErrors.value[f.field] = f.message;
            });
        } else if (!['20003', '22001', '22002'].includes(code)) {
            // 非全局错误 → toast 提示
            message.error(parseGraphQLError(error as any).message);
        }
    }
}
</script>

<template>
    <n-form-item :validation-status="formErrors.email ? 'error' : undefined" :feedback="formErrors.email">
        <n-input v-model:value="form.email" placeholder="邮箱" />
    </n-form-item>
</template>
```

### 错误处理原则

| 原则                                   | 说明                                                         |
| -------------------------------------- | ------------------------------------------------------------ |
| **全局错误全局处理，业务错误局部处理** | 20003/22001/22002 由拦截器统一处理，其他错误码由页面自行决定 |
| **不吞错误**                           | 全局拦截后仍 throw，让调用方知道请求失败了                   |
| **表单错误走字段级**                   | 10001 + fields → 显示在对应字段下方，不用 toast              |
| **网络错误友好提示**                   | 断网 / 超时 → "网络请求失败，请检查连接"，不是技术性报错     |
| **错误码与用户提示分离**               | 错误码给开发者看，用户提示文案集中管理（可后续 i18n 化）     |

#### 大列表虚拟滚动

超过 100 条的列表必须使用虚拟滚动，禁止全量渲染：

```ts
// main.ts — 上报 Web Vitals 到后端
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

function reportMetric(metric: { name: string; value: number; rating: string }) {
    // 使用 sendBeacon 确保页面卸载时也能上报
    const body = JSON.stringify({
        name: metric.name,
        value: Math.round(metric.value),
        rating: metric.rating,
        page: window.location.pathname,
        timestamp: Date.now(),
    });

    if (navigator.sendBeacon) {
        navigator.sendBeacon('/api/vitals', body);
    } else {
        fetch('/api/vitals', { body, method: 'POST', keepalive: true });
    }
}

onLCP(reportMetric);
onINP(reportMetric);
onCLS(reportMetric);
onFCP(reportMetric);
onTTFB(reportMetric);
```
