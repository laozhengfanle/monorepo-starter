# 命名规范

> 本项目的命名约定。统一规则，减少决策成本。

---

## 一、数据库

### 表名

```text
规则：小写 + 下划线（snake_case），完整单词，禁止缩写

认证层：  account、account_identity
档案层：  admin_profile、member_profile
管理端 RBAC：admin_role、admin_menu、admin_account_role、admin_role_menu、admin_account_menu
C端 RBAC：  member_role、member_menu、member_account_role、member_role_menu、member_account_menu
系统配置：system_config、verification_code、audit_log、upload_file

前缀约定：admin_ / member_ 标识归属端
```

**禁止**：表名缩写。如 `sys_config` 应写作 `system_config`。`sys` 不是行业通用标识符（不像 `id`、`ip`、`url` 那样展开后反而无人认识），省掉四个字母换来的是一次额外的决策成本——团队成员每次起表名都要想"这个能不能缩"。统一用全称，零歧义。

### 列名

```text
规则：小写 + 下划线（snake_case）

主键：  id（UUID v7）
外键：  {table}_id（如 account_id、role_id、menu_id）
自引用：parent_id
时间：  created_at、updated_at、deleted_at、last_login_at
审计：  created_by、updated_by
布尔：  visible、keep_alive、enabled、locked（BOOLEAN，true/false）
状态：  status（VARCHAR + CHECK 约束，如 CHECK (status IN ('active','disabled','locked'))）
```

**禁止**：驼峰（camelCase）、大写、复数列名、`is_` 前缀（用 `visible` 而非 `is_visible`）。

> **布尔 vs 状态**：`visible`、`enabled` 是二值开关；`status` 是多值枚举，两者语义不同，不要混用。
> 不要用 `status` 表示布尔（如 `status = 1/0` 表示启用/禁用），当状态可能扩展为多种时用 `status`。
> **多个布尔列**：每个布尔列用语义化名称描述其控制的含义，而不是编号或加前缀。例如 `admin_menu` 表同时有 `visible`（控制显示）、`keep_alive`（控制缓存）、`enabled`（控制启用），三个列名互不冲突，一看就懂。

### 软删除

```text
字段：deleted_at（DateTime, nullable）
策略：需要软删除的表才加此字段，不是所有表都必须
查询：Prisma 层通过 middleware 统一过滤 deleted_at IS NULL
恢复：将 deleted_at 置为 NULL 即可恢复
永久删除：定时任务清理 deleted_at 超过 N 天的记录
```

### 索引

```text
UNIQUE：  uq_{table}_{columns}（如 uq_account_identity_type_identifier）
普通索引：idx_{table}_{columns}（如 idx_admin_profile_account_id、idx_account_created_at）
```

> **原则**：索引名必须带表名前缀，避免多表间索引名冲突。联合索引用下划线连接列名。

### 枚举存储

```text
规则：数据库存储枚举值的小写字符串，禁止存数字

Prisma 映射：
enum UserType {
  ADMIN  = 'admin'
  MEMBER = 'member'
}

数据库列：user_type VARCHAR 存储 'admin'、'member'
```

> **禁止存数字**（0、1、2），可读性差，排查问题时无法直接理解含义。

### Prisma 映射

```prisma
model AdminProfile {
  id        String   @id @db.Uuid  // UUID v7 由应用层 Client Extension 注入，不依赖 DB 扩展

  // 列名用 @map 映射到 snake_case
  accountId String   @map("account_id") @db.Uuid
  createdAt DateTime @default(now()) @map("created_at")
  deletedAt DateTime? @map("deleted_at")

  @@map("admin_profile")
}
```

原则：**Prisma 模型层用 camelCase，数据库层用 snake_case**。

## 二、后端（NestJS）

### 文件

```text
规则：kebab-case.{type}.ts

controller：  admin-auth.controller.ts
service：     auth.service.ts
dto：         auth.dto.ts
module：      auth.module.ts
guard：       jwt-auth.guard.ts
decorator：   permission.decorator.ts
filter：      http-exception.filter.ts
strategy：    jwt.strategy.ts
interface：   jwt-payload.interface.ts
```

### 类名

```text
规则：PascalCase + 后缀

Controller： AdminAuthController
Service：    AuthService
Module：     AuthModule
Guard：      JwtAuthGuard、AdminPermissionGuard
DTO：        LoginDto、CreateUserDto
VO：         UserVo、RoleVo
Interface：  JwtPayload
```

### DTO / VO 命名

```text
规则：按场景区分入参和出参

CreateXxxDto    创建请求
UpdateXxxDto    更新请求（Zod .partial() 或自定义）
QueryXxxDto     列表查询（分页、筛选条件）
XxxVo           响应视图（不暴露内部字段，如 passwordHash）
```

```typescript
// 入参：Dto
class CreateUserDto {
    username: string;
    roleIds?: string[];
}

// 出参：Vo
class UserVo {
    id: string;
    username: string;
    roles: RoleVo[];
}
```

> **原则**：Dto 是入参，Vo 是出参。禁止直接返回 Prisma Model，必须通过 Vo 过滤敏感字段。
> Update 用 `CreateXxxSchema.partial()` 复用 Create 的校验规则，避免重复定义相同字段。

### GraphQL 类型命名

REST 和 GraphQL 的命名空间区分：

| 场景                    | 命名规则                                   | 示例                                                 |
| ----------------------- | ------------------------------------------ | ---------------------------------------------------- |
| REST 入参               | `{Action}{Domain}Dto`                      | `CreateUserDto`、`UpdateUserDto`                     |
| REST 出参               | `{Domain}Vo`                               | `UserVo`、`RoleVo`                                   |
| GraphQL InputType       | `{Action}{Domain}Input`                    | `CreateAdminAccountInput`、`UpdateAdminAccountInput` |
| GraphQL ObjectType      | `{Domain}` 或 `{Domain}Type`               | `AdminAccount`、`AdminRole`                          |
| GraphQL 分页 ObjectType | `{Domain}Connection` / `Paginated{Domain}` | `AdminAccountConnection` / `PaginatedAdminAccount`   |

> **原则**：GraphQL InputType 对应 REST 的 Dto，ObjectType 对应 REST 的 Vo。命名上 InputType 用 `Input` 后缀，ObjectType 用纯粹名词，避免和 Dto/Vo 混用。

### 路由

```text
规则：复数名词 + 嵌套资源，kebab-case

GET    /admin/accounts          管理员列表
POST   /admin/accounts          新增管理员
PUT    /admin/accounts/:id      编辑管理员
DELETE /admin/accounts/:id      删除管理员
PUT    /admin/accounts/:id/roles    分配角色

GET    /admin/roles              角色列表
PUT    /admin/roles/:id/menus    分配菜单

GET    /admin/menus/tree         菜单树
```

**禁止**：单数名词（`/admin/user`）、驼峰（`/adminRole`）、关联表名暴露（`/admin/role-menus`）。

### API 响应格式

统一使用 `{ code, message, data }` 结构：

- `code`: 业务错误码（`0` 表示成功，非 0 为错误）
- `message`: 用户可读的消息
- `data`: 响应数据（错误时为 null）

完整响应体示例（成功、分页、错误）和错误码规范见 [错误码.md](./错误码.md)。REST 和 GraphQL 共用同一套错误码。

## 三、前端（Vue 3）

### 文件

```text
规则：页面用 kebab-case，组件用 PascalCase

页面：     features/iam/admins/LoginPage.vue
布局：     layouts/AdminLayout.vue
组件：     components/layout/SideMenu.vue
composable：composables/usePermission.ts
store：    features/{domain}/store.ts
路由：     router/index.ts、router/guard.ts
类型：     features/{domain}/types.ts
```

### Vue 组件名

```text
规则：PascalCase，多词（页面级加 Page 后缀）

页面组件（Page 后缀）：  LoginPage.vue、RoleListPage.vue
功能组件（必须多词）：SideMenu.vue、RoleForm.vue、PermissionTree.vue
```

### composable

```text
规则：use + PascalCase

usePermission、useTurnstile、useTable
```

### store

```text
规则：camelCase，Feature-First 目录，文件名即 store 名

features/auth/store.ts      → useAuthStore()
features/users/store.ts      → useUserStore()
features/roles/store.ts      → useRoleStore()
features/dashboard/store.ts  → useDashboardStore()
```

> **原则**：Store 放在对应 feature 目录下，与组件、composable、API 共处。命名格式：use{Domain}Store，按业务域区分而非技术模块。

### 常量与工具

```text
常量：  constants/ 下，按领域分文件（env.ts、app.ts），值 UPPER_SNAKE
类型：  types/ 下，按模块分文件（types/iam/account.d.ts）
工具：  utils/ 下，按功能分文件（format.ts、request.ts），导出函数 camelCase
```

## 四、权限标识

```text
规则：小写 + 冒号分隔，三级命名

iam:admin:view
iam:admin:create
iam:admin:update
iam:admin:delete

iam:role:view
iam:role:update

product:goods:view
product:goods:export

member:course:vip
member:analysis:svip
```

**禁止**：中文、空格、数字、超过三级。

> **层级含义**：`模块:资源:操作`，操作词统一用 list / create / update / delete / export / view / import。

## 五、Redis Key

```text
规则：mono:{domain}:{entity}:{identifier}

段说明：
  mono       项目前缀，防多项目 key 冲突
  domain     功能域（auth、role、lock、rate、data）
  entity     实体类型（perm、menus、login、sms）
  identifier 唯一标识（accountId、roleCode、phone 等）
```

> **原则**：所有 key 必须加项目前缀 `mono:`。key 中的变量用 camelCase，与代码层一致。
> 完整 Key 清单（含 TTL、限流 key、业务缓存 key）见 [缓存设计.md](./缓存设计.md)。

#### 禁止缩写

Redis Key 的 entity 段、identifier 段都禁止缩写（与 §一表名规则一致）：

- ✅ `mono:role:permission:*`
- ❌ `mono:role:perm:*`（`perm` 是 `permission` 的缩写）
- ✅ `mono:auth:*`（`auth` 是行业通用缩写，见 §一例外条款）
- ❌ `mono:au:*`（项目内部缩写）

行业通用缩写（auth / req / res / id / ip / url / http / api / ttl 等）放行（与 §一原文「不像 id、ip、url 那样展开反而无人认识」原则一致）；其他缩写必须写完整单词。

## 六、Git

### 分支

```text
main                生产分支
feat/{name}         功能分支
fix/{name}          修复分支
refactor/{name}     重构分支（不新增功能、不修复 bug 的代码优化）
docs/{name}         文档分支
hotfix/{name}       紧急修复（从 main 拉出，修完合回 main 和开发分支）
release/{version}   发布准备（如 release/1.0.0）
```

### Commit

```text
规则：conventional commits

feat: 新功能
fix: 修复
refactor: 重构（不改变外部行为）
docs: 文档
chore: 杂项（依赖、配置）
style: 格式调整（不影响逻辑）
perf: 性能优化
test: 测试相关
ci: CI/CD 配置
```

> **scope 可选**：`feat(iam): 新增管理员管理页面`、`fix(auth): 修复 token 过期未跳转`

## 七、TypeScript 通用

| 场景       | 规范                           | 示例                         |
| ---------- | ------------------------------ | ---------------------------- |
| 变量/函数  | camelCase                      | `getUserById()`              |
| 类/接口    | PascalCase                     | `UserService`、`JwtPayload`  |
| 常量       | UPPER_SNAKE                    | `DEFAULT_HOME`、`JWT_SECRET` |
| 类型文件   | `{name}.d.ts`                  | `vite-env.d.ts`              |
| 枚举       | PascalCase，值小写字符串       | `UserType.ADMIN = 'admin'`   |
| 未使用参数 | `_` 前缀                       | `(_req, res)`                |
| 泛型       | 大写单字母或 PascalCase        | `T`、`TResponse`、`TEntity`  |
| 事件处理   | `on` + PascalCase              | `onClick`、`onSubmit`        |
| 布尔变量   | is/has/can/should + PascalCase | `isLoading`、`hasPermission` |

> **注意**：TypeScript 代码层的布尔变量用 `is_` 语义（如 `isLoading`），但**数据库列名**禁止 `is_` 前缀（用 `visible`），两层规则不同。

## 八、依赖安装

```text
规则：不手动指定版本号，让 pnpm 自动确定最新版本范围
```

```bash
# ✅ 正确 — pnpm add 自动获取最新 stable 并写入版本范围
pnpm add zod
pnpm add -D typescript

# ❌ 错误 — 手动锁定旧版本，会导致依赖陈旧
pnpm add zod@^3.24.0
pnpm add -D typescript@~6.0.2
```

> **原则**：始终使用 `pnpm add`（不手动指定版本号），让 pnpm 自动确定最新稳定版的范围并写入 `package.json`。`pnpm-lock.yaml` 会锁定具体版本保证可复现构建。公共依赖的版本范围由 `pnpm-workspace.yaml` 中的 catalog 统一管理（详见 [monorepo管理规范.md](./monorepo管理规范.md)）。
> 如果某次升级导致问题，在 pnpm overrides 中临时锁定，附 issue 链接和移除日期。
