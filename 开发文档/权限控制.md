# 一、权限系统核心思想

系统权限基于：

# RBAC（Role-Based Access Control）

即：

```text
账户 -> 角色 -> 菜单权限
```

同时支持：

```text
账户 -> 额外菜单权限（授权/禁止）
```

用于特殊授权和禁止。

# 5、admin_account_menu（管理端 账户额外菜单权限）

用于特殊授权和禁止。

```sql
CREATE TABLE admin_account_menu (
  id UUID PRIMARY KEY,  -- UUID v7 由应用层生成（Prisma Client Extension 注入）
  account_id UUID NOT NULL,
  menu_id UUID NOT NULL,
  type VARCHAR(10) DEFAULT 'grant' CHECK (type IN ('grant', 'deny')),
  UNIQUE(account_id, menu_id)
);
```

说明：

- type='grant'（授权）：角色没有的权限，可以额外授予
- type='deny'（禁止）：角色有的权限，可以额外禁止（如某角色有删除权限，但某个人不应该有）
- 聚合权限时：先合并角色权限，再应用账户级覆盖（授权追加，禁止移除）

# 6、member_role（C端角色/等级表）

```sql
CREATE TABLE member_role (
  id UUID PRIMARY KEY,  -- UUID v7 由应用层生成（Prisma Client Extension 注入）
  name VARCHAR(50) NOT NULL,
  code VARCHAR(50) NOT NULL UNIQUE,
  description VARCHAR(255) DEFAULT '',
  enabled BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  created_by UUID DEFAULT NULL,
  updated_by UUID DEFAULT NULL,
  deleted_at TIMESTAMPTZ
);

```

说明：

- C端的角色本质上是"等级"，不同等级拥有不同的功能权限
- 内置等级：guest（游客）→ normal（普通用户）→ vip → svip
- 游客（guest）是特殊角色：未登录用户自动应用 guest 角色的权限，不需要在 account 表中创建记录
- 如：普通用户只能看基础功能，VIP 可以看专属内容，SVIP 可以使用高级功能
- 与管理端角色表结构一致，只是语义上是"等级"
- `created_by` / `updated_by` 对种子数据（guest / normal / vip / svip）无意义，预置等级此字段为 NULL

# 16、数据库索引设计

完整索引策略见 [数据库设计.md](./数据库设计.md)。关键点：

- **account_identity.account_id** — 最高频，每次 `me` 查询都要查用户绑定的登录方式
- **各 menu 表的 parent_id** — 菜单树查询核心
- **关联表 UNIQUE 约束** — `UNIQUE(account_id, role_id)` 覆盖 `WHERE account_id=?` 查询（account*id 是复合索引前缀列）；但 `WHERE role_id=?` 查询（如"哪些用户拥有此角色"）无法利用该索引，需单独加 `idx*{table}\_role_id` 索引

# 五、权限标识规范

统一使用三级命名：

```text
模块:子模块:行为
```

例如：

```text
iam:admin:view
iam:admin:create
iam:admin:update
iam:admin:delete

iam:role:view

product:goods:view
product:goods:export

member:course:vip

```

说明：

- 三级命名避免模块冲突（如 product:goods:view vs product:category:view）

- member = C端功能权限前缀

禁止：

- 中文
- 空格
- 数字权限

# 2、menus 数据结构

数据库同时存储 `routeName` 和 `component`：

- `routeName`：前端路由名称（唯一标识），用于路由注册和菜单高亮匹配
- `component`：前端组件名（如 `UserListPage`），用于动态路由 `componentMap` 映射

设计取舍：

- `component` 存 DB 的好处：运营新增菜单时直接在后台选择组件名，无需前端代码修改；`routeRegistry` 从硬编码变为由后端菜单数据驱动的 `componentMap`
- `routeName` 仍然保留：用于路由跳转（`router.push({ name: routeName })`）和 `activeMenuId` 高亮匹配

```json
[
    {
```

# 七、前端动态路由实现

# componentMap — 前端维护的组件名 → 懒加载映射表

后端菜单的 `component` 字段存储组件名（如 `UserListPage`），前端维护一个 `componentMap` 做懒加载映射：

```typescript
// 前端维护 component name → 懒加载 Vue 组件的映射表
export const componentMap: Record<string, () => Promise<Component>> = {
    Layout: () => import('@/app/layouts/MainLayout.vue'),
    ProductListPage: () => import('@/features/product/ProductListPage.vue'),
    ProductDetailPage: () => import('@/features/product/ProductDetailPage.vue'),
    UserListPage: () => import('@/features/users/UserListPage.vue'),
    RoleListPage: () => import('@/features/roles/RoleListPage.vue'),
};

// 运营新增菜单时，在后端管理后台选择 component（从 componentMap 的 key 列表中下拉选取）
// 前端新增页面时，在 componentMap 中添加一行 import，同时在 DB 中创建对应菜单记录
```

# 路由守卫设计

刷新页面时 Pinia 状态会丢失，需要路由守卫恢复动态路由：

```text

  → 检查是否有 token？
    → 无 token → 跳转登录页

```

# 4、多 Guard 分层校验

```typescript
// 通用 JWT 守卫 — 解析 token，把 user 挂到 request 上
@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') { ... }

// 管理员权限守卫 — 校验 RBAC 菜单/按钮权限
@Injectable()
export class AdminPermissionGuard implements CanActivate {
  canActivate(context: ExecutionContext) {
    const { user } = context.switchToHttp().getRequest();
    // 第一步：校验身份类型
    if (user.userType !== 'admin') throw new ForbiddenException();
    // 第二步：超级管理员直接放行
    if (user.roles.includes('super_admin')) return true;
    // 第三步：校验 RBAC 权限
    const requiredPermission = this.reflector.get('permission', context.getHandler());
    if (!requiredPermission) return true;
    return user.permissions.includes(requiredPermission);
  }
}

// C端用户权限守卫 — 校验等级功能权限
@Injectable()
export class MemberPermissionGuard implements CanActivate {
  canActivate(context: ExecutionContext) {
    const { user } = context.switchToHttp().getRequest();
    // 第一步：校验身份类型
    if (user.userType !== 'member') throw new ForbiddenException();
    // 第二步：校验功能权限（等级决定可访问功能）
    const requiredPermission = this.reflector.get('permission', context.getHandler());
    if (!requiredPermission) return true;
    return user.permissions.includes(requiredPermission);
  }
}

// C端游客权限守卫 — 无需登录，直接返回 guest 角色的权限
@Injectable()
export class GuestPermissionGuard implements CanActivate {
  constructor(private readonly cacheService: ICacheService) {}
  async canActivate(context: ExecutionContext) {
    // 游客不需要 JWT，直接从缓存读取 guest 角色的权限
    const guestPermissions = await this.cacheService.get<string[]>('mono:role:permission:member:guest');
    const requiredPermission = this.reflector.get('permission', context.getHandler());
    if (!requiredPermission) return true;
    return guestPermissions?.includes(requiredPermission) ?? false;
  }
}
```

# 5、Guard 使用

Guard 在 REST Controller 和 GraphQL Resolver 中共享同一套代码，用法完全一致。数据 CRUD 已迁移到 GraphQL，Resolver 示例见 [API设计规范.md](./API设计规范.md)。

```typescript
// ===== REST Controller =====
@Controller('admin/auth')
@UseGuards(JwtAuthGuard, AdminPermissionGuard)
export class AdminAuthController { ... }

// ===== GraphQL Resolver（Guard 用法完全一致）=====
@Resolver(() => AdminAccount)
@UseGuards(JwtAuthGuard, AdminPermissionGuard)
export class AdminAccountResolver { ... }

// C端用户接口 — 需要 MemberPermissionGuard
@Controller('member/profile')
@UseGuards(JwtAuthGuard, MemberPermissionGuard)
export class MemberController { ... }

// C端公开接口 — 游客可访问，GuestPermissionGuard
@Controller('products')
@UseGuards(GuestPermissionGuard)
export class PublicController { ... }
```

# 十、用户权限聚合逻辑

# 登录后：

查询：

```text
账户角色 → 角色菜单 → 账户额外菜单权限
```

聚合步骤：

```text
1. 查询账户拥有的所有角色 → 得到角色ID列表
2. 查询所有角色关联的菜单 → 得到菜单ID列表
3. 查询账户额外菜单权限 → 授权的追加，禁止的移除
4. 从菜单中提取 permission_code → 得到 permissions 列表
5. 从菜单中构建菜单树 → 得到 menus
```

两端逻辑完全一致，只是查的表不同：

```text
管理端：admin_account_role → admin_role_menu → admin_account_menu
C端：   member_account_role → member_role_menu → member_account_menu
```

代码复用策略：使用纯函数做聚合算法，两端各写一个薄的 Service 调这个函数。

```typescript
// 纯函数 — 不依赖 Prisma，只操作数据，容易测试
// 以下类型定义补全，原文档函数签名缺少 TS 类型
type RoleMenu = { menu: { permissionCode: string } };
type AccountOverride = { menu: { permissionCode: string }; type: 'grant' | 'deny' };

function aggregatePermissions(roles: Array<{ roleMenus: RoleMenu[] }>, overrides: AccountOverride[] = []): string[] {
    // 1. 从角色菜单中提取权限
    const rolePermissions: string[] = roles.flatMap((r: { roleMenus: RoleMenu[] }) =>
        r.roleMenus.map((rm: RoleMenu) => rm.menu.permissionCode),
    );
    // 2. 处理额外权限：授权的追加，禁止的移除
    const grantPermissions: string[] = overrides
        .filter((o: AccountOverride) => o.type === 'grant')
        .map((o: AccountOverride) => o.menu.permissionCode);
    const denyPermissions: string[] = overrides
        .filter((o: AccountOverride) => o.type === 'deny')
        .map((o: AccountOverride) => o.menu.permissionCode);
    // 3. 合并去重（Set 自动去重；deny 优先于 grant）
    return [...new Set([...rolePermissions, ...grantPermissions])].filter((p: string) => !denyPermissions.includes(p));
}
```

**单测覆盖**（放在 `apps/server/src/modules/rbac/aggregate-permissions.spec.ts`，要求 ≥ 90% 覆盖）：

# 十一、超级管理员设计

超级管理员通过 admin_role 中 code='super_admin' 的角色来标识，不再使用字段。

```text
admin_role: id=1, code='super_admin', name='超级管理员'
```

特点：

- 拥有所有权限（Guard 检测到 super_admin 角色时直接放行）
- 不受角色限制
- 不校验菜单权限

安全保护（4 个保护点）：

| 操作                         | 保护方式                     |
| ---------------------------- | ---------------------------- |
| 删除 super_admin 角色        | 直接拒绝，与数量无关         |
| 禁用 super_admin 角色        | 直接拒绝，角色本身不允许禁用 |
| 移除账户的 super_admin 角色  | 检查活跃超管数 ≥ 2           |
| 禁用/软删除 super_admin 账户 | 检查活跃超管数 ≥ 2           |

代码实现：

```typescript

  return prisma.adminAccountRole.count({
    where: {
      role: { code: 'super_admin', enabled: true },
      account: { enabled: true, deleted_at: null },

    const role = await tx.adminRole.findUnique({ where: { id: roleId } });

      const activeCount = await this.activeSuperAdminCount(tx);
      if (activeCount <= 1) {
        throw new ForbiddenException('至少保留一个可用的超级管理员账户');
      }
    }
    await tx.adminAccountRole.deleteMany({
      where: { account_id: accountId, role_id: roleId },
    });
  });
}

// 禁用/软删除账户时：检查是否是最后一个活跃超管
async disableAccount(accountId: string) {
  await this.prisma.$transaction(async (tx) => {
    const isSuperAdmin = await tx.adminAccountRole.findFirst({
      where: {
        account_id: accountId,
        role: { code: 'super_admin' },
      },
    });
    if (isSuperAdmin) {
      const activeCount = await this.activeSuperAdminCount(tx);
      if (activeCount <= 1) {
        throw new ForbiddenException('至少保留一个可用的超级管理员账户');
      }
    }
    await tx.account.update({
      where: { id: accountId },
      data: { enabled: false },
    });
  });
}
```

并发安全说明：

- 所有检查 + 操作包在 `$transaction` 里
- **关键**：在事务内使用 `SELECT ... FOR UPDATE` 锁住超管相关行，防止两个事务同时通过 count 检查的竞态问题。必须在 count 查询前加锁：

```typescript
// 移除账户的 super_admin 角色：使用行级锁防并发
async removeRoleFromAccount(accountId: string, roleId: string) {
  await this.prisma.$transaction(async (tx) => {
    const role = await tx.adminRole.findUnique({ where: { id: roleId } });
    if (role.code === 'super_admin') {
      // 关键：先锁住所有超管关联行，再计数
      // $queryRaw 使用 SELECT ... FOR UPDATE 获取行级锁
      await tx.$queryRaw`
        SELECT 1 FROM admin_account_role
        WHERE role_id IN (
          SELECT id FROM admin_role WHERE code = 'super_admin' AND enabled = true
        )
        FOR UPDATE
      `;
      const activeCount = await this.activeSuperAdminCount(tx);
      if (activeCount <= 1) {
        throw new ForbiddenException('至少保留一个可用的超级管理员账户');
      }
    }
    await tx.adminAccountRole.deleteMany({
      where: { account_id: accountId, role_id: roleId },
    });
  });
}
```

- count 只算活跃超管（account.enabled=1 + deleted_at=null + role.enabled=1），避免"禁用账户但 count 仍通过"的漏洞
- 如需更强保证，可考虑在数据库层面增加 CHECK 约束（需 PostgreSQL 触发器），确保活跃超管数始终 ≥ 1

# 十二、权限缓存设计

## JWT 与 Redis 的分工

核心原则：**JWT = 身份证号（不可变），Redis = 档案袋（随时更新）。**

JWT 只存两个字段：

```typescript
// JWT Payload — 只存不可变的身份标识
{ sub: accountId, userType: 'admin' }
```

为什么需要 userType 而不只是 accountId：

- accountId 只能告诉你"这是谁"，但不知道"这是哪端的用户"
- 登录后，Guard 需要知道该走哪个权限校验逻辑（AdminPermissionGuard / MemberPermissionGuard）
- 如果只有 accountId，每次请求都要查数据库判断 userType，失去了 JWT 无状态的优势
- userType 在账户创建后不会改变，属于不可变信息，适合放 JWT

---

## 两级缓存 + 懒构建

### 为什么需要两级缓存

如果只用一层缓存（每个账户一条 key），修改角色菜单时需要失效所有关联账户的缓存：

```text
场景：管理员把角色"运营专员"的菜单删掉了一个按钮
  → 需要失效的 key = 所有分配了"运营专员"角色的 account 的缓存
  → 20 个运营？200 个运营？
  → 每次改角色 = 全量扫描 admin_account_role 表 → 逐个删 Redis
```

两层缓存的核心思路：**角色级缓存吸收大量重复查询，账户级缓存保证热路径只需 1 次 Redis GET。**

---

### 缓存 Key 设计

```text
第一层 — 源数据（角色级，变更频率低）：
  mono:role:permission:{userType}:{roleCode}    → SET ['iam:admin:view', 'iam:admin:create', ...]
  mono:role:menus:{userType}:{roleCode}   → JSON [{id, name, type, path, parentId, ...}, ...]  # 扁平列表
  mono:role:accounts:{userType}:{roleCode} → SET ['uuid-1', 'uuid-2', ...]  # 持有该角色的账户 ID 列表

第二层 — 计算结果（账户级，请求直接读）：
  mono:auth:{accountId}  → JSON {
      roles: ['super_admin', 'editor'],
      permissions: ['iam:admin:view', 'iam:admin:create', ...],
      menus: [{...目录树...}, ...]  # 拼好的树
  }
```

说明：

- role 级 key 加了 `{userType}` 前缀，因为不同端可能有同名角色码
- `mono:role:menus` 存扁平列表（不拼树），`mono:auth` 存拼好的树，拼树只在 cache miss 时做一次
- `mono:role:accounts` 维护角色→账户的反向映射，角色菜单变更时可直接找到受影响的账户，无需查 DB
- `mono:auth:{accountId}` 的 TTL 设 30 分钟，自然过期后触发懒重建

---

### 读取流程

```text
请求 query { me { ... } } (accountId=100)

  ┌─ Redis GET mono:auth:100
  │   命中 → 直接返回（热路径：1 次 Redis GET）
  │
  │   未命中 ↓
  │
  │   1. DB 查账户角色 → ['super_admin', 'editor']
  │   2. 对每个 roleCode：
  │        Redis GET mono:role:permission:admin:super_admin   → 命中，直接用
  │        Redis GET mono:role:menus:admin:super_admin  → 命中，直接用
  │        Redis GET mono:role:permission:admin:editor        → 未命中 → 查 DB → 写 mono:role:permission:admin:editor
  │        Redis GET mono:role:menus:admin:editor       → 未命中 → 查 DB → 写 mono:role:menus:admin:editor
  │   3. 合并：所有角色的 permissions 去重
  │   4. 合并：所有角色的 menus 去重 + 拼树
  │   5. 应用账户级覆盖（DB 查 account_menu 表，grant 追加，deny 移除）
  │   6. Redis SET mono:auth:100 ← 合并后的结果
  │   7. 返回
```

第一次 miss 后，后续请求都是 1 次 Redis GET。

拼树逻辑只在 cache miss 时执行一次：

```typescript
// 只有 mono:auth:{accountId} miss 时才执行
function buildMenuTree(flatMenus: Menu[]): MenuNode[] {
    const map = new Map<string, MenuNode>();
    const tree: MenuNode[] = [];

    for (const m of flatMenus) {
        map.set(m.id, { ...m, children: [] });
    }
    for (const m of flatMenus) {
        const node = map.get(m.id)!;
        if (m.parentId === null) {
            tree.push(node);
        } else {
            map.get(m.parentId)?.children.push(node);
        }
    }
    return tree;
}
```

### 各场景的失效操作

#### 场景 A：修改账户角色（给用户分配/移除角色）

```text
→ Redis DEL mono:auth:{accountId}
→ Redis SADD/SREM mono:role:accounts:{userType}:{roleCode} {accountId}
→ 下一次请求触发懒构建，角色级缓存不受影响
```

成本：1 次 DEL

---

#### 场景 B：修改角色菜单（角色管理里勾选/取消菜单）

```text
→ Redis DEL mono:role:permission:{userType}:{roleCode}
→ Redis DEL mono:role:menus:{userType}:{roleCode}
→ Redis SMEMBERS mono:role:accounts:{userType}:{roleCode} → 得到账户 ID 列表（无需查 DB）
→ Redis DEL mono:auth:{id} for each  (pipeline 批量删)
```

说明：

- 通过 `mono:role:accounts` 反向映射直接获取受影响账户，无需查 DB
- 一个角色通常对应几十到几百个账户，pipeline 批量删很快
- 删完后，这些账户的下次请求自动懒重建
- 不会雪崩：auth 的 TTL 设 30min，请求均匀到达，不会同时回源

成本：2 次 DEL + 1 次 DB 查 + pipeline 批量 DEL

---

#### 场景 C：菜单结构变更（删除/重命名/移动菜单）

关键结论：**新增菜单不会自动赋给任何角色，不需要失效。** 只有删除/重命名/移动才需要处理。

```text
→ Redis DEL mono:role:permission:{userType}:*   (按前缀批量删该端所有角色权限缓存)
→ Redis DEL mono:role:menus:{userType}:*  (按前缀批量删该端所有角色菜单缓存)
→ mono:auth:* 不主动删，而是缩短 TTL 为 5 分钟，让用户分批自然过期
```

防雪崩策略：

```typescript
// 菜单结构变更时
async function onMenuStructureChange(userType: string) {
    // 1. 删所有角色级缓存（下次构建 account 缓存时从 DB 拿最新数据）
    await cache.delByPattern(`mono:role:permission:${userType}:*`);
    await cache.delByPattern(`mono:role:menus:${userType}:*`);

    // 2. mono:auth:* 不主动删，但把 TTL 缩短为 5 分钟
    //    让用户在 5 分钟内自然分批过期，避免同时回源
    await cache.setTtlByPattern('mono:auth:*', 300);
}
```

成本：批量 DEL + 自然过期（场景 C 一周可能都不发生一次）

### 失效操作汇总

| 操作         | 失效范围                                                           | 成本            |
| ------------ | ------------------------------------------------------------------ | --------------- |
| 改账户角色   | mono:auth:{accountId} × 1                                          | 1 次 DEL        |
| 改角色菜单   | mono:role:permission:{code} + mono:role:menus:{code} + auth:\* × N | pipeline 批量删 |
| 菜单结构变更 | mono:role:permission:_ + mono:role:menus:_ + auth 缩短 TTL         | 批量 + 自然过期 |

最贵的操作（菜单结构变更）一周可能一次都没有。角色级缓存吸收了大量"改角色菜单"场景中的账户级重复查询。

## 缓存适配器设计（基于 ioredis 直接封装）

### 为什么不基于 @nestjs/cache-manager

`@nestjs/cache-manager` 是 NestJS 官方缓存方案，但它提供的 `Cache` 接口只有基本操作（`get` / `set` / `del` / `reset`）。权限缓存场景需要三个官方不提供的操作：

| 我们需要        | `Cache` 有吗 | 用途                         |
| --------------- | ------------ | ---------------------------- |
| get / set / del | ✅ 有        | 基本读写                     |
| delByPattern    | ❌ 没有      | 场景 C：菜单结构变更时批量删 |
| mget            | ❌ 没有      | 批量读角色级缓存             |
| setTtlByPattern | ❌ 没有      | 防雪崩：批量缩短 TTL         |

此外 `cache-manager-redis-store` 只支持 Redis v3，无法利用新版 Redis 的 SCAN 等命令，性能和功能都受限制。

直接基于 `ioredis` 封装 ICacheService，需要 Redis 时有完整能力（SCAN、pipeline 批量 DEL、MGET），无 Redis 时降级为内存缓存（本地开发）。接口定义之后，业务代码不依赖底层实现。

---

### 接口设计

```typescript
// 缓存接口 — 业务代码只依赖这个
interface ICacheService {
    get<T>(key: string): Promise<T | null>; // 读取缓存
    set(key: string, value: any, ttl?: number): Promise<void>; // 写入缓存，ttl 单位秒
    del(key: string): Promise<void>; // 删除缓存
    delByPattern(pattern: string): Promise<void>; // 按模式批量删除（场景 C 用）
    mget<T>(keys: string[]): Promise<(T | null)[]>; // 批量读取
    setTtlByPattern(pattern: string, ttl: number): Promise<void>; // 按模式批量设置 TTL（防雪崩用）
    exists(key: string): Promise<boolean>; // 检查 key 是否存在
    incr(key: string): Promise<number>; // 计数器 +1（用于限流计数）
    setex(key: string, ttl: number, value: any): Promise<void>; // 写入缓存并设置 TTL（原子操作）
    ttl(key: string): Promise<number>; // 获取 key 剩余 TTL（秒）
}
```

---

### 实现：ioredis 封装 + 内存降级

```typescript
// cache.service.ts — 基于 ioredis 封装，无 Redis 时降级内存
@Injectable()
class CacheService implements ICacheService {
    constructor(@Optional() @Inject('REDIS_CLIENT') private redis: Redis | null) {}

    // 内存降级存储（本地开发无 Docker 时使用）
    private readonly memoryStore = new Map<string, { value: any; expiresAt: number }>();

    private _getFromMemory<T>(key: string): T | null {
        const item = this.memoryStore.get(key);
        if (!item) return null;
        if (Date.now() > item.expiresAt) {
            this.memoryStore.delete(key);
            return null;
        }
        return item.value as T;
    }

    async get<T>(key: string): Promise<T | null> {
        if (this.redis) {
            const val = await this.redis.get(key);
            return val ? (JSON.parse(val) as T) : null;
        }
        return this._getFromMemory<T>(key);
    }

    async set(key: string, value: any, ttl?: number): Promise<void> {
        if (this.redis) {
            const serialized = JSON.stringify(value);
            if (ttl) await this.redis.setex(key, ttl, serialized);
            else await this.redis.set(key, serialized);
        } else {
            this.memoryStore.set(key, { value, expiresAt: Date.now() + (ttl ?? 300) * 1000 });
        }
    }

    async del(key: string): Promise<void> {
        if (this.redis) await this.redis.del(key);
        else this.memoryStore.delete(key);
    }

    // 补充方法 — Redis 模式用 ioredis，内存模式降级处理
    async delByPattern(pattern: string): Promise<void> {
        if (this.redis) {
            const keys = await this.scanKeys(pattern);
            if (keys.length) await this.redis.del(...keys);
        } else {
            this.memoryStore.clear(); // 内存模式无法按 pattern 精确删
        }
    }

    async mget<T>(keys: string[]): Promise<(T | null)[]> {
        if (this.redis) {
            const vals = await this.redis.mget(...keys);
            return vals.map((v) => (v ? (JSON.parse(v) as T) : null));
        }
        return keys.map((k) => this._getFromMemory<T>(k));
    }

    async setTtlByPattern(pattern: string, ttl: number): Promise<void> {
        if (this.redis) {
            const keys = await this.scanKeys(pattern);
            const pipeline = this.redis.pipeline();
            for (const key of keys) pipeline.expire(key, ttl);
            await pipeline.exec();
        }
        // 内存模式：不支持
    }

    async exists(key: string): Promise<boolean> {
        if (this.redis) return (await this.redis.exists(key)) === 1;
        return this.memoryStore.has(key);
    }

    async incr(key: string): Promise<number> {
        if (this.redis) return this.redis.incr(key);
        const val = (this._getFromMemory<number>(key) ?? 0) + 1;
        this.memoryStore.set(key, { value: val, expiresAt: Date.now() + 300000 });
        return val;
    }

    async setex(key: string, ttl: number, value: any): Promise<void> {
        await this.set(key, value, ttl);
    }

    async ttl(key: string): Promise<number> {
        if (this.redis) return this.redis.ttl(key);
        const item = this.memoryStore.get(key);
        if (!item) return -2;
        return Math.ceil((item.expiresAt - Date.now()) / 1000);
    }

    // 辅助方法：SCAN 匹配 key
    private async scanKeys(pattern: string): Promise<string[]> {
        const keys: string[] = [];
        let cursor = '0';
        do {
            const [next, matched] = await this.redis!.scan(cursor, 'MATCH', pattern, 'COUNT', 100);
            cursor = next;
            keys.push(...matched);
        } while (cursor !== '0');
        return keys;
    }
}
```

### 模块注册

```typescript
// cache.module.ts — 基于 ioredis 的 DynamicModule
import { DynamicModule, Module } from '@nestjs/common';
import Redis from 'ioredis';

@Module({})
export class CacheModule {
    static forRootAsync(): DynamicModule {
        return {
            module: CacheModule,
            global: true,
            providers: [
                {
                    provide: 'REDIS_CLIENT',
                    useFactory: (configService: ConfigService) => {
                        const redisUrl = configService.get('REDIS_URL');
                        if (!redisUrl) return null;
                        return new Redis(redisUrl, {
                            maxRetriesPerRequest: 3,
                            retryStrategy: (times) => Math.min(times * 100, 3000),
                        });
                    },
                    inject: [ConfigService],
                },
                {
                    provide: ICacheService,
                    useClass: CacheService,
                },
            ],
            exports: [ICacheService],
        };
    }
}
```

# 十八、系统核心原则

# 1、菜单即权限资源

不单独设计 permission 表。

---

# 2、按钮也是菜单

按钮属于：

```text
type = button
```

---

# 3、前端只负责显示

真正权限校验：

必须后端完成。

---

# 4、权限永远是字符串

例如：

```text
iam:admin:view
product:goods:delete
member:course:vip
```

---

# 5、各端 RBAC 独立

管理端、C端有各自的角色表、菜单表、关联表，互不干扰。

---

# 6、统一认证，按端分流

所有用户共用 account 认证，通过 user_type 决定走哪个 Guard。user_type 一旦创建不可修改。

---

# 6.1、登录标识与业务数据分离

登录标识（username / phone / email / openid）统一放在 account_identity 表，profile 只存业务数据。新增登录方式只需 INSERT 一行记录，零 DDL。

---

# 7、两端权限逻辑一致

两端的权限聚合逻辑完全一致（账户→角色→菜单→额外权限），只是查的表不同。代码复用使用纯函数，不使用抽象类或泛型注入。

---

# 8、JWT 只存不可变信息

JWT 只存 accountId + userType，角色/权限/菜单等可变数据全部存 Redis。

---

# 9、接口分离原则

登录接口必须分离（参数不同），profile 和业务接口共用（Guard 按 userType 分流）。

---

# 10、游客通过 guest 角色控制

未登录用户自动应用 guest 角色的权限，不需要在 account 表中创建记录。guest 权限是角色级缓存的一部分（mono:role:permission:member:guest），不需要单独的 key，也没有账户级缓存这一层。管理员修改 guest 角色菜单时，走标准的角色级缓存失效逻辑（删 mono:role:permission:member:guest + mono:role:menus:member:guest），下次游客访问触发懒构建。

---

# 11、缓存两层设计 + 懒构建

角色级缓存（源数据）+ 账户级缓存（计算结果），热路径只需 1 次 Redis GET。角色变更只删角色级 key，不扩散到所有用户。

---

# 12、缓存基于 ioredis 直接封装

缓存层直接基于 ioredis 封装 ICacheService，补充 delByPattern / mget / setTtlByPattern 等批量操作。无 Redis 时降级为内存 Map，不影响本地开发。

# 二十二、未来扩展点

以下功能当前不实现，但架构已预留扩展能力：

## 0、并发更新防护（乐观锁）

当前 RBAC 表的并发更新（如两个管理员同时编辑同一角色）采用"最后写入胜出"策略。如需防止覆盖他人修改，可在关键表增加 `version` 字段：

```sql
ALTER TABLE admin_role ADD COLUMN version INT NOT NULL DEFAULT 1;
```

```typescript
// 更新时检查版本号
async updateRole(roleId: string, data: UpdateRoleInput, expectedVersion: number) {
  const result = await this.prisma.adminRole.updateMany({
    where: { id: roleId, version: expectedVersion },
    data: { ...data, version: { increment: 1 } },
  });
  if (result.count === 0) {
    throw new ConflictException('数据已被他人修改，请刷新后重试');
  }
  return result;
}
```

> **当前策略**：暂不实现乐观锁，使用"最后写入胜出"。管理后台场景下并发冲突概率极低（同一角色很少被两人同时编辑），复杂度收益比不高。后续如有需求再添加。

## 1、角色互斥

某些角色不能同时分配给同一用户（如"出纳"和"审核"不能是同一人，防止自己发起自己审批）。

预留方案：

```sql
CREATE TABLE admin_role_conflict (
  role_id_a UUID NOT NULL,
  role_id_b UUID NOT NULL,
  PRIMARY KEY (role_id_a, role_id_b)
);
```

分配角色时校验互斥关系。

## 2、数据权限

当前只做了功能权限（能看什么菜单、点什么按钮），未做数据权限（能看到什么范围的数据）。

未来可扩展：

- 销售只能看自己的订单
- 部门经理能看本部门的数据
