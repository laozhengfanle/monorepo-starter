# 日志与监控

> 生产环境必备：结构化日志（Pino）、健康检查（Terminus）、Prometheus 指标暴露、异常告警。开发环境用 human-readable 格式，生产环境用 JSON。

## 概述

基座日志使用 `nestjs-pino`（结构化 JSON）+ `pino-pretty`（dev 友好）。监控使用 Prometheus 指标 + Grafana dashboard（推荐接入）。

本文档覆盖：

- **Pino 结构化日志** — 配置、行为差异、字段脱敏、请求 ID
- **LoggingInterceptor** — 全局请求/响应日志
- **健康检查** — liveness / readiness / 综合
- **Prometheus 监控** — 端点保护、自定义指标、Grafana 接入
- **告警规则** — 应用层 / 基础设施层 / 业务层
- **分布式追踪（预留）** — OpenTelemetry 接入指引

## Pino 结构化日志

### 配置

- 库：`nestjs-pino@latest` + `pino-http@latest` + `pino-pretty@latest`（dev）
- 配置文件：`apps/server/src/common/logger/logger.config.ts`
- 全局模块：`apps/server/src/common/logger/logger.module.ts`
- 接入：`apps/server/src/app.module.ts`（imports[0]，第一位）+ `apps/server/src/main.ts`（`app.useLogger(app.get(Logger))`）

```ts
// apps/server/src/common/logger/logger.config.ts（节选）
export function getPinoConfig(env: 'dev' | 'prod'): Params {
    const commonPinoHttp: PinoHttpOptions = {
        // 用 crypto.randomUUID() 给每个请求生成唯一 ID
        genReqId: (req, res) => {
            const id = (req.headers['x-request-id'] as string | undefined) ?? crypto.randomUUID();
            res.setHeader('X-Request-ID', id);
            return id;
        },
        // 自定义字段：每条日志都会带上 requestId + userId
        customProps: (req) => {
            const user = (req as { user?: { id?: string } }).user;
            return {
                requestId: (req as { id?: string }).id,
                userId: user?.id,
            };
        },
    };

    if (env === 'dev') {
        return {
            pinoHttp: {
                ...commonPinoHttp,
                level: 'debug',
                transport: {
                    target: 'pino-pretty',
                    options: { singleLine: true, colorize: true, translateTime: 'HH:MM:ss.l' },
                },
                redact: { paths: REDACT_PATHS, censor: REDACT_CENSOR },
            },
        };
    }

    return {
        pinoHttp: {
            ...commonPinoHttp,
            level: 'info',
            redact: { paths: REDACT_PATHS, censor: REDACT_CENSOR },
        },
    };
}
```

### 行为差异

| 环境                       | 格式             | level | 颜色 | 脱敏 |
| -------------------------- | ---------------- | ----- | ---- | ---- |
| dev (NODE_ENV=development) | 单行 pino-pretty | debug | ✅   | ✅   |
| prod (NODE_ENV=production) | JSON             | info  | ❌   | ✅   |

### 脱敏规则（dev + prod 都启用）

| 字段路径                                            | 处理                |
| --------------------------------------------------- | ------------------- |
| `req.headers.authorization`                         | 替换为 `[REDACTED]` |
| `req.headers.cookie`                                | 替换为 `[REDACTED]` |
| `*.password`                                        | 替换为 `[REDACTED]` |
| `*.token`                                           | 替换为 `[REDACTED]` |
| `*.newPassword` / `confirmPassword` / `oldPassword` | 替换为 `[REDACTED]` |
| `res.body.accessToken` / `refreshToken`             | 替换为 `[REDACTED]` |
| `req.body.phone` / `req.body.email`                 | 替换为 `[REDACTED]` |

> **原则**：subagent / 业务代码不掌握具体格式，全字段 redact 比部分打码更安全。dev 环境也启用脱敏，开发日志不应暴露密码。

### 请求 ID

- 每个请求生成 UUID（或复用 `X-Request-ID` header），注入 `req.id`
- 响应头 `X-Request-ID` 回写
- 日志条目自动含 `requestId` 字段（通过 `customProps`）
- 前端 / 客户端报错时回传 `X-Request-ID`，运维可一键定位日志

### 日志级别

| 级别    | 说明                   | 使用场景                       |
| ------- | ---------------------- | ------------------------------ |
| `fatal` | 致命错误，进程即将退出 | DB 连接断开、内存耗尽          |
| `error` | 错误，功能不可用       | 请求处理失败、外部服务调用失败 |
| `warn`  | 警告，功能可用但需关注 | 限流触发、登录失败、缓存失效   |
| `info`  | 正常业务流程           | 请求完成、用户登录成功         |
| `debug` | 调试信息               | 开发环境启用，生产默认关闭     |
| `trace` | 最详细                 | 仅排查问题时临时开启           |

### 日志内容规范

每条日志必须包含：

| 字段        | 说明                         | 示例                       |
| ----------- | ---------------------------- | -------------------------- |
| `requestId` | 请求唯一 ID                  | `550e8400-e29b-...`        |
| `msg`       | 日志消息                     | `request completed`        |
| `level`     | 日志级别                     | `info`                     |
| `time`      | 时间戳（ISO 8601）           | `2026-06-15T10:30:00.000Z` |
| `context`   | NestJS 上下文（模块/服务名） | `AuthService`              |

业务相关日志额外包含：

| 字段        | 说明                                              |
| ----------- | ------------------------------------------------- |
| `accountId` | 操作者 ID                                         |
| `action`    | 操作类型（`login`、`create_user`、`update_role`） |
| `resource`  | 操作的资源                                        |
| `result`    | 操作结果（`success`、`failure`）                  |
| `duration`  | 耗时（毫秒）                                      |

## LoggingInterceptor

- 路径：`apps/server/src/common/interceptors/logging.interceptor.ts`
- 全局注册：`app.module.ts` 通过 `APP_INTERCEPTOR` 注册
- 记录：method / url / statusCode / latencyMs / requestId
- 成功 → `info` 级别；失败 → `error` 级别
- 兼容 GraphQL：识别 `context.getType() === 'graphql'` + `GqlExecutionContext`

```ts
// 节选
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
    constructor(@InjectPinoLogger(LoggingInterceptor.name) private readonly logger: PinoLogger) {}

    intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
        const start = Date.now();
        const isGraphQL = context.getType<string>() === 'graphql';
        const gqlCtx = isGraphQL ? GqlExecutionContext.create(context).getContext() : null;
        const req = (gqlCtx?.req ?? context.switchToHttp().getRequest?.()) as
            | { method?: string; url?: string; id?: string; body?: { operationName?: string } }
            | undefined;
        const method = req?.method ?? 'GRAPHQL';
        const url = isGraphQL ? (req?.body?.operationName ?? 'anonymous') : req?.url;
        const requestId = req?.id;

        return next.handle().pipe(
            tap({
                next: () => {
                    const latencyMs = Date.now() - start;
                    this.logger.info({ requestId, method, url, latencyMs }, 'request completed');
                },
                error: (err: unknown) => {
                    const latencyMs = Date.now() - start;
                    this.logger.error({ requestId, method, url, latencyMs, err }, 'request failed');
                },
            }),
        );
    }
}
```

## 健康检查

> 基于 `@nestjs/terminus`，提供 K8s 友好的 liveness / readiness 探针端点。

### 端点

| 端点                    | 用途                             | 鉴权     | 触发者             |
| ----------------------- | -------------------------------- | -------- | ------------------ |
| `GET /health`           | 综合（DB + Redis + 磁盘 + 内存） | **公开** | 人工 / 监控        |
| `GET /health/liveness`  | 进程存活（不查依赖）             | **公开** | K8s livenessProbe  |
| `GET /health/readiness` | 依赖就绪（DB + Redis 可达）      | **公开** | K8s readinessProbe |

> 公开但只返回 up/down + 依赖名，不返回 DB schema / 业务数据 / 内部 hostname，防止信息泄露。

### Docker 健康检查

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --spider -q http://localhost:3000/health/liveness || exit 1
```

### K8s 探针配置

```yaml
livenessProbe:
    httpGet:
        path: /health/liveness
        port: 3000
    initialDelaySeconds: 15
    periodSeconds: 30

readinessProbe:
    httpGet:
        path: /health/readiness
        port: 3000
    initialDelaySeconds: 5
    periodSeconds: 10
```

## Prometheus 监控

### 暴露端点

- 路径：`GET /metrics`
- 实现：`apps/server/src/metrics/metrics.controller.ts` 继承 `PrometheusController`
- 模块：`@willsoto/nestjs-prometheus@latest` + `prom-client@latest`
- 内容：prom-client 默认指标（CPU/内存/GC/event loop lag）+ 自定义指标

### 端点保护

| 维度      | 措施                                                                                                                                                     |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| IP 白名单 | `MetricsIpGuard`（`apps/server/src/common/guards/metrics-ip.guard.ts`）：允许 `10/8`、`172.16/12`、`192.168/16`、`127/8`、`::1`、`fc00::/7`、`fe80::/10` |
| 外网访问  | 403 Forbidden                                                                                                                                            |
| 抓取方式  | Prometheus 走 `host.docker.internal:9090`（开发）或 SSH 隧道（生产），不开放公网                                                                         |

```ts
// 节选：MetricsIpGuard
const PRIVATE_NETS: RegExp[] = [
    /^10\./, // 10.0.0.0/8
    /^172\.(1[6-9]|2[0-9]|3[0-1])\./, // 172.16.0.0/12
    /^192\.168\./, // 192.168.0.0/16
    /^127\./, // 127.0.0.0/8 loopback
    /^::1$/, // IPv6 loopback
    /^::ffff:127\./, // IPv4-mapped IPv6 loopback
    /^fc00:/, // IPv6 ULA
    /^fe80:/, // IPv6 link-local
];

@Injectable()
export class MetricsIpGuard implements CanActivate {
    canActivate(context: ExecutionContext): boolean {
        const req = context.switchToHttp().getRequest<Request>();
        const ip = req.ip || req.socket.remoteAddress || '';
        if (PRIVATE_NETS.some((re) => re.test(ip))) return true;
        throw new ForbiddenException('metrics endpoint only accessible from private network');
    }
}
```

### 自定义指标

| 指标名                       | 类型      | 标签                           | 说明                                                 |
| ---------------------------- | --------- | ------------------------------ | ---------------------------------------------------- |
| `http_requests_total`        | Counter   | method, route, status_code     | HTTP 请求总数                                        |
| `http_request_duration_ms`   | Histogram | method, route                  | HTTP 响应延迟（buckets: 10/50/100/500/1000/5000 ms） |
| `http_requests_in_flight`    | Gauge     | -                              | 当前 in-flight 请求数                                |
| `graphql_query_duration_ms`  | Histogram | operation_name, operation_type | GraphQL 查询延迟                                     |
| `graphql_query_errors_total` | Counter   | operation_name, code           | GraphQL 错误数                                       |
| `db_query_duration_ms`       | Histogram | model, action                  | Prisma 查询延迟（**暂未集成埋点**）                  |
| `db_connections_active`      | Gauge     | -                              | 活跃 DB 连接（**暂未集成埋点**）                     |
| `login_failures_total`       | Counter   | reason                         | 登录失败次数                                         |
| `rate_limit_exceeded_total`  | Counter   | limit                          | 限流触发次数（short/medium/long）                    |
| `cache_hit_ratio`            | Gauge     | key_prefix                     | 缓存命中率                                           |

### Collectors 路径

| 文件                                                     | 负责指标                                                |
| -------------------------------------------------------- | ------------------------------------------------------- |
| `apps/server/src/metrics/collectors/http.metrics.ts`     | `http_*`                                                |
| `apps/server/src/metrics/collectors/graphql.metrics.ts`  | `graphql_*`                                             |
| `apps/server/src/metrics/collectors/db.metrics.ts`       | `db_*`（占位）                                          |
| `apps/server/src/metrics/collectors/business.metrics.ts` | `login_failures_*` / `rate_limit_*` / `cache_hit_ratio` |

### 接入业务埋点

`BusinessMetrics` collector 已定义，业务埋点由调用方在 service / guard 中主动触发。集成方式如下：

```ts
// auth.service.ts 登录失败时
this.businessMetrics.incLoginFailure('invalid_credentials');

// gql-aware-throttler.guard.ts 限流时
this.businessMetrics.incRateLimit('short');

// redis.service.ts 缓存读写时（每 30s 采样）
this.businessMetrics.setCacheHitRatio('user:', hits / (hits + misses));
```

### Grafana 接入（推荐）

1. 添加 Prometheus data source（`http://prometheus:9090`）
2. 导入 dashboard：Node.js Full / NestJS Prometheus
3. 关键面板：
    - HTTP request rate + p95 latency
    - GraphQL error rate
    - Login failures (alert: > 10/min)
    - Rate limit exceeded (alert: > 100/min)

## 告警规则

### 告警分级

| 级别       | 响应时间  | 示例                           |
| ---------- | --------- | ------------------------------ |
| `critical` | 5 分钟内  | 服务宕机、数据库断连           |
| `warning`  | 30 分钟内 | 错误率升高、延迟恶化           |
| `info`     | 次日处理  | 缓存命中率下降、磁盘使用 > 80% |

### 应用层告警

| 规则名          | PromQL 表达式                                                                         | 持续时间 | 级别     |
| --------------- | ------------------------------------------------------------------------------------- | -------- | -------- |
| 服务宕机        | `up{job="mono-server"} == 0`                                                          | 1m       | critical |
| 高错误率        | `rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05` | 5m       | critical |
| P95 延迟过高    | `histogram_quantile(0.95, rate(http_request_duration_ms_bucket[5m])) > 1000`          | 5m       | warning  |
| P99 延迟过高    | `histogram_quantile(0.99, rate(http_request_duration_ms_bucket[5m])) > 3000`          | 5m       | warning  |
| 限流触发频繁    | `rate(rate_limit_exceeded_total[5m]) > 10`                                            | 10m      | warning  |
| Event Loop 阻塞 | `event_loop_lag_seconds > 1`                                                          | 2m       | warning  |

### 基础设施层告警

| 规则名         | PromQL 表达式                                                                                    | 持续时间 | 级别     |
| -------------- | ------------------------------------------------------------------------------------------------ | -------- | -------- |
| 堆内存过高     | `memory_heap_bytes > 400 * 1024 * 1024`                                                          | 5m       | warning  |
| RSS 内存过高   | `process_resident_memory_bytes > 600 * 1024 * 1024`                                              | 5m       | warning  |
| 数据库连接失败 | `health_check_status{check="database"} == 0`                                                     | 1m       | critical |
| Redis 连接失败 | `redis_up == 0`                                                                                  | 1m       | critical |
| 缓存命中率低   | `cache_hit_ratio < 0.5`                                                                          | 10m      | info     |
| 磁盘使用率高   | `node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"} < 0.2` | 30m      | warning  |

### 业务层告警

| 规则名               | PromQL 表达式                                                    | 持续时间 | 级别     |
| -------------------- | ---------------------------------------------------------------- | -------- | -------- |
| 登录失败激增         | `rate(login_failures_total[5m]) > 5`                             | 5m       | warning  |
| Token 重用检测       | `increase(token_reuse_detected_total[1h]) > 0`                   | 0m       | critical |
| 验证码发送失败       | `rate(verification_code_sent_total{status="failed"}[10m]) > 0.1` | 10m      | warning  |
| GraphQL 查询深度超限 | `rate(graphql_depth_limit_exceeded_total[5m]) > 1`               | 5m       | warning  |
| GraphQL 复杂度超限   | `rate(graphql_complexity_limit_exceeded_total[5m]) > 1`          | 5m       | warning  |

### Prometheus 告警配置示例

```yaml
# docker/alerts.yml
groups:
    - name: application
      rules:
          - alert: ServiceDown
            expr: up{job="mono-server"} == 0
            for: 1m
            labels:
                severity: critical
            annotations:
                summary: '服务宕机'
                description: 'mono-server 实例 {{ $labels.instance }} 已宕机超过 1 分钟'

          - alert: HighErrorRate
            expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
            for: 5m
            labels:
                severity: critical
            annotations:
                summary: '5xx 错误率超过 5%'

          - alert: LoginFailuresSpike
            expr: rate(login_failures_total[5m]) > 5
            for: 5m
            labels:
                severity: warning
            annotations:
                summary: '登录失败激增'
```

### 告警静默与抑制

| 场景             | 策略                                   | 说明                                   |
| ---------------- | -------------------------------------- | -------------------------------------- |
| 计划内维护       | Alertmanager `silence` 静默            | 维护窗口期不发送告警                   |
| 服务宕机引发连锁 | Alertmanager `inhibit_rules` 抑制      | 服务宕机时，抑制其下游的连接失败告警   |
| 重复告警         | `for` 持续时间 + `repeat_interval: 4h` | 同一告警 4 小时内不重复通知            |
| 深夜低优先级     | 路由规则：info 级别仅白天通知          | 22:00-08:00 的 info 告警延迟到次日发送 |

## 日志采集架构

```text
┌─────────┐     ┌──────────┐     ┌─────────┐     ┌─────────┐
│ NestJS   │────→│ stdout   │────→│ Promtail│────→│  Loki   │
│ (Pino)   │     │ (JSON)   │     │         │     │         │
└─────────┘     └──────────┘     └─────────┘     └────┬────┘
                                                      │
                 ┌─────────┐                    ┌─────▼─────┐
                 │Metrics  │───────────────────→│  Grafana  │
                 │(Prometheus)│                  │(Dashboards)│
                 └─────────┘                    └───────────┘
```

> Prometheus 管指标，Loki 管日志，OpenTelemetry 管链路——三者组成可观测性三支柱（Metrics / Logs / Traces）。

## 分布式追踪：OpenTelemetry（预留）

> OpenTelemetry 当前**未集成**到基座，本节为后续接入指引。

### 为什么需要 OpenTelemetry

| 场景             | 没有 Trace                       | 有 Trace                                |
| ---------------- | -------------------------------- | --------------------------------------- |
| 请求延迟高       | 只知道 P99 高，不知道卡在哪      | 精确定位到哪个 DB 查询 / 外部调用耗时   |
| 错误排查         | 日志中搜索 requestId，手动拼链路 | 一条 Trace 展示完整调用树               |
| GraphQL 嵌套查询 | 不知道哪个字段解析慢             | 每个 resolver 是一个 Span，耗时一目了然 |

### 接入步骤（待办）

1. `pnpm -F @apps/server add @opentelemetry/api @opentelemetry/sdk-node @opentelemetry/auto-instrumentations-node @opentelemetry/exporter-trace-otlp-http @opentelemetry/instrumentation-prisma`
2. 写 `apps/server/src/tracing.ts`，**必须是 main.ts 的第一行 import**
3. 在 `PinoLogger` 的 `customProps` 中注入 `traceId` / `spanId`
4. 部署 Grafana Tempo 接收 OTLP

### 环境变量

| 变量                          | 说明          | 默认值                     |
| ----------------------------- | ------------- | -------------------------- |
| `OTEL_ENABLED`                | 是否启用追踪  | `false`（开发环境关闭）    |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | OTLP 导出地址 | `http://localhost:4318`    |
| `OTEL_TRACES_SAMPLER`         | 采样策略      | `parentbased_traceidratio` |
| `OTEL_TRACES_SAMPLER_ARG`     | 采样率（0-1） | `0.1`（生产环境 10% 采样） |

> **采样策略建议**：
>
> - 开发环境：100% 采样（方便调试）
> - 生产环境：10% 采样（降低存储和性能开销，出错时通过 `X-Request-ID` 补充排查）
> - 异常请求：在 error 处理中间件中强制记录完整 Trace（不受采样率影响）
