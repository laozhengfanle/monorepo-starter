# 一、多端用户体系设计

本系统有两类用户，它们的登录方式、权限模型不同：

| 维度       | 后台管理员                      | C端用户                         |
| ---------- | ------------------------------- | ------------------------------- |
| 登录方式   | 用户名+密码                     | 手机号+验证码 / 微信 / Apple    |
| 权限模型   | RBAC（角色+菜单+按钮）          | RBAC（等级+功能权限）           |
| Token 内容 | accountId + userType            | accountId + userType            |
| 权限来源   | Redis 两层缓存（角色级+账户级） | Redis 两层缓存（角色级+账户级） |
| 游客访问   | 不允许                          | 允许（guest 角色控制）          |

## 核心架构：account + account_identity + profile + 按端拆 RBAC

```text
account（统一身份容器）
    ↓ 1:N
account_identity（登录标识，一个账户可有多种登录方式）
    ↓ 1:1
admin_profile / member_profile（各端档案，纯业务数据）
    ↓
admin_role → admin_role_menu → admin_menu（管理端 RBAC）
member_role → member_role_menu → member_menu（C端 RBAC）
```

## 设计原则

- 统一账户表（account）只做身份容器，不存登录标识和密码
- 登录标识（username / phone / email / openid）统一放在 account_identity 表，支持一个账户多种登录方式
- 各端 Profile 表只存业务数据，不再承担登录标识职责
- 各端 RBAC 表完全独立，菜单树、角色体系互不干扰
- 新增一个端（如代理商），加表即可，不影响已有逻辑
- 新增一种登录方式（如微信），INSERT 一行记录即可，零 DDL
- JWT 只存 accountId + userType（不可变），角色/权限/菜单等可变数据全部存 Redis
- user_type 一旦创建不可修改，需要"降级"应新建账户

# 二、数据库设计

# 1、account（统一账户表 — 认证层）

只存储登录凭证和认证信息，不存储业务字段。

```sql
CREATE TABLE account (
  id UUID PRIMARY KEY,  -- UUID v7 由应用层生成（Prisma Client Extension 注入）
  user_type VARCHAR(10) NOT NULL CHECK (user_type IN ('admin', 'member')),
  enabled BOOLEAN DEFAULT true,
  last_login_ip VARCHAR(50),
  last_login_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  created_by UUID DEFAULT NULL,
  updated_by UUID DEFAULT NULL,
  deleted_at TIMESTAMPTZ
);
```

说明：

- account 表只做身份容器，不存登录标识（username / phone / email）和密码
- 登录标识和凭证统一放在 account_identity 表，支持一个账户多种登录方式
- user_type 一旦创建不可修改，原因：修改会导致 profile 关联断裂，JWT 需要重新签发
- deleted_at 用于软删除，权限数据属于核心业务数据，误删后需可恢复
- created_by / updated_by 用于数据审计，记录"谁创建/修改了这条记录"

# 1.5、account_identity（账户认证标识表）

```sql
CREATE TABLE account_identity (
  id UUID PRIMARY KEY,  -- UUID v7 由应用层生成（Prisma Client Extension 注入）
  account_id UUID NOT NULL REFERENCES account(id),
  identity_type VARCHAR(20) NOT NULL,
  identifier VARCHAR(255) NOT NULL,
  credential TEXT DEFAULT NULL,
  verified BOOLEAN DEFAULT false,
  verified_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(identity_type, identifier)
);
```

说明：

- 一个账户可以有多种登录方式（如手机号+微信），每种方式一条记录
- UNIQUE(identity_type, identifier)：全局唯一约束，一个手机号/邮箱/openid 只能关联一个账户
    - 取舍：如果业务需要"同一个人在不同端有不同账户"，可改为 UNIQUE(user_type, identity_type, identifier)
    - 当前选择全局唯一：一个手机号就是一个身份，不应跨端重复。这意味着手机号 13800138000 在 C 端注册后，管理员不能用同一个手机号注册，因为全局只有一个 identity_type='phone' + identifier='13800138000'
- credential 字段格式由 identity_type 隐含决定，不需要额外的 credential_type 字段：
    - username / email → bcrypt 密码 hash（如 `$2b$10$xxx...`）
    - phone → null（验证码是一次性的，不持久化，验证时直接查 Redis 中的验证码）
    - wechat_unionid / wechat_openid / apple → AES 加密的 JSON（含 access_token、refresh_token、expires_at 等，解密后使用）
- OAuth credential 安全说明：
    - refresh_token 泄露比 bcrypt hash 更危险（可直接换 access_token），必须加密存储
    - 使用 AES 对称加密，密钥由环境变量管理，不入库
    - 加密后 credential 字段类型改为 TEXT（加密后的 JSON 比原行长）
- `verified` + `verified_at` 双重验证状态：verified 表示该登录方式是否已验证，verified_at 记录验证完成时间
- verified 初始值由 identity_type 决定：
    - phone（验证码登录）→ verified=true，验证码本身就是验证
    - email（密码注册）→ verified=false，需要后续邮件验证
    - wechat / apple（OAuth）→ verified=true，第三方已验证
    - username（管理员）→ verified=true，由系统创建
- 新增登录方式（如微信、Apple ID）只需 INSERT 一行记录，零 DDL

数据示例：

```text
account_id='uuid-1', identity_type='username', identifier='admin001',  credential='$2b$10$xxx...'           ← bcrypt hash
account_id='uuid-2', identity_type='phone',           identifier='13800138000', credential=null                      ← 验证码不持久化
account_id='uuid-2', identity_type='wechat_openid',  identifier='oXXXX...',    credential='AES_ENCRYPTED...'        ← AES 加密 JSON
account_id='uuid-3', identity_type='email',           identifier='partner@co',  credential='$2b$10$yyy...'           ← bcrypt hash
```

# 3、member_profile（C端用户档案）

```sql
CREATE TABLE member_profile (
  id UUID PRIMARY KEY,  -- UUID v7 由应用层生成（Prisma Client Extension 注入）
  account_id UUID NOT NULL UNIQUE,
  phone VARCHAR(20) DEFAULT NULL,
  nickname VARCHAR(50) DEFAULT '',
  avatar VARCHAR(255) DEFAULT '',
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  created_by UUID DEFAULT NULL,
  updated_by UUID DEFAULT NULL,
  deleted_at TIMESTAMPTZ
);
```

说明：

- 登录标识（phone）已移到 account_identity 表，profile 的 phone 只是联系方式
- phone 从 NOT NULL UNIQUE 改为 DEFAULT NULL：微信登录的用户可能不填手机号
- C端用户的等级由 member_role 决定，不再在 profile 上存 level 字段
- 不同等级（游客 / 普通用户 / VIP / SVIP）拥有不同的功能权限，通过 RBAC 控制
- 游客（未登录）自动应用 guest 角色的权限，不需要在 account 表中创建记录

### 审计字段说明

- 所有核心表（account、各端 profile、各端 role、各端 menu）均包含 created_by / updated_by 字段，用于数据审计
- 关联表（account_role、role_menu、account_menu）不需要审计字段，因为它们只是关联关系
- created_by 在两端的语义不同：admin 端有意义（管理员由超管创建），member 端常为 NULL（用户自己注册）。统一保留字段以保持结构一致，不在不同 profile 表间搞字段差异
- admin_profile 不再使用 is_super_admin 字段，超级管理员通过 admin_role 中 code='super_admin' 的角色来标识

# 2.1、数据库索引设计

完整索引策略见 [数据库设计.md](./数据库设计.md)。关键点：

- **account_identity.account_id** — 最高频，每次 `me` 查询都要查用户绑定的登录方式
- **各 menu 表的 parent_id** — 菜单树查询核心
- **关联表 UNIQUE 约束** — `UNIQUE(account_id, role_id)` 覆盖 `WHERE account_id=?` 查询（account_id 是复合索引前缀列）；但 `WHERE role_id=?` 查询无法利用该索引，需单独加索引
- **account_identity 复合索引** — 频繁按 `identity_type + identifier` 查询登录标识，UNIQUE 约束已覆盖；按 `account_id` 查用户所有登录方式，需 `idx_identity_account(account_id)`

# 三、密码加密策略

使用 bcrypt（NestJS 官方推荐），禁止 MD5 / SHA256 / 明文存储。

密码 hash 存储在 `account_identity.credential` 字段（identity_type 为 username 或 email 时）。

```typescript
import * as bcrypt from 'bcrypt';

// 注册时：哈希密码，存入 account_identity.credential
const saltOrRounds = 10; // 盐值轮数
const hash = await bcrypt.hash(password, saltOrRounds);

// 登录时：验证密码，从 account_identity.credential 取出 hash 对比
// bcrypt.compare(用户输入的明文, 数据库中存储的 hash)
const isMatch = await bcrypt.compare(inputPassword, hash);
```

> **bcrypt rounds 说明**：当前 rounds=10（约 70ms/次哈希），是性能与安全的平衡点。生产环境建议在上线前用实际硬件做压测，如单次哈希耗时 < 250ms 则考虑升级到 rounds=12（约 280ms/次）。存量用户无需强制重置密码，采用"渐进式重新哈希"——用户下次登录时用新 rounds 重新哈希并更新数据库。详见 [安全防护.md](./安全防护.md) 第 25 项。

# 四、多端登录设计

## 接口分离策略

原则：**登录接口必须分离，profile 接口可共用，业务接口共用。**

```text
登录接口（必须分离）：
  POST /admin/auth/login      → AdminAuthService   → 用户名+密码
  POST /member/auth/sms       → MemberAuthService  → 手机号+验证码
  POST /member/auth/wechat    → MemberAuthService  → 微信OAuth

  原因：各端登录参数完全不同（username vs phone vs openid），必须独立

profile 接口（共用）：
  GraphQL `me` 查询          → 根据 JWT 中的 userType 自动返回对应端 profile
  POST /auth/logout           → 根据 JWT 中的 userType 清除对应 Redis

  原因：输入只有 JWT，内部按 userType 分流即可，不需要重复写三遍

业务接口（共用）：
  GET /products               → Guard 按 userType 决定返回什么数据
  GET /orders                 → Guard 按 userType 决定数据范围

  原因：输入输出相同，只是权限/数据范围不同，Guard 控制即可
```

## 登录接口分离

各端登录接口独立，避免"C端用户误撞管理后台登录接口"：

```http
POST /admin/auth/login       → AdminAuthService   → 签发 admin token
POST /member/auth/sms        → MemberAuthService  → 签发 member token
POST /member/auth/wechat     → MemberAuthService  → 签发 member token（微信登录）
```

## 登录流程

```text
管理员登录 POST /admin/auth/login { username, password }
  → 查 account_identity WHERE identity_type='username' AND identifier=?
  → 取 account_id → 验证 bcrypt 密码
  → 签发 JWT: { sub: accountId, userType: 'admin' }

C端手机号登录 POST /member/auth/sms { phone, code }
  → 查 account_identity WHERE identity_type='phone' AND identifier=?
  → 取 account_id → 验证短信验证码
  → 签发 JWT: { sub: accountId, userType: 'member' }

C端微信登录 POST /member/auth/wechat { code }
  → 微信 OAuth 换取 openid / unionid
  → 查 account_identity WHERE identity_type='wechat_openid' AND identifier=openid
  → 找到 → 取 account_id → 签发 JWT（老用户）
  → 没找到 → 事务创建 account + account_identity + member_profile → 签发 JWT（新用户）
```

## 注册流程（必须使用事务）

新用户注册需要三条 INSERT，必须包在数据库事务里：

```typescript
// C端手机号注册
await db.$transaction(async (tx) => {
    // 1. 创建账户（身份容器）
    const account = await tx.account.create({
        data: { user_type: 'member', enabled: true },
    });
    // 2. 创建登录标识
    await tx.accountIdentity.create({
        data: {
            account_id: account.id,
            identity_type: 'phone',
            identifier: '13800138000',
            credential: null, // 验证码登录不存密码
            verified: true, // 验证码本身就是验证
        },
    });
    // 3. 创建业务档案
    await tx.member_profile.create({
        data: { account_id: account.id, phone: '13800138000', nickname: '用户xxx' },
    });
});
```

```typescript
// C端微信注册
await db.$transaction(async (tx) => {
    const account = await tx.account.create({
        data: { user_type: 'member', enabled: true },
    });

            account_id: account.id,
            identity_type: 'wechat_openid',
            identifier: openid,
            credential: AES.encrypt(JSON.stringify({ refresh_token, expires_at: 1234567890 }), SECRET_KEY),
            verified: true, // 第三方已验证
        },
    });

```

## 密码重置流程

> 完整链路：申请重置 → 身份验证 → 下发重置 token → 执行重置 → 旧 session 失效

```text
1. 用户提交手机号/邮箱 → 发送验证码（purpose=reset_password）
2. 用户提交 验证码 + 新密码
3. 服务端校验验证码 → 哈希新密码 → 更新 account_identity.credential
4. 清除该用户所有 Redis session（强制重新登录）
```

### 接口设计

```text
# Step 1: 发送重置验证码（复用短信/邮件服务）
POST /member/auth/send-code  { phone, purpose: 'reset_password' }

# Step 2: 验证码 + 新密码 → 重置
POST /member/auth/reset-password  { phone, code, newPassword }
```

### 安全约束

- 验证码 TTL 5 分钟，验证成功后立即删除（一次性消费）
- 新密码必须满足安全防护.md 的强度策略（≥8 位，必须包含字母+数字）
- 重置成功后，清除该用户所有 Redis session key（`mono:auth:*` + `mono:role:*`），强制所有端重新登录
- 同一手机号 60 秒内只能发一次验证码，每日上限 5 次
- 验证码错误 5 次后锁定该手机号 30 分钟

**手机号正则校验**（Zod schema 强制）：

```ts
// packages/shared/src/schemas/regex.ts
export const PHONE_REGEX = /^1[3-9]\d{9}$/;
export const PHONE_REGEX_DESCRIPTION = '中国大陆 11 位手机号（13-19 段）';

// 校验失败时统一错误码：20001 "手机号格式不正确"
```

> **为什么用 `1[3-9]\d{9}` 而不是完整号段表**：完整号段表需要 200+ 行业号段（如 16x / 17x / 19x），维护成本高且易遗漏；`1[3-9]\d{9}` 覆盖 130-199 共 70 个号段，命中 99.9% 真实号段且无明显误判。

### 错误码

| code  | 说明                                 |
| ----- | ------------------------------------ |
| 30004 | 验证码过期                           |
| 30005 | 验证码错误                           |
| 30007 | 验证码尝试次数超限                   |
| 21001 | 用户已禁用（禁用账号不允许重置密码） |

## JWT Payload 设计

```typescript
// 所有端的 JWT 只存这两个字段，其他全部从 Redis 读取
{ sub: accountId, userType: 'admin' | 'member' }

// JWT 标准声明（由 jsonwebtoken 库自动注入，不需要手动设置）：
// - iat: 签发时间（issued at）
// - exp: 过期时间（expiration），由 JWT_ACCESS_TTL / JWT_REFRESH_TTL 控制
// - jti: JWT 唯一 ID（用于 refresh token 重用检测，见安全防护.md）
```

说明：

- JWT 只存 accountId + userType，这是不可变信息
- 角色、权限、菜单、企业ID 等可变数据全部从 Redis 读取
- 原因：JWT 签发后无法撤销，如果角色/权限放 JWT，修改后旧 token 仍携带旧权限
- userType 必须放 JWT：Guard 需要知道走哪个权限校验逻辑，如果只有 accountId 每次都要查数据库

# 五、核心原则

# 1、account 只做身份容器

account 表不存登录标识和密码，只存 user_type、enabled 和审计字段。

# 2、登录标识与业务数据分离

登录标识（username / phone / email / openid）统一放在 account_identity 表，profile 只存业务数据。新增登录方式只需 INSERT 一行记录，零 DDL。

# 3、user_type 不可变

user_type 在账户创建后不可修改。需要"降级"或"转端"应新建账户，避免 profile 关联断裂和 JWT 重新签发。

# 4、JWT 只存不可变信息

JWT 只存 accountId + userType。角色、权限、菜单等可变数据从 Redis 读取，确保修改后即时生效。

# 5、接口分离原则

登录接口必须分离（各端参数不同），profile 和业务接口共用（Guard 按 userType 分流）。

# 6、注册必须使用事务

account + account_identity + profile 三条 INSERT 必须包在 `db.$transaction` 里，防止脏数据。
