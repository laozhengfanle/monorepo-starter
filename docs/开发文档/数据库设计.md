## 主键策略：UUID v7

所有表 `id` 统一使用 UUID v7。详见 [技术架构.md](./技术架构.md#主键策略uuid-v7)。

```ts
// packages/shared/src/id.ts
import { uuidv7 } from 'uuidv7';
export const newId = () => uuidv7();
```

应用层生成，不依赖 PostgreSQL 扩展（`pg_uuidv7`）。Prisma Client Extensions 自动注入：

```ts
// apps/server/src/common/prisma.ts
const prisma = new PrismaClient().$extends({
    query: {
        $allModels: {
            async create({ args, query }) {
                args.data.id ??= newId();
                return query(args);
            },
            async createMany({ args, query }) {
                if (Array.isArray(args.data)) {
                    args.data.forEach((row: any) => {
                        row.id ??= newId();
                    });
                }
                return query(args);
            },
        },
    },
});
```

完整实现见下方 [Prisma Client Extensions](#prisma-client-extensions) 章节。

## 四、审计日志（1 张表）

### 7. audit_log — 安全审计日志表

```sql
CREATE TABLE audit_log (
  id            UUID PRIMARY KEY,  -- UUID v7 由应用层生成（Prisma Client Extension 注入）
  account_id    UUID,                           -- 操作者，NULL 表示匿名操作（不设外键：审计日志不应因用户删除而级联影响）
  action        VARCHAR(50) NOT NULL,            -- 操作类型
  resource_type VARCHAR(50),                     -- 资源类型
  resource_id   UUID,                            -- 资源 ID
  detail        JSONB,                           -- 变更前后快照（结构自由）
  ip            VARCHAR(50),                     -- 客户端 IP
  user_agent    TEXT,                            -- User-Agent
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_audit_log_account ON audit_log(account_id);
CREATE INDEX idx_audit_log_action ON audit_log(action);
CREATE INDEX idx_audit_log_resource ON audit_log(resource_type, resource_id);
CREATE INDEX idx_audit_log_created_at ON audit_log(created_at);
```

**保留期与清理策略**：

| 维度       | 策略                                                                                                |
| ---------- | --------------------------------------------------------------------------------------------------- |
| 保留期     | **90 天**（合规要求：GDPR / 等保 2.0 三级要求至少 90 天日志留存）                                   |
| 清理方式   | **物理删除**（无软删除标记）：`DELETE FROM audit_log WHERE created_at < now() - INTERVAL '90 days'` |
| 清理触发   | `apps/server/src/tasks/cleanup.task.ts` 中 `@Cron('0 3 * * *')` 每天凌晨 3 点触发                   |
| 备份归档   | 清理前 7 天导出到 OSS 冷归档（`oss://mono-audit-log-archive/yyyy-MM-dd.parquet`），保留 1 年        |
| 高水位告警 | 当 `audit_log` 表行数 > 500 万行时，钉钉/企微机器人告警（DBA 介入评估是否需要扩容或缩短保留期）     |

| `action` 枚举        | 记录时机                   |
| -------------------- | -------------------------- |
| `login_success`      | 登录成功                   |
| `login_failed`       | 登录失败                   |
| `password_changed`   | 密码修改                   |
| `role_assigned`      | 角色分配                   |
| `role_revoked`       | 角色撤销                   |
| `permission_changed` | 权限变更                   |
| `account_created`    | 创建账号                   |
| `account_disabled`   | 禁用/启用账号              |
| `account_deleted`    | 软删除账号                 |
| `token_refreshed`    | Token 刷新                 |
| `token_reused`       | Token 重用检测（疑似盗用） |
| `account_updated`    | 更新管理员资料             |
| `account_enabled`    | 启用账号                   |
| `role_created`       | 创建角色                   |
| `role_updated`       | 更新角色                   |
| `role_deleted`       | 删除角色                   |
| `menu_created`       | 创建菜单                   |
| `menu_updated`       | 更新菜单                   |
| `menu_deleted`       | 删除菜单                   |
| `config_updated`     | 更新系统配置               |
| `file_uploaded`      | 上传文件                   |
| `file_deleted`       | 删除文件                   |

> `detail` 用 JSONB 存储变更前后快照，灵活适应不同 action 的数据结构。例如角色变更时记录 `{"before": ["role_a"], "after": ["role_a", "role_b"]}`。

## 五、文件管理（1 张表）

### 8. upload_file — 文件元数据表

每次上传成功后在数据库留一条记录，用于文件追踪和定时清理。

```sql
CREATE TABLE upload_file (
  id            UUID PRIMARY KEY,  -- UUID v7 由应用层生成
  account_id    UUID NOT NULL REFERENCES account(id) ON DELETE RESTRICT,
  original_name VARCHAR(255) NOT NULL,          -- 原始文件名（上传时的名字）
  stored_name   VARCHAR(255) NOT NULL,          -- 存储后文件名（UUID 重命名，防冲突）
  mime_type     VARCHAR(100) NOT NULL,          -- MIME 类型（如 image/png）
  size          BIGINT NOT NULL,                -- 文件大小（byte）
  storage       VARCHAR(20) NOT NULL DEFAULT 'local',  -- 'local' | 's3'
  folder        VARCHAR(100) NOT NULL DEFAULT 'files', -- 存储目录（avatars / files）
  url           VARCHAR(500) NOT NULL,           -- 访问 URL
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at    TIMESTAMPTZ                      -- 软删除标记（定时任务 30 天后物理删除）
);

CREATE INDEX idx_upload_file_account ON upload_file(account_id);
CREATE INDEX idx_upload_file_created_at ON upload_file(created_at);
CREATE INDEX idx_upload_file_mime_type ON upload_file(mime_type);
CREATE INDEX idx_upload_file_deleted_at ON upload_file(deleted_at) WHERE deleted_at IS NOT NULL;
```

| 字段            | 类型         | 说明                                                                  |
| --------------- | ------------ | --------------------------------------------------------------------- |
| `id`            | UUID v7      | 主键                                                                  |
| `account_id`    | UUID         | 上传者 account ID                                                     |
| `original_name` | VARCHAR(255) | 用户上传时的文件名                                                    |
| `stored_name`   | VARCHAR(255) | 存储后的 UUID 文件名（如 `550e8400-e29b-11d4-a716-446655440000.jpg`） |
| `mime_type`     | VARCHAR(100) | 文件类型（用于过滤搜索和管理端图标）                                  |
| `size`          | BIGINT       | 字节数（用于展示和限额计算）                                          |
| `storage`       | VARCHAR(20)  | 存储方式：`local` 或 `s3`                                             |
| `folder`        | VARCHAR(100) | 所在目录（`avatars`、`files` 等）                                     |
| `url`           | VARCHAR(500) | 完整访问 URL                                                          |
| `deleted_at`    | TIMESTAMPTZ  | 软删除，不立即物理删除文件                                            |

### 物理删除策略

软删除后，定时任务（见 [部署运维.md](./部署运维.md)）在 30 天后物理删除：

```typescript
// 每天凌晨 3:00 —— 物理删除超过 30 天的软删除文件
@Cron('0 3 * * *')
async cleanupDeletedFiles() {
  const threshold = new Date();
  threshold.setDate(threshold.getDate() - 30);

  const expired = await prisma.uploadFile.findMany({
    where: { deleted_at: { lte: threshold } },
  });

  for (const file of expired) {
    // 从存储中删除（本地磁盘或 S3）
    await this.storage.delete(file.storedName, file.folder);
    // 从数据库删除记录
    await prisma.uploadFile.delete({ where: { id: file.id } });
  }
}
```

## 六、系统配置（2 张表）

### 9. system_config — 系统动态配置表

存储后台可动态修改的配置项（短信服务商、存储配置、邮件配置等），管理后台修改后即时生效（Redis 缓存失效策略见 [缓存设计.md](./缓存设计.md)）。

```sql
CREATE TABLE system_config (
  id            UUID PRIMARY KEY,  -- UUID v7 由应用层生成
  key           VARCHAR(100) NOT NULL,          -- 配置键（如 sms.provider、storage.driver）
  value         JSONB NOT NULL,                 -- 配置值（JSON 结构，灵活适应不同配置类型）
  remark        VARCHAR(255),                   -- 配置备注（管理后台展示用）
  updated_by    UUID,                           -- 最后更新者 account ID
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at    TIMESTAMPTZ                    -- 软删除时间戳
);

CREATE UNIQUE INDEX uq_system_config_key ON system_config(key);
CREATE INDEX idx_system_config_updated_at ON system_config(updated_at);
```

> 注：`updated_by` 在 schema 中无显式外键约束，service 层保证其指向有效的 `account.id`。

| 字段         | 类型         | 说明                                                                                         |
| ------------ | ------------ | -------------------------------------------------------------------------------------------- |
| `id`         | UUID v7      | 主键                                                                                         |
| `key`        | VARCHAR(100) | 配置键，全局唯一。命名规范：`{domain}.{subdomain}`，如 `sms.provider`                        |
| `value`      | JSONB        | 配置值，结构由各服务方定义（见 [短信服务方案.md](./短信服务方案.md) 的 `SmsProviderConfig`） |
| `remark`     | VARCHAR(255) | 配置备注，管理后台展示用                                                                     |
| `updated_by` | UUID         | 最后更新者 account ID                                                                        |

### 10. verification_code — 验证码审计表

记录每次验证码发送的审计日志，用于追溯和防刷统计。**不存明文验证码**（明文只存 Redis，TTL 过期自动清除）。

```sql
CREATE TABLE verification_code (
  id            UUID PRIMARY KEY,  -- UUID v7 由应用层生成
  identifier    VARCHAR(100) NOT NULL,          -- 接收标识（手机号 / 邮箱）
  code          VARCHAR(10) NOT NULL DEFAULT '******',  -- 审计表不存明文，只记录"发送了"
  purpose       VARCHAR(50) NOT NULL,           -- 用途：login / register / reset_password / bind_phone / verify_email
  channel       VARCHAR(20) NOT NULL DEFAULT 'sms',  -- 渠道：sms / email
  ip            VARCHAR(50),                    -- 客户端 IP
  status        VARCHAR(20) NOT NULL DEFAULT 'sent',  -- 状态：sent / verified / expired / failed
  expires_at    TIMESTAMPTZ NOT NULL,           -- 过期时间
  verified_at   TIMESTAMPTZ,                    -- 验证通过时间
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_verification_code_identifier ON verification_code(identifier);
CREATE INDEX idx_verification_code_purpose ON verification_code(purpose);
CREATE INDEX idx_verification_code_created_at ON verification_code(created_at);
CREATE INDEX idx_verification_code_ip ON verification_code(ip);
CREATE INDEX idx_verification_code_status ON verification_code(status);
```

**保留期与清理策略**：

| 维度       | 策略                                                                                                        |
| ---------- | ----------------------------------------------------------------------------------------------------------- |
| 保留期     | **30 天**（防刷统计和追溯足够；过老的验证码审计无业务价值）                                                 |
| 清理方式   | **物理删除**（无软删除标记）：`DELETE FROM verification_code WHERE created_at < now() - INTERVAL '30 days'` |
| 清理触发   | `apps/server/src/tasks/cleanup.task.ts` 中 `@Cron('0 3 * * *')` 每天凌晨 3 点（与 audit_log 同一任务）      |
| Redis 配合 | Redis 中的明文验证码 TTL（5 分钟）到期自动清除，无需应用清理                                                |
| 高水位告警 | 当 `verification_code` 表行数 > 50 万行时告警（DBA 介入评估是否需要扩容或缩短保留期）                       |

| 字段          | 类型         | 说明                                                                                    |
| ------------- | ------------ | --------------------------------------------------------------------------------------- |
| `id`          | UUID v7      | 主键                                                                                    |
| `identifier`  | VARCHAR(100) | 接收标识（手机号 E.164 格式或邮箱地址）                                                 |
| `code`        | VARCHAR(10)  | 固定存 `******`，不存明文验证码（安全合规要求）                                         |
| `purpose`     | VARCHAR(50)  | 用途：`login`、`register`、`reset_password`、`bind_phone`、`verify_email`               |
| `channel`     | VARCHAR(20)  | 渠道：`sms`（短信）或 `email`（邮件）                                                   |
| `ip`          | VARCHAR(50)  | 客户端 IP，用于防刷统计                                                                 |
| `status`      | VARCHAR(20)  | 状态：`sent`（已发送）、`verified`（已验证）、`expired`（已过期）、`failed`（发送失败） |
| `expires_at`  | TIMESTAMPTZ  | 验证码过期时间                                                                          |
| `verified_at` | TIMESTAMPTZ  | 验证通过时间（验证成功时更新）                                                          |

### 审计查询示例

```sql
-- 查某手机号今天发送了多少次验证码
SELECT COUNT(*) FROM verification_code
WHERE identifier = '+8613800138000'
  AND created_at >= CURRENT_DATE;

-- 查某 IP 最近 1 小时的发送次数（防刷监控）
SELECT COUNT(*) FROM verification_code
WHERE ip = '192.168.1.100'
  AND created_at >= now() - interval '1 hour';

-- 查发送失败的记录（排查服务商问题）
SELECT * FROM verification_code
WHERE status = 'failed'
  AND created_at >= now() - interval '24 hours'
ORDER BY created_at DESC;
```

> **与 Redis 的分工**：Redis 存明文验证码（短 TTL，用于实时校验），`verification_code` 表存审计记录（长期保留，用于追溯和统计）。两者互补，不重复。

## 已删除记录的可见性策略

> 本章节说明业务表（带 `deleted_at` 软删除标记）的 unique 约束在软删场景下的处理策略，配套 SPEC `.trae/specs/soft-delete-visibility-and-policy/`。

### 背景

当前所有业务表都使用 `deleted_at` 软删除（详见 [Prisma Client Extensions](#prisma-client-extensions)），但 UI 默认只展示 `deleted_at IS NULL` 的行。当用户**重复创建**业务标识符类字段（如角色 `code`、登录 `identifier`）撞到 DB 层 `@unique` 约束时，后端冒泡一个 Prisma 原始错误（`P2002`），前端只能看到「GraphQL internal error 10999」之类的兜底文案——用户**根本不知道**是被某条已软删的旧记录占着唯一坑位。

> 典型 case：先创建 `code: "guest"` 角色 → 软删 → 再创建 `code: "guest"` → 撞 unique → 用户看到 10999 错误，一脸懵。

正解不是 DB 层放开 unique，而是在 UX 层解决：让用户能"看见"软删记录 → 主动决策"复用 / 物理删旧 / 改新 code"。同时给软删视图提供"彻底删除"能力，让"被软删的旧记录"能真正释放出 unique 坑位。

### 三方案对比

| 方案                                     | 描述                                                                                                                                                | 优点                                                                                | 缺点                                                                                                                                             |
| ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **A. 全局 `@unique`**                    | DB 层强制 unique，软删后记录仍占坑                                                                                                                  | 简单；PostgreSQL 原生支持；无迁移成本                                               | 软删除的"保留历史可恢复"语义被破坏（用户看不见软删记录就以为是 bug）；撞 unique 时用户不知道为什么                                               |
| **B. Partial unique index**              | DB 层 unique 仅约束 `WHERE deleted_at IS NULL`（如 `CREATE UNIQUE INDEX ... ON admin_role(code) WHERE deleted_at IS NULL`）                         | 软删后允许重名同 `code`                                                             | 破坏软删除的"可恢复"语义（恢复软删记录时会反撞 unique，事务回滚）；PostgreSQL 特定写法，Prisma `@unique` 不直接支持，迁移复杂；schema 表达力下降 |
| **C. 列表可见性 + 彻底删除**（**采用**） | 保留全局 `@unique`，UI 提供筛选显示已删除/全部；给软删行提供「恢复正常」「彻底删除」两个操作；service 层撞 unique 时给明确 `ConflictException` 文案 | 简单（DB 层零改动）；符合软删本意（保留历史 + 可恢复）；用户能主动决策；UX 引导清晰 | 需要前后端配合（list Query 加 `includeDeleted`、新增两个 mutation、UI 三种视图切换）                                                             |

### 为什么选 C 方案

1. **软删除的核心价值是"保留历史可恢复"**。B 方案的 partial unique 把这条破坏掉了——恢复一个 6 个月前软删的角色，结果发现 `code` 已被新记录占着，回滚失败，业务用户一脸懵。C 方案保持"软删 = 占坑但可恢复"这个直觉，UX 层用「先彻底删除旧的，再创建新的」或「改新 code」两条路径主动解决。
2. **C 方案是"三修"，不是"一修"**：
    - **DB 层**：保持简单（`@unique` 一行），无特殊索引，迁移工具链零负担
    - **Service 层**：把裸 `P2002` 翻译成 `ConflictException`，并区分"撞活跃"vs"撞软删"两种文案（前者直接报错，后者引导去"显示已删除"视图）
    - **UI 层**：让用户能"看见"软删记录，主动决策「恢复正常 / 彻底删除 / 改用新 code」
3. **撞 unique 时的根因不是 DB 设计问题**，而是用户没有"软删占用"的可见性。B 方案只修了第 1 层，没修第 2、3 层，治标不治本；C 方案三修。
4. **可扩展性**：未来要加新的"业务标识符类"字段（菜单 `permissionCode`、字典 `dictCode` 等），C 方案的模板可零成本复用；B 方案每加一列就要加一个 partial unique 索引，迁移成本线性增长。

### Service 层撞 unique 标准错误处理模板

以 `AdminRoleService.create` 为例（含 `code` 字段）。其他业务表（`MemberRole`、`AccountIdentity` 等）的 unique 字段处理完全套这个模板，只需替换"业务字段名"和"文案"。

```typescript
/**
 * 标准模板：createXxx 撞 unique 时的明确错误处理
 *
 * 三步走：
 *   1) 预查 findByXxx(deletedAt: null)  →  撞到活跃行 → 抛 ConflictException
 *   2) 预查 findByXxx(deletedAt: not null)  →  撞到软删行 → 抛 ConflictException（带引导文案）
 *   3) prisma.create 包 try/catch  →  兜底 TOCTOU race（步骤 1、2 没撞到，
 *      但并发场景下两个请求都过了预查，DB 层 @unique 触发 P2002）
 */
import { BadRequestException, ConflictException, Injectable, Logger, NotFoundException } from '@nestjs/common';
import { Prisma } from '@prisma/client'; // 用于 instanceof 判断

@Injectable()
export class AdminRoleService {
    private readonly logger = new Logger(AdminRoleService.name);

    constructor(private readonly prisma: PrismaService) {}

    /** 预查：按 code 查活跃行（deletedAt: null） */
    async findActiveByCode(code: string) {
        return this.prisma.client.adminRole.findFirst({
            where: { code, deletedAt: null },
        });
    }

    /** 预查：按 code 查已软删行（deletedAt IS NOT NULL） */
    async findDeletedByCode(code: string) {
        return this.prisma.client.adminRole.findFirst({
            where: { code, deletedAt: { not: null } },
        });
    }

    /**
     * 创建角色（C 方案标准模板）
     * @param input CreateRoleInput — 业务入参
     * @param operatorId 操作者账户 ID（用于审计日志）
     */
    async createRole(
        input: { name: string; code: string; description?: string; enabled?: boolean },
        operatorId?: string,
    ) {
        // 步骤 1：预查撞到活跃行
        const active = await this.findActiveByCode(input.code);
        if (active) {
            throw new ConflictException(`角色编码 ${input.code} 已被使用`);
        }

        // 步骤 2：预查撞到软删行（引导用户去"显示已删除"视图处理）
        const deleted = await this.findDeletedByCode(input.code);
        if (deleted) {
            throw new ConflictException(
                `角色编码 ${input.code} 已被使用（旧记录已删除，可在「显示已删除」视图下找到并彻底删除后重试）`,
            );
        }

        // 步骤 3：try/catch 兜底 TOCTOU race（步骤 1、2 并发竞态）
        try {
            const role = await this.prisma.client.adminRole.create({
                data: {
                    id: newId(),
                    name: input.name,
                    code: input.code,
                    description: input.description ?? '',
                    enabled: input.enabled ?? true,
                },
            });

            // 写审计日志（详见审计日志章节）
            await this.auditService.record({
                accountId: operatorId || '',
                action: 'role_created',
                resourceType: 'admin_role',
                resourceId: role.id,
                detail: { code: input.code, name: input.name },
            });

            return role;
        } catch (err) {
            // 并发场景：两个请求都通过了步骤 1、2，但 DB 层 @unique 触发 P2002
            if (err instanceof Prisma.PrismaClientKnownRequestError && err.code === 'P2002') {
                // 二次区分：再查一次当前占用者是活跃还是软删
                const racer = await this.findActiveByCode(input.code);
                if (racer) {
                    throw new ConflictException(`角色编码 ${input.code} 已被使用`);
                }
                throw new ConflictException(
                    `角色编码 ${input.code} 已被使用（旧记录已删除，可在「显示已删除」视图下找到并彻底删除后重试）`,
                );
            }
            // 非 P2002 异常 → 抛给全局过滤器统一处理
            throw err;
        }
    }
}
```

**关键点**：

- `ConflictException`（来自 `@nestjs/common`）会被全局异常过滤器（`GlobalExceptionFilter`）翻译成 409 HTTP 状态码 + 业务错误码（不是 10999）
- 步骤 3 的 `try/catch P2002` 是**兜底**而非主路径：正常情况下步骤 1、2 已经能拦住 99% 的冲突；只有并发竞态（两个请求都过了预查）才会走到 catch
- "撞软删"的文案要明确引导用户**去哪里处理**（「显示已删除」视图 + 「彻底删除」按钮），否则用户不知道下一步该干啥
- `update` 操作如果改 unique 字段（如 `MemberProfile.phone`），同样要套这个模板：预查活跃行 → 预查软删行 → try/catch P2002

### 18 张表 unique 字段策略判定表

>
> - list Query 加 `includeDeleted: Boolean = false`（默认仅活跃）
> - Mutation `hardDeleteXxx(id: ID!)`（彻底删除，前置校验 `deletedAt IS NOT NULL`）
> - Mutation `restoreXxx(id: ID!)`（恢复，前置校验 + 二次 unique 冲突预查）
> - 审计日志新增 `xxx_hard_deleted` / `xxx_restored` action 枚举

## 字段规范

所有表遵循统一约定。详见 [命名规范.md](./命名规范.md)。

### SQL 风格约定

保证所有 DDL 一致：

| 场景     | 规则                                                     | 示例                                            |
| -------- | -------------------------------------------------------- | ----------------------------------------------- |
| 字符串列 | 有默认值用 `DEFAULT ''`，无默认值用 `NOT NULL`（不混用） | `VARCHAR(100) NOT NULL`                         |
| 外键     | 统一声明 `ON DELETE RESTRICT`（防止误删级联）            | `REFERENCES account(id) ON DELETE RESTRICT`     |
| 布尔列   | `BOOLEAN DEFAULT true/false`                             | `enabled BOOLEAN DEFAULT true`                  |
| 时间列   | `TIMESTAMPTZ NOT NULL DEFAULT now()`                     | `created_at TIMESTAMPTZ NOT NULL DEFAULT now()` |
| 软删除列 | `TIMESTAMPTZ`（可空，不设默认值）                        | `deleted_at TIMESTAMPTZ`                        |

### 必备字段

| 字段         | 类型        | 说明                             |
| ------------ | ----------- | -------------------------------- |
| `id`         | UUID        | 主键，应用层 `uuidv7()` 生成     |
| `created_at` | TIMESTAMPTZ | 创建时间，`DEFAULT now()`        |
| `updated_at` | TIMESTAMPTZ | 更新时间，应用层更新             |
| `deleted_at` | TIMESTAMPTZ | 软删除标记（可选，业务数据必加） |

### 审计字段（配置/权限表）

| 字段         | 类型 | 说明                  |
| ------------ | ---- | --------------------- |
| `created_by` | UUID | 创建者 account ID     |
| `updated_by` | UUID | 最后更新者 account ID |

### 命名约定

| 对象     | 规范                    | 示例                                  |
| -------- | ----------------------- | ------------------------------------- |
| 表名     | snake_case，单数        | `account`, `admin_role`               |
| 主键     | `id`                    | —                                     |
| 外键     | `{referenced_table}_id` | `account_id`, `company_id`            |
| 索引     | `idx_{table}_{column}`  | `idx_account_user_type`               |
| 唯一约束 | `uq_{table}_{columns}`  | `uq_account_identity_type_identifier` |
| 枚举     | VARCHAR + CHECK         | `CHECK (user_type IN (...))`          |

## 索引策略

### 必须建索引的场景

| 场景       | 示例                          | 索引                                                               |
| ---------- | ----------------------------- | ------------------------------------------------------------------ |
| 外键       | `account_identity.account_id` | `CREATE INDEX ON account_identity(account_id)`                     |
| 查询条件   | `WHERE user_type = 'admin'`   | `CREATE INDEX ON account(user_type)`                               |
| 软删除过滤 | `WHERE deleted_at IS NULL`    | `CREATE INDEX ON account(deleted_at) WHERE deleted_at IS NOT NULL` |
| 唯一查找   | `identity_type + identifier`  | `UNIQUE(identity_type, identifier)`                                |
| 排序字段   | `ORDER BY created_at DESC`    | UUID v7 天然有序，不需额外索引                                     |

### 不建索引的场景

| 场景                                        | 原因             |
| ------------------------------------------- | ---------------- |
| 低基数列（如 gender、status 只有 2-3 个值） | 全表扫描可能更快 |
| 小表（< 1000 行）                           | 索引开销大于收益 |
| 从不用于 WHERE/ORDER BY/JOIN 的列           | 维护成本         |

## 迁移策略

### 开发环境（本地）

```bash
# schema 变更后
npx prisma migrate dev --name add_xxx_field

# 回滚：直接编辑 migration SQL，然后
npx prisma migrate reset  # 清空数据库重新跑
```

**原则**：`migrate dev` 只在本地使用。生成的 migration 文件提交到 Git。

### 预发布 / 生产环境

```bash
# CI/CD 中执行（永远不用 migrate dev）
npx prisma migrate deploy

# 部署前检查迁移状态
npx prisma migrate status
```

### 迁移文件管理

```text
prisma/migrations/
├── 20250101000000_init/              # 初始 schema
│   └── migration.sql
├── 20250102000000_add_xxx_field/     # 增量迁移
│   └── migration.sql
└── migration_lock.toml               # 锁定 provider 版本
```

**铁律**：

1. **永远不要修改已应用的 migration 文件** — 发现错误就新建一个 migration 修复
2. **永远不要在生产环境用 `migrate dev` 或 `db push`**
3. **migration 文件夹必须提交到 Git**，与代码版本绑定
4. **每个 migration 只做一件事**，命名清晰（`add_xxx`、`rename_xxx`、`drop_xxx`）

### 已应用 Migration 的修复流程

当已应用的 migration 发现 bug 时，**绝不修改原文件**，而是新建一个修复 migration：

```bash
# 错误：修改已应用的 migration
# ✗ 直接编辑 20250102000000_add_xxx_field/migration.sql

# 正确：新建修复 migration
npx prisma migrate dev --name fix_xxx_field_type
```

```sql
-- prisma/migrations/20250103000000_fix_xxx_field_type/migration.sql
-- 修复：将 xxx 字段从 VARCHAR(50) 改为 VARCHAR(100)
ALTER TABLE "account" ALTER COLUMN "xxx" TYPE VARCHAR(100);
```

**生产环境紧急回滚**：

如果上线后发现问题需要回滚：

1. **应用层回滚**：重新部署上一个版本（Docker image tag 回退）
2. **数据库层回滚**：新建一个反向 migration（不删除原 migration）
3. **严禁**：`prisma migrate reset`（会清空所有数据）

```sql
-- 回滚 migration 示例
ALTER TABLE "account" ALTER COLUMN "xxx" TYPE VARCHAR(50);
```

> **关键原则**：数据库变更只进不退。每次变更都是一个新的 migration 文件，即使回滚也是新文件。这与 Git 的"revert commit"思想一致。

### 基线化（从已有数据库接入）

当 fork 基座的业务项目已有自己的数据库时：

```bash
mkdir -p prisma/migrations/0_init
npx prisma migrate diff \
  --from-empty \
  --to-schema-datamodel prisma/schema.prisma \
  --script > prisma/migrations/0_init/migration.sql
npx prisma migrate resolve --applied 0_init
```

```ts
// prisma/seed.ts
import { PrismaClient } from '@prisma/client';
import { newId } from '@packages/shared'; // 需在 packages/shared/src/id.ts 中实现并从 index.ts re-export

const prisma = new PrismaClient();

async function main() {
    // 1. 创建超管角色
    const superAdminRole = await prisma.adminRole.create({
        data: {
            id: newId(),
            code: 'super_admin',
            name: '超级管理员',
        },
    });

    // 2. 创建超管账号
    const account = await prisma.account.create({
        data: {
            id: newId(),
            userType: 'admin',
```

## Prisma Client Extensions

Prisma 最新版本使用 **Client Extensions** 方式实现中间件逻辑（替代旧版 `$use()` API）。

### 自动 UUID v7

```ts
// apps/server/src/common/prisma.ts
// 推荐：使用 Prisma Client Extensions
const prisma = new PrismaClient().$extends({
    query: {
        $allModels: {
            async create({ args, query }) {
                args.data.id ??= newId();
                return query(args);
            },
            async createMany({ args, query }) {
                if (Array.isArray(args.data)) {
                    args.data.forEach((row: any) => {
                        row.id ??= newId();
                    });
                }
                return query(args);
            },
        },
    },
});
```

### 软删除过滤

```ts
// Prisma Client Extensions 方式
// 注意：query 钩子的 params.action 是只读的，不能在回调中改写
// 正确做法：在 delete 钩子中直接调用 update
const prisma = new PrismaClient().$extends({
    name: 'softDelete',
    query: {
        $allModels: {
            // 自动过滤已删除记录
            async findUnique({ args, query, model }) {
                args.where = { ...args.where, deleted_at: null };
                return query(args);
            },
            async findFirst({ args, query, model }) {
                args.where = { ...args.where, deleted_at: null };
                return query(args);
            },
            async findMany({ args, query, model }) {
                args.where = { ...args.where, deleted_at: null };
                return query(args);
            },
            // delete → 改为 update deleted_at（不能改写 action，需直接调用 update）
            async delete({ args, model }) {
                // model.name 返回 PascalCase（如 AdminProfile），Prisma Client 属性是 camelCase（如 adminProfile）
                const modelName = (model.name.charAt(0).toLowerCase() + model.name.slice(1)) as keyof typeof prisma;
                const modelDelegate = prisma[modelName] as any;
                return modelDelegate.update({
                    where: args.where,
                    data: { deleted_at: new Date() },
                });
            },
            // deleteMany → 改为 updateMany
            async deleteMany({ args, model }) {
                const modelName = (model.name.charAt(0).toLowerCase() + model.name.slice(1)) as keyof typeof prisma;
                const modelDelegate = prisma[modelName] as any;
                return modelDelegate.updateMany({
                    where: args.where,
                    data: { deleted_at: new Date() },
                });
            },
        },
    },
});
```

### 查看已删除数据

管理后台需要查看已软删除的记录（如回收站功能），默认的软删除过滤会阻止这类查询。推荐方案：创建无软删除扩展的 Prisma Client 实例用于管理端查询。

```ts
// common/prisma.ts — 导出两个客户端实例
import { PrismaClient } from '@prisma/client';
import { softDeleteExtension } from './soft-delete.extension';
import { uuidv7Extension } from './uuidv7.extension';

// C端：带软删除过滤 + UUID 自动注入
export const prisma = new PrismaClient().$extends(softDeleteExtension).$extends(uuidv7Extension);

// 管理端：仅 UUID 自动注入，不过滤软删除
export const prismaAdmin = new PrismaClient().$extends(uuidv7Extension);
```

```ts
// 管理端 Service 中使用 prismaAdmin
class AdminAccountService {
    constructor(
        @Inject('PRISMA_ADMIN') private readonly prismaAdmin: PrismaClient,
        @Inject('PRISMA') private readonly prisma: PrismaClient,
    ) {}

    // 普通查询：显式过滤未删除
    async listActive() {
        return this.prismaAdmin.account.findMany({
            where: { deleted_at: null },
        });
    }

    // 回收站：查看已删除
    async listDeleted() {
        return this.prismaAdmin.account.findMany({
            where: { deleted_at: { not: null } },
        });
    }
}
```

> **建议**：管理端 Service 统一使用 `prismaAdmin`（仅含 UUID 扩展，不含软删除过滤），在查询时**显式指定** `deleted_at` 条件，避免隐式过滤带来的困惑。C端 Service 使用 `prisma`（含软删除扩展），自动过滤已删除数据。两个客户端通过 NestJS 自定义 Provider 注入。

> **注意**：绕过 Prisma 直接写 SQL（如数据修复脚本、`$queryRaw`）时，UUID v7 自动注入不会生效，需要手动调用 `newId()` 生成主键。

> **物理删除**：如需真正删除记录（如定时清理过期数据），绕过软删除扩展直接用 `$queryRaw` 或 `prisma.model.delete({ where: { id } })` 前加 `$extends({ name: 'bypassSoftDelete' })` 创建无扩展客户端。

```prisma
datasource db {

}
```

### 连接池配置

```env
# Prisma 连接池参数（通过 DATABASE_URL 传递）
DATABASE_URL=postgresql://user:pass@host:5432/db?connection_limit=20&pool_timeout=10&socket_timeout=30
```

| 参数               | 默认值    | 建议值 | 说明                       |
| ------------------ | --------- | ------ | -------------------------- |
| `connection_limit` | (CPU×2+1) | 20     | 连接池最大连接数，超过排队 |
| `pool_timeout`     | 10s       | 10s    | 等待连接的最长时间         |
| `socket_timeout`   | 60s       | 30s    | 单查询最大执行时间         |

> 生产环境建议使用 PgBouncer（事务模式）+ `DIRECT_URL` 用于 `prisma migrate deploy`。
