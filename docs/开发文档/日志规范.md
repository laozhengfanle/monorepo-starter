# 日志规范

> MonoKit 后端使用 `nestjs-pino` 包装 Pino 日志库，结构化输出 + 字段脱敏 + 请求 ID 串联。本文定义日志的**输出格式、级别使用、关联 ID 串联、敏感字段脱敏**四大规范。

## 一、输出格式

| 环境 | 格式                                   | 工具          | 用途                                     |
| ---- | -------------------------------------- | ------------- | ---------------------------------------- |
| dev  | human-readable（单行 + 颜色 + 时间戳） | `pino-pretty` | 本地开发肉眼可读                         |
| prod | JSON（每行一个对象）                   | 原生 Pino     | 日志聚合系统（ELK / Loki / Datadog）解析 |

切换由 `NODE_ENV` 控制（`logger.config.ts` 已实现）：

```ts
// apps/server/src/common/logger/logger.config.ts
if (env === 'dev') {
    return {
        pinoHttp: {
            level: 'debug',
            transport: {
                target: 'pino-pretty',
                options: {
                    singleLine: true,
                    colorize: true,
                    translateTime: 'HH:MM:ss.l',
                    ignore: 'pid,hostname',
                },
            },
            // dev 也启用脱敏（开发日志不应暴露密码）
            redact: { paths: REDACT_PATHS, censor: REDACT_CENSOR },
        },
    };
}

// prod：JSON 输出到 stdout
return {
    pinoHttp: {
        level: 'info',
        redact: { paths: REDACT_PATHS, censor: REDACT_CENSOR },
    },
};
```

## 二、级别使用指南

Pino 6 个级别（数值越大越严重）：

| 级别    | 数值 | 使用场景             | 示例                                                      |
| ------- | ---- | -------------------- | --------------------------------------------------------- |
| `trace` | 10   | **本基座不用**       | —                                                         |
| `debug` | 20   | 调试信息，开发用     | `logger.debug('查询参数: %o', params)`                    |
| `info`  | 30   | 关键业务节点         | `logger.info('用户登录成功', { accountId })`              |
| `warn`  | 40   | 预期内的异常         | `logger.warn('登录密码错误', { accountId, attempts: 3 })` |
| `error` | 50   | 非预期错误，需要关注 | `logger.error('数据库连接失败', err)`                     |
| `fatal` | 60   | 进程级崩溃           | `logger.fatal('配置缺失，无法启动', err)`                 |

### 决策流程

```
出问题？
├─ 业务正常进行（即使有异常分支）→ info
├─ 预期内可恢复异常（密码错 / 限流触发）→ warn
├─ 不可恢复 / 不可预期（DB 连不上 / 第三方 API 挂）→ error
├─ 影响进程启动或核心功能（配置缺失 / 关键依赖全挂）→ fatal
└─ 调试信息（堆栈细节 / 中间状态）→ debug（prod 不输出）
```

### NestJS Logger 兼容

业务代码里常用的是 `@nestjs/common` 的 `Logger`，不是直接用 Pino：

```ts
import { Logger } from '@nestjs/common';

@Injectable()
export class AuthService {
    private readonly logger = new Logger(AuthService.name);

    async login(account: string, password: string) {
        if (await this.verifyPassword(account, password)) {
            this.logger.log('用户登录成功'); // → Pino info
        } else {
            this.logger.warn('登录密码错误'); // → Pino warn
        }
    }
}
```

NestJS Logger 在 `LoggerModule` 注入后会被 nestjs-pino 接管，`logger.log/warn/error` 等方法走 Pino 格式输出，自动带上 `context` 字段（类名）。

### 直接注入 PinoLogger

需要更细粒度控制时直接注入：

```ts
import { PinoLogger, InjectPinoLogger } from 'nestjs-pino';

@Injectable()
export class AuditService {
    constructor(@InjectPinoLogger('AuditService') private readonly logger: PinoLogger) {}

    async logAction(action: AuditAction) {
        // 用 info / warn / error / debug / fatal
        this.logger.info({ action, userId: action.userId }, '审计操作记录');
    }
}
```

> **注意**：本基座在 v4 的 PinoLogger 注入时**不要用 `@InjectPinoLogger(contextName)`**（v4 改 API 后该装饰器会生成多余模块）。统一用 `@Inject(PinoLogger)`，详见 `logging.interceptor.ts` 注释。

## 三、敏感字段脱敏

> **绝对不要在日志中输出明文密码、Token、密钥**。一旦日志聚合系统被入侵或误开放公网，所有脱敏失效的字段会立刻泄露。

Pino 的 `redact` 机制在日志序列化时**自动替换**为 `[REDACTED]`，业务代码无需手动处理。

### 当前脱敏路径清单

来源：`apps/server/src/common/logger/logger.config.ts`

| 路径                        | 脱敏理由                                  |
| --------------------------- | ----------------------------------------- |
| `req.headers.authorization` | Bearer Token 完整值                       |
| `req.headers.cookie`        | Cookie 全集（含 session / refresh token） |
| `*.password`                | 任意层级的 password 字段                  |
| `*.token`                   | 任意层级的 token 字段                     |
| `*.newPassword`             | 修改密码接口的 newPassword                |
| `*.confirmPassword`         | 确认密码字段                              |
| `*.oldPassword`             | 旧密码字段                                |
| `res.body.accessToken`      | 登录响应 body                             |
| `res.body.refreshToken`     | 登录响应 body                             |
| `req.body.phone`            | 手机号（PII）                             |
| `req.body.email`            | 邮箱（PII）                               |

### 占位符

```ts
const REDACT_CENSOR = '[REDACTED]';
```

所有被脱敏的字段在日志中显示为字符串 `[REDACTED]`。

### 验证脱敏

测试覆盖（参见 `apps/server/src/common/logger/__tests__/logger.config.spec.ts`）：

```ts
import { describe, it, expect } from 'vitest';
import { getPinoConfig } from '../logger.config';

describe('Pino redact 脱敏', () => {
    const config = getPinoConfig('prod').pinoHttp!;

    it('所有 11 个 redact 路径都被注册', () => {
        const paths = (config.redact as { paths: string[] }).paths;
        expect(paths).toContain('req.headers.authorization');
        expect(paths).toContain('req.headers.cookie');
        expect(paths).toContain('*.password');
        expect(paths).toContain('*.token');
        expect(paths).toContain('*.newPassword');
        expect(paths).toContain('*.confirmPassword');
        expect(paths).toContain('*.oldPassword');
        expect(paths).toContain('res.body.accessToken');
        expect(paths).toContain('res.body.refreshToken');
        expect(paths).toContain('req.body.phone');
        expect(paths).toContain('req.body.email');
    });

    it('脱敏占位符为 [REDACTED]', () => {
        const censor = (config.redact as { censor: string }).censor;
        expect(censor).toBe('[REDACTED]');
    });
});
```

### 新增脱敏路径

当业务代码出现新的敏感字段时（如 `accessKey` / `secretKey` / `smsCode`），**第一时间**加到 `REDACT_PATHS`：

```ts
// apps/server/src/common/logger/logger.config.ts
const REDACT_PATHS: string[] = [
    // ... 已有路径
    '*.accessKey', // AWS / 阿里云 AccessKey
    '*.secretKey', // SecretKey
    '*.apiKey', // 第三方 API key
    '*.verificationCode', // 验证码
    '*.smsCode', // 短信验证码
    '*.otpCode', // OTP 一次性密码
    '*.jwt', // JWT 字符串
    '*.csrfToken', // CSRF token
    '*.sessionToken', // 会话 token
    'req.body.sms', // 整个 SMS body 兜底
];
```

> **为什么不部分打码（如 `password: "abc\***"）？\*\*
> 业务代码的密码格式多变（可能含特殊字符、emoji、长度异常），严格全字段 redact 更安全。日志是排查工具，不是审计工具——能确认"密码错了"就够了，没必要知道明文。

## 四、关联 ID（X-Request-ID）

> **核心价值**：一次请求触发的所有日志（HTTP / GraphQL / DB / 业务）能通过 `requestId` 字段串联，定位问题时一秒找到完整调用链。

### 工作流

```
客户端请求
   │ X-Request-ID: <uuid>（可选）
   ↓
NestJS 中间件
   │ genReqId() 读 header 或生成新 uuid
   │ res.setHeader('X-Request-ID', <uuid>)  // 回写到响应
   ↓
PinoHttp transport
   │ 把 <uuid> 写入每条日志的 requestId 字段
   ↓
所有 logger.info / warn / error 自动带 requestId
   ↓
响应返回，客户端可拿到 X-Request-ID 报错时反馈给运维
```

### 实现位置

`apps/server/src/common/logger/logger.config.ts`（已实现）：

```ts
const commonPinoHttp: PinoHttpOptions = {
    genReqId: (req, res) => {
        // 优先用客户端传入的 X-Request-ID，否则生成新 uuid
        const id = (req.headers['x-request-id'] as string | undefined) ?? crypto.randomUUID();
        res.setHeader('X-Request-ID', id); // 回写到响应
        return id;
    },
    customProps: (req) => {
        const user = (req as { user?: { id?: string } }).user;
        return {
            requestId: (req as { id?: string }).id,
            userId: user?.id, // 把当前用户 ID 也带上
        };
    },
};
```

`apps/server/src/main.ts`（已实现 X-Request-ID 中间件）：

```ts
// ── 4. X-Request-ID ──
// 即使 Pino 没接管，也先在 Express 层把 X-Request-ID 写到 header，
// 保证下游 / 客户端能拿到
app.use((req, res, next) => {
    const requestId = (req.headers['x-request-id'] as string) || crypto.randomUUID();
    req.headers['x-request-id'] = requestId;
    res.setHeader('X-Request-ID', requestId);
    next();
});
```

> **两层冗余**是有意为之：Pino 内部已经做了，但 Express 中间件是更早执行的，能保证即使用户在 Pino 之前就处理了某些边缘情况，请求 ID 也已经存在。

### 客户端发起带 ID 的请求

```ts
// apps/admin/src/api/graphql-client.ts
import { v4 as uuidv4 } from 'uuid';

async function graphqlFetch(query: string, variables: object) {
    const requestId = uuidv4();
    const res = await fetch('/api/graphql', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'X-Request-ID': requestId, // 透传给后端
        },
        body: JSON.stringify({ query, variables }),
    });

    // 拿到后端回写的 X-Request-ID（即使生成的 ID 不一样，也能对齐）
    const serverRequestId = res.headers.get('X-Request-ID');
    if (!res.ok) {
        // 错误上报时带上 requestId
        reportError({ requestId: serverRequestId, query, variables });
    }
    return res.json();
}
```

### 后端日志串联效果

一次「创建订单」请求会触发：

```

14:23:51.105 DEBUG requestId=a1b2 order.service   校验库存 { sku: 'S-1', count: 2 }
14:23:51.110 DEBUG requestId=a1b2 prisma          INSERT INTO order ...
14:23:51.115 INFO  requestId=a1b2 order.service   订单创建成功 { orderId: 'o-99' }

```

运维 / 客服根据用户反馈的 `X-Request-ID: a1b2`，在 Loki / ES 里直接搜就能拿到完整调用链。

### 关联字段

| 字段                     | 来源                       | 用途                   |
| ------------------------ | -------------------------- | ---------------------- |
| `requestId`              | 每请求生成（X-Request-ID） | 串联单次请求的所有日志 |
| `userId`                 | JWT 解出的用户 ID          | 关联用户行为           |
| `context`                | NestJS Logger 类名         | 区分日志来源           |
| `req.method` / `req.url` | Pino-http 自动注入         | HTTP 上下文            |
| `responseTime`           | Pino-http 自动注入         | 请求耗时               |

## 五、实践 Check-list

新增业务代码时：

- [ ] 错误用 `logger.error(err, context)` 或 `this.logger.error(msg, err)`，**不要只传字符串**
- [ ] 用户操作类（登录 / 下单 / 修改）用 `info` 级别
- [ ] 业务校验失败（密码错 / 余额不足）用 `warn` 级别
- [ ] 系统级错误（DB 连不上 / 第三方 API 挂）用 `error` 级别
- [ ] 进程启动失败 / 配置缺失用 `fatal`
- [ ] 日志上下文用 `Logger(ServiceName)` 自动带 context，**不要手写字符串**
- [ ] 业务对象传入 logger（如 `logger.info({ orderId, amount }, '创建订单')`）便于聚合
- [ ] **永远不打印** 用户密码 / Token / 验证码 / 手机号 / 邮箱明文
- [ ] 新增敏感字段时同步更新 `REDACT_PATHS`
