# 缓存设计

> Redis 在项目中承担三种角色：**权限数据缓存**（读多写少）、**限流计数器**（高频写入）、**临时状态存储**（验证码、state token、refresh token 黑名单）。本文定义统一的 key 命名空间、TTL 策略和缓存治理规范。

## 缓存架构概览

```text
┌─────────────────────────────────────────────────────────┐
│  应用层                                                  │
│  ioredis → ICacheService                                │
├─────────────────────────────────────────────────────────┤
│  数据缓存（权限 / 配置）     │  计数缓存（限流 / 锁定）      │
│  读多写少，Cache-Aside  │  高频写入，直接操作 Redis  │
│  全局 TTL 阶梯管理        │  Lua 脚本原子操作           │
├─────────────────────────────────────────────────────────┤
│  临时存储（验证码 / state）                                │
│  短 TTL，一次性消费                                       │
└─────────────────────────────────────────────────────────┘
```

## Key 命名规范

### 命名格式

```
mono:{domain}:{entity}:{identifier}
```

| 段           | 说明                                     | 示例                     |
| ------------ | ---------------------------------------- | ------------------------ |
| `mono`       | 项目前缀，防 key 冲突                    | —                        |
| `domain`     | 功能域（`auth`、`role`、`lock`、`rate`） | `auth`、`role`           |
| `entity`     | 实体类型                                 | `perm`、`menus`、`login` |
| `identifier` | 唯一标识（accountId、roleCode、IP）      | `abc123`、`super_admin`  |

### 完整 Key 清单

```text
# ───── 权限缓存 ─────
mono:role:permission:{userType}:{roleCode}     → SET    角色权限码集合
mono:role:menus:{userType}:{roleCode}    → JSON   角色菜单树（扁平，不拼树）
mono:auth:{accountId}                    → JSON   账户计算结果（roles + perms + 拼好的菜单树）

# ───── 限流计数器 ─────
mono:rate:{tracker}:{throttlerName}      → INT    限流窗口内请求计数

# ───── 登录安全 ─────
mono:lock:login:{accountId}             → INT    账号失败计数
mono:lock:login:ip:{ip}                 → INT    IP 失败计数

# ───── 临时状态 ─────
mono:verify:sms:{phone}                  → STR    短信验证码
mono:verify:turnstile:{token}            → STR    Turnstile 验证结果
mono:oauth:state:{state}                 → STR    OAuth state 防 CSRF
mono:refresh:used:{tokenHash}            → STR    refresh token 使用标记
mono:refresh:family:{accountId}          → SET    该账号所有 refresh token 哈希

# ───── 角色-账户映射 ─────
mono:role:accounts:{userType}:{roleCode}    → SET    持有该角色的 account ID 列表

# ───── 系统配置缓存 ─────
mono:data:system_config:{key}               → JSON   系统配置项（如 sms.provider、storage.driver）

# ───── 业务缓存（预留）─────
mono:data:{entity}:{id}                  → JSON   通用业务数据缓存
mono:data:{entity}:list:{queryHash}      → JSON   列表查询缓存
```

## TTL 策略

### 三级 TTL 阶梯

| 级别               | TTL     | 适用场景                   | 示例               |
| ------------------ | ------- | -------------------------- | ------------------ |
| **Hot**（秒级）    | 5-60s   | 实时性要求高，变更频繁     | 限流计数窗口       |
| **Warm**（分钟级） | 1-30min | 读多写少，可容忍短暂不一致 | 权限数据、验证码   |
| **Cold**（小时级） | 1-24h   | 几乎不变                   | 菜单结构、全局配置 |

### 各 Key 的 TTL

| Key                        | TTL      | 原因                                     |
| -------------------------- | -------- | ---------------------------------------- |
| `mono:role:permission:{*}` | 30 min   | 角色变更频率低                           |
| `mono:role:menus:{*}`      | 30 min   | 同上                                     |
| `mono:auth:{accountId}`    | 30 min   | 自然过期后触发懒重建，权限变更时主动失效 |
| `mono:rate:{*}`            | 窗口大小 | 限流窗口结束自动过期                     |
| `mono:lock:login:{*}`      | 15 min   | 登录锁定窗口                             |
| `mono:verify:sms:{phone}`  | 5 min    | 验证码有效期                             |
| `mono:oauth:state:{*}`     | 10 min   | OAuth 回调等待窗口                       |
| `mono:refresh:used:{*}`    | 7 days   | refresh token 有效期一致                 |
| `mono:data:{entity}:{id}`  | 5-30 min | 取决于业务                               |

## 缓存更新策略

### 权限缓存：Write-Invalidate

权限变更是低频操作，直接删缓存，下次读取时重建：

```ts
// 角色变更时 — 批量失效
async function invalidateRoleCache(userType: string, roleCode: string) {
    // 1. 删除角色级 key
    await redis.del(`mono:role:permission:${userType}:${roleCode}`);
    await redis.del(`mono:role:menus:${userType}:${roleCode}`);

    // 2. 用 pipeline 批量删除所有持有该角色的账户 key
    const accountIds = await redis.smembers(`mono:role:accounts:${userType}:${roleCode}`);
    if (accountIds.length > 0) {
        const pipeline = redis.pipeline();
        accountIds.forEach((id) => pipeline.del(`mono:auth:${id}`));
        await pipeline.exec();
    }
}

// 菜单结构变更时 — SCAN 遍历 + TTL 缩短防雪崩
async function invalidateMenuCache(userType: string) {
    // 使用 SCAN 而非 KEYS（生产安全）
    let cursor = '0';
    do {
        const result = await redis.scan(cursor, 'MATCH', `mono:role:menus:${userType}:*`, 'COUNT', 100);
        cursor = result[0];
        const keys = result[1];
        if (keys.length > 0) await redis.del(...keys);
    } while (cursor !== '0');

    // 缩短账户 TTL（1min），让新菜单快速生效
    cursor = '0';
    const pipeline = redis.pipeline();
    do {
        const result = await redis.scan(cursor, 'MATCH', 'mono:auth:*', 'COUNT', 100);
        cursor = result[0];
        result[1].forEach((key) => pipeline.expire(key, 60));
    } while (cursor !== '0');
    await pipeline.exec();
}
```

### 业务缓存：Cache-Aside

```ts
async function getCached<T>(
    key: string,
    fetcher: () => Promise<T>,
    ttl: number = 300, // 默认 5 分钟
): Promise<T> {
    const cached = await redis.get(key);
    if (cached) return JSON.parse(cached);

    const data = await fetcher();
    if (data !== null && data !== undefined) {
        await redis.setex(key, ttl, JSON.stringify(data));
    }
    return data;
}

// 使用
const user = await getCached(`mono:data:user:${userId}`, () => prisma.user.findUnique({ where: { id: userId } }), 600);
```

### 写入策略对比

| 策略             | 适用场景                     |       本项目使用       |
| ---------------- | ---------------------------- | :--------------------: |
| Cache-Aside      | 读多写少，可容忍短暂不一致   |      ✅ 业务数据       |
| Write-Through    | 写操作同时更新缓存，强一致性 |  不采用（增加写延迟）  |
| Write-Behind     | 异步批量写，高吞吐           | 不采用（数据丢失风险） |
| Write-Invalidate | 写操作删除缓存，下次读重建   |      ✅ 权限数据       |
| **Version-Lazy** | **写时不强制失效，读时校验** | **✅ 菜单兜底（v2）**  |

## 声明式失效：数据版本号（v2 引入）

> 背景：Write-Invalidate 是命令式的——写完代码的人必须记得调 `invalidate*()`。一旦有人漏调（seed 脚本、Prisma Studio 改、临时直 SQL），缓存就脏了，30 分钟内任何再次访问都吃旧数据。本节引入「数据版本号 + 懒失效」做兜底。

### 核心思想

写路径只做一件事：**把版本号 +1**。读路径检查版本号是否一致，**不一致就视为脏数据并重建**。命令式失效成为「立即生效」的优化手段，版本号成为「最终兜底」。

```text
菜单发生 create/update/delete
  ├─ 主动失效：invalidateMenuStructure()    ← 让现有用户下次请求立刻看到新数据
  └─ 兜底：bumpMenuVersion()                 ← 写路径不调失效也无所谓

任何账户下次调 getAccountAuth()
  ├─ 读账户缓存 mono:auth:{accountId}
  ├─ 读当前版本号 mono:data:menu_version
  ├─ 二者一致 → 缓存有效 → 直接返回（快路径）
  └─ 不一致 → 视为脏数据 → DEL 账户缓存 → 重建（懒失效）
```

### 代码实现

#### 1. 写路径 — bumpMenuVersion

```ts
// admin-permission-cache.service.ts
async bumpMenuVersion(): Promise<number> {
    const next = await this.cacheService.incr(CACHE_KEYS.MENU_VERSION);
    this.logger.log(`菜单版本号已 bump → ${next}`);
    return next;
}
```

#### 2. 写路径 — 调用时机

| Service 方法                     | 是否 bump | 备注                                                 |
| -------------------------------- | --------- | ---------------------------------------------------- |
| `AdminMenuService.create()`      | ✅        | 未来若做「新建菜单自动赋给角色」，下游用户立即能看到 |
| `AdminMenuService.update()`      | ✅        | 改 icon / name / path / sort 必须 bump               |
| `AdminMenuService.delete()`      | ✅        | 软删后从用户菜单树消失                               |
| `seed.ts`                        | ✅        | seed 末尾显式 `INCR mono:data:menu_version`          |
| 直接 `prisma.adminMenu.update()` | ⚠️ 不会   | 但下次读会自愈（这就是版本号的意义）                 |

> **设计取舍**：主动失效（`invalidateMenuStructure()`）+ 版本号（`bumpMenuVersion()`）**双保险**。前者让数据立刻生效（无延迟），后者保证「万一有人漏调，30 分钟内也能自愈」。不要只用一个。

#### 3. 读路径 — 版本校验

```ts
// admin-permission-cache.service.ts
async getAccountAuth(accountId: string): Promise<AuthCacheData | null> {
    const cacheKey = this.authKey(accountId);
    const cached = await this.cacheService.get<AuthCacheData>(cacheKey);
    if (cached) {
        const currentVersion = await this.getCurrentMenuVersion();
        if (cached.menuVersion !== currentVersion) {
            // 缓存脏了 → 删掉并重建（懒失效）
            this.logger.log(
                `账户缓存菜单版本不一致 (cached=${cached.menuVersion}, current=${currentVersion})，触发懒失效`,
            );
            await this.cacheService.del(cacheKey);
            return this.buildAccountAuth(accountId);
        }
        return cached; // 快路径
    }
    return this.buildAccountAuth(accountId);
}
```

#### 4. 数据结构 — 嵌入版本号

```ts
// admin-permission-cache.service.ts
export interface AuthCacheData {
    roles: string[];
    permissions: string[];
    menus: MenuNode[];
    /** 缓存构建时的菜单数据版本号，读时与 mono:data:menu_version 比对 */
    menuVersion: number;
}
```

#### 5. 缓存 Key

```
mono:data:menu_version   → INT    全局菜单数据版本号（INCR）
```

写入 `AuthCacheData` 时嵌入当前值；读取时与当前值比对，不一致则 miss。

### 向后兼容

旧版缓存（v1 升级前）**没有 `menuVersion` 字段**。`cached.menuVersion` 为 `undefined`，与任何非零的当前版本号比对必失败 → 自动重建。这是「升级即自愈」的优雅过渡：不需要手动 `redis-cli del mono:auth:*` 也能让所有用户下次访问走新逻辑。

### 不适用场景

版本号模式**不适合数据量大、版本比较代价高的场景**（比如百万级行级缓存）。本项目菜单是「全局单维」——一个版本号管所有菜单，所以完美适配。如果未来要做「角色 A 的菜单版本号独立于角色 B」，需要扩展为多维版本（如 `mono:data:menu_version:{roleCode}`），或退回到纯 Write-Invalidate。

## Seed 脚本必须接失效

> 背景：seed 脚本绕过了所有 Service 层，直接用 Prisma Client 写 DB。这是「不在失效白名单里的写路径」——如果不显式调失效，30 分钟内所有用户吃旧数据。

### 规范

- 所有 `apps/server/prisma/seed*.ts` 文件的 `main().finally()` 里**必须**调用 `invalidateCaches()`
- 不调失效的 seed PR 应被打回
- 测试 fixture 写 DB 后也必须调（e2e / integration 测试用）

### 范例

```ts
// apps/server/prisma/seed.ts
import Redis from 'ioredis';

async function invalidateCaches(): Promise<void> {
    const redisUrl = process.env['REDIS_URL'];
    if (!redisUrl) {
        console.log('ℹ️  未配置 REDIS_URL，跳过缓存失效');
        return;
    }
    const redis = new Redis(redisUrl);
    try {
        // 1. 角色级缓存全删
        let cursor = '0';
        do {
            const [next, keys] = await redis.scan(cursor, 'MATCH', 'mono:role:permission:admin:*', 'COUNT', 100);
            cursor = next;
            if (keys.length > 0) await redis.del(...keys);
        } while (cursor !== '0');
        // ... 同样处理 mono:role:menus:admin:*

        // 2. 账户级缓存缩短 TTL 到 60s（防雪崩）
        // ... SCAN mono:auth:* → EXPIRE 60

        // 3. bump 版本号（触发懒失效兜底）
        const newVersion = await redis.incr('mono:data:menu_version');
        console.log(`✅ 缓存已失效：菜单版本号 → ${newVersion}`);
    } finally {
        await redis.quit();
    }
}

main()
    .catch((e) => {
        /* ... */
    })
    .finally(async () => {
        try {
            await invalidateCaches();
        } catch (e) {
            /* 不阻塞主流程 */
        }
        await prisma.$disconnect();
    });
```

## 缓存防护

### 缓存穿透（大量不存在的 key）

**问题**：请求大量不存在的 ID，每次都穿透到数据库。

**方案**：空值缓存 + Bloom Filter。

```ts
async function getCachedWithNull<T>(
    key: string,
    fetcher: () => Promise<T | null>,
    ttl: number = 300,
): Promise<T | null> {
    const cached = await redis.get(key);
    if (cached === '__NULL__') return null; // 空值标记
    if (cached) return JSON.parse(cached);

    const data = await fetcher();
    if (data === null) {
        await redis.setex(key, 60, '__NULL__'); // 空值 TTL 更短
        return null;
    }
    await redis.setex(key, ttl, JSON.stringify(data));
    return data;
}
```

### 缓存击穿（热点 key 过期瞬间大量请求）

**方案**：互斥锁（分布式锁）。

```ts
async function getCachedWithLock<T>(
    key: string,
    fetcher: () => Promise<T>,
    ttl: number = 300,
    maxRetries: number = 10,
): Promise<T> {
    for (let retry = 0; retry <= maxRetries; retry++) {
        const cached = await redis.get(key);
        if (cached) return JSON.parse(cached);

        // 尝试获取重建锁
        const lockKey = `${key}:lock`;
        const acquired = await redis.set(lockKey, '1', 'NX', 'EX', 5);
        if (!acquired) {
            // 没拿到锁，等 100ms 后重试读缓存
            await new Promise((resolve) => setTimeout(resolve, 100));
            continue;
        }

        try {
            // 双重检查
            const recheck = await redis.get(key);
            if (recheck) return JSON.parse(recheck);

            const data = await fetcher();
            await redis.setex(key, ttl, JSON.stringify(data));
            return data;
        } finally {
            await redis.del(lockKey);
        }
    }

    throw new Error(`Cache rebuild lock timeout for key: ${key}`);
}
```

### 缓存雪崩（大量 key 同时过期）

**方案**：TTL 加随机偏移。

```ts
function ttlWithJitter(baseTtl: number, jitterPercent: number = 0.1): number {
    const jitter = Math.floor(baseTtl * jitterPercent * Math.random());
    return baseTtl + jitter;
}

// 100 条缓存各自过期在 270-330s 之间，不会同时失效
await redis.setex(key, ttlWithJitter(300), value);
```

### 识别 Big Key

```bash
redis-cli --bigkeys
redis-cli MEMORY USAGE mono:auth:{accountId}
```

### 预防

| 规则                 | 说明                   |

## 缓存与权限联动

权限变更时的缓存失效链：

```text
角色权限变更
  ├── DEL mono:role:permission:{userType}:{roleCode}     (角色级)
  ├── DEL mono:role:menus:{userType}:{roleCode}    (角色级)
  ├── 查询该角色下所有 account ID
  └── PIPELINE DEL mono:auth:{accountId}           (账户级，批量)

角色新增/删除
  ├── 同上（权限集变化）
  └── 更新 mono:role:accounts:{userType}:{roleCode} (角色-账户映射)

菜单结构变更
  ├── SCAN mono:role:menus:* → 全量 DEL            (角色级)
  ├── SCAN mono:auth:* → EXPIRE 60                 (账户级缩短 TTL)
  └── 不直接 DEL，避免雪崩

账号角色变更
  ├── DEL mono:auth:{accountId}                     (账户级)
  └── SADD/SREM mono:role:accounts:{userType}:{roleCode}
```

## 大规模缓存失效策略

当账户数很多（如 10 万+）时，角色菜单变更后的 pipeline 批量删除需要分页执行，避免阻塞 Redis：

```typescript
async function invalidateRoleCacheSafe(userType: string, roleCode: string) {
    // 1. 删除角色级 key
    await redis.del(`mono:role:permission:${userType}:${roleCode}`);
    await redis.del(`mono:role:menus:${userType}:${roleCode}`);

    // 2. 分页获取受影响账户 ID
    const accountIds = await redis.smembers(`mono:role:accounts:${userType}:${roleCode}`);

    // 3. 分批 pipeline 删除账户级缓存（每批 100 条）
    const BATCH_SIZE = 100;
    for (let i = 0; i < accountIds.length; i += BATCH_SIZE) {
        const batch = accountIds.slice(i, i + BATCH_SIZE);
        const pipeline = redis.pipeline();
        batch.forEach((id) => pipeline.del(`mono:auth:${id}`));
        await pipeline.exec();
    }
}
```

> **为什么分批**：Redis pipeline 虽然减少了网络往返，但单次 pipeline 包含过多命令仍会阻塞 Redis 事件循环（Redis 6.x+ 的 IO 多线程仅处理网络 IO，命令执行仍是单线程）。建议每批不超过 100 条命令。

### 缓存一致性窗口说明

场景 C（菜单结构变更）中使用 `EXPIRE` 缩短 TTL 而非直接 `DEL`，会存在最长 5 分钟的不一致窗口——期间用户看到的仍是旧菜单结构。这是设计取舍：

- **直接 DEL**：实时生效，但大量账户同时回源 DB，可能引发缓存击穿
- **缩短 TTL**：最多 5 分钟不一致，但回源请求均匀分散，系统更稳定

如果业务要求实时生效，可用以下替代方案：

```typescript
// 替代方案：使用 SCAN + 分批 DEL，避免雪崩
async function invalidateMenuCacheImmediate(userType: string) {
    // 1. 删所有角色级缓存
    await cache.delByPattern(`mono:role:permission:${userType}:*`);
    await cache.delByPattern(`mono:role:menus:${userType}:*`);

    // 2. 分批删账户级缓存（每批 100 条，SCAN 游标遍历）
    let cursor = '0';
    do {
        const [next, keys] = await redis.scan(cursor, 'MATCH', 'mono:auth:*', 'COUNT', 100);
        cursor = next;
        if (keys.length > 0) {
            // 每批之间加 50ms 延迟，避免瞬间大量回源
            await redis.del(...keys);
            await new Promise((r) => setTimeout(r, 50));
        }
    } while (cursor !== '0');
}
```

> **建议**：管理后台场景下，5 分钟不一致窗口可接受（菜单变更本身就很低频），优先使用 `EXPIRE` 缩短 TTL 方案。

## Redis 集群模式

单实例足够起步。如后续需要高可用：

```
单实例 → Sentinel（主从 + 自动故障转移）→ Cluster（分片，16384 slots）
```

限流场景的 Lua 脚本需要确保所有 key 在同一个 slot（用 `{hashtag}` 保证）：

```lua
-- 限流 Lua 脚本，{tracker} 保证同 slot
local key = KEYS[1]  -- mono:rate:{tracker}:default
local limit = tonumber(ARGV[1])
local ttl = tonumber(ARGV[2])
local current = redis.call('GET', key)
if current and tonumber(current) >= limit then
  return 0
end
redis.call('INCR', key)
redis.call('EXPIRE', key, ttl)
return 1
```

## 缓存统计字段

缓存统计端点（后续 GraphQL query `cacheStats`）返回的 4 个统计字段定义如下（与 Redis 命令行 `INFO` 一致）：

| 字段       | Redis 数据来源                                 | 计算公式                      | 备注                                                                                   |
| ---------- | ---------------------------------------------- | ----------------------------- | -------------------------------------------------------------------------------------- |
| totalKeys  | `SCAN MATCH mono:* COUNT 100` 遍历             | 遍历计数                      | 实时；含敏感 key（mono:auth:\* 等）                                                    |
| usedMemory | `INFO Memory.used_memory`                      | 字节数 → 人类可读（KB/MB/GB） | Redis 不主动归还内存（jemalloc 行为），清空 key 不会立即下降；需 `MEMORY PURGE` 或重启 |
| hitRate    | `INFO Stats.keyspace_hits` / `keyspace_misses` | `hits / (hits + misses)`      | 累计计数；无样本时显示 `-`                                                             |
| uptime     | `INFO Server.uptime_in_seconds`                | 秒 → "X 天 Y 小时"            | 累计时长；与 key 数量无关                                                              |

### Redis 数据类型

`SCAN` 返回的 key 还要通过 `TYPE` 命令取数据类型，6 种：

| TYPE 返回 | 含义                         | Naive UI Tag 颜色 |
| --------- | ---------------------------- | ----------------- |
| string    | 字符串                       | success (绿)      |
| hash      | 哈希                         | info (蓝)         |
| list      | 列表                         | warning (橙)      |
| set       | 集合                         | error (红)        |
| zset      | 有序集合                     | primary (蓝)      |
| stream    | 流                           | info (蓝)         |
| none      | 已过期（不应在 SCAN 结果里） | default (灰)      |
| unknown   | TYPE 命令失败兜底            | default (灰)      |

### 清空缓存后 stats 行为

- `totalKeys`：从 N 变 0（实时刷新）
- `usedMemory`：**不变**（Redis 不主动归还内存）
- `hitRate`：**不变**（hits/misses 是 Redis 启动以来累计）
- `uptime`：**不变**（Redis 进程运行时长，与 key 数量无关）

这是 Redis 行为，不是 bug。
