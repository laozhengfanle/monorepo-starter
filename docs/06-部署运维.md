## 部署架构概览

```text
                    ┌────────────────────┐
                    │   CDN / WAF        │
                    │  (Cloudflare)      │
                    └─────────┬──────────┘
                              │
                              ▼
        ┌────────────────────────────────────────┐
        │   反向代理 / 负载均衡                   │
        │   (Nginx / ALB / CloudFront)            │
        │   - 终止 TLS                            │
        │   - 转发到 admin:80 / web:80 / server:3000 │
        └─────┬──────────────┬──────────────┬────┘
              │              │              │
              ▼              ▼              ▼
         ┌────────┐    ┌────────┐    ┌──────────┐
         │ admin  │    │  web   │    │  server  │
         │ :80    │    │ :80    │    │  :3000   │
         │ nginx  │    │ nginx  │    │  Node    │
         └────┬───┘    └────┬───┘    └────┬─────┘
              │             │              │
              └─────────────┴──────────────┘
                            │
                            ▼
              ┌──────────────────────────┐
              │   PostgreSQL 16          │  (内网)
              │   Redis 7                │  (内网)
              └──────────────────────────┘
```

**关键原则**：

- 前端 (admin/web) 走 nginx 静态服务 + 反代 `/api` / `/graphql` / `/uploads` 到 server
- 后端 (server) **不直接对外暴露**：只在 docker network 内可访问
- DB / Redis **不暴露端口到宿主机**

## 构建镜像

### 多阶段构建

每个 app 都有独立的多阶段 `Dockerfile`：

| App | Dockerfile | 预期镜像体积 |
|---|---|---|
| 后端 | [`apps/server/Dockerfile`](../apps/server/Dockerfile) | 250-360 MB |
| 管理后台 | [`apps/admin/Dockerfile`](../apps/admin/Dockerfile) | 50-70 MB |
| C 端 | [`apps/web/Dockerfile`](../apps/web/Dockerfile) | 50-70 MB |

### 构建命令

```bash
# 在仓库根执行（用 BuildKit 启用 cache mount）
DOCKER_BUILDKIT=1 docker build \
    -f apps/server/Dockerfile \
    -t ghcr.io/<your-org>/<project>-server:v1.2.3 \
    .

DOCKER_BUILDKIT=1 docker build \
    -f apps/admin/Dockerfile \
    -t ghcr.io/<your-org>/<project>-admin:v1.2.3 \
    .

DOCKER_BUILDKIT=1 docker build \
    -f apps/web/Dockerfile \
    -t ghcr.io/<your-org>/<project>-web:v1.2.3 \
    .
```

> **打 tag 时必须带版本号**（不要只打 `latest`），方便回滚。

### 推送镜像

```bash
# 登录 GHCR
echo $GITHUB_TOKEN | docker login ghcr.io -u <your-org> --password-stdin

# 推送
docker push ghcr.io/<your-org>/<project>-server:v1.2.3
docker push ghcr.io/<your-org>/<project>-admin:v1.2.3
docker push ghcr.io/<your-org>/<project>-web:v1.2.3
```

## 生产编排（Docker Compose）

仓库根 [`docker-compose.prod.yml`](../docker-compose.prod.yml) 是生产编排模板，**所有安全基线都开了**：

- [x] non-root 运行（UID 1000 / 999）
- [x] read-only 根文件系统
- [x] tmpfs 写 `/tmp` / `/var/cache/nginx`
- [x] `no-new-privileges`
- [x] `cap_drop: ALL`，只 `cap_add` 必要项
- [x] 资源限制（cpus / memory / pids）
- [x] 健康检查
- [x] 日志大小限制（json-file 10MB × 3）

### 启动

```bash
# 1. 准备生产环境变量文件（git 忽略，单独放服务器上）
cat > .env.production <<EOF
POSTGRES_USER=myapp
POSTGRES_PASSWORD=<strong-password-32-chars>
POSTGRES_DB=myapp_prod
JWT_SECRET=$(openssl rand -hex 32)
AES_ENCRYPTION_KEY=$(openssl rand -hex 32)
COOKIE_DOMAIN=.example.com
CORS_ORIGINS=https://admin.example.com,https://app.example.com
TURNSTILE_SITE_KEY=<your-site-key>
TURNSTILE_SECRET_KEY=<your-secret-key>
IMAGE_TAG=v1.2.3
EOF

# 2. 启动
docker compose -f docker-compose.prod.yml --env-file .env.production up -d

# 3. 跑迁移（首次部署 / 升级基座时）
docker compose -f docker-compose.prod.yml --env-file .env.production exec server pnpm db:migrate
docker compose -f docker-compose.prod.yml --env-file .env.production exec server pnpm db:seed

# 4. 验证
docker compose -f docker-compose.prod.yml ps
curl http://localhost:3000/health
```

### 停服

```bash
# 停服（保留数据卷）
docker compose -f docker-compose.prod.yml down

# 停服 + 清数据卷（**危险**，会丢数据）
docker compose -f docker-compose.prod.yml down -v
```

## 生产环境变量清单

> 详见 [04-配置参考.md](./04-配置参考.md)。下面是**生产必填**的精简版：

| 变量 | 必填 | 建议值 |
|---|---|---|
| `POSTGRES_USER` / `_PASSWORD` / `_DB` | 是 | 强密码，不与 dev 相同 |
| `JWT_SECRET` | 是 | `openssl rand -hex 32` |
| `AES_ENCRYPTION_KEY` | 是 | `openssl rand -hex 32` |
| `NODE_ENV` | 是 | `production` |
| `COOKIE_SECURE` | 是 | `true` |
| `CSRF_COOKIE_SECURE` | 是 | `true` |
| `CORS_ORIGINS` | 是 | `https://admin.example.com,https://app.example.com` |
| `COOKIE_DOMAIN` | 是 | `.example.com`（管理后台 / C 端共享 cookie 域时） |
| `REDIS_URL` | 是 | `redis://redis:6379/0`（compose 内部） |
| `DATABASE_URL` | 是 | `postgresql://...@postgres:5432/...` |
| `TURNSTILE_SITE_KEY` / `TURNSTILE_SECRET_KEY` | 视需求 | Cloudflare 控制台拿 |
| `LOG_LEVEL` | 否 | `warn`（生产） |
| `IMAGE_TAG` | 是 | `v1.2.3`（镜像 tag） |

## 数据库迁移与备份

### 升级前：备份

```bash
# 备份整个数据库
docker exec monorepo-postgres pg_dump -U $POSTGRES_USER -d $POSTGRES_DB > backup-$(date +%Y%m%d-%H%M%S).sql

# 压缩（生产数据库大时）
docker exec monorepo-postgres pg_dump -U $POSTGRES_USER -d $POSTGRES_DB | gzip > backup-$(date +%Y%m%d-%H%M%S).sql.gz
```

### 应用迁移

```bash
# 查看待应用的迁移
docker compose -f docker-compose.prod.yml exec server pnpm db:migrate:status

# 应用
docker compose -f docker-compose.prod.yml exec server pnpm db:migrate
```

### 升级失败：回滚

```bash
# 1. 停服
docker compose -f docker-compose.prod.yml down

# 2. 启动临时 Postgres
docker run -d --name pg-restore \
    -v monorepo-postgres-data:/var/lib/postgresql/data \
    -e POSTGRES_USER=$POSTGRES_USER \
    -e POSTGRES_PASSWORD=$POSTGRES_PASSWORD \
    -e POSTGRES_DB=$POSTGRES_DB \
    postgres:16-alpine

# 3. 还原（如果是 pg_dump 纯 SQL）
cat backup-20260617-103000.sql | docker exec -i pg-restore psql -U $POSTGRES_USER -d $POSTGRES_DB

# 4. 回滚镜像到上一个 tag
# 修改 .env.production 的 IMAGE_TAG=v1.2.2

# 5. 启动
docker compose -f docker-compose.prod.yml --env-file .env.production up -d
```

### 定时备份（建议）

加 cron：

```cron
# 每天凌晨 3 点备份，保留 7 天
0 3 * * * /usr/local/bin/backup-postgres.sh
```

```bash
#!/bin/bash
# /usr/local/bin/backup-postgres.sh
BACKUP_FILE="/backups/postgres-$(date +\%Y\%m\%d-\%H\%M\%S).sql.gz"
docker exec monorepo-postgres pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" | gzip > "$BACKUP_FILE"
find /backups -name "postgres-*.sql.gz" -mtime +7 -delete
```

## 健康检查与监控

### 健康端点

| 端点 | 用途 | 适用 |
|---|---|---|
| `/health` | 综合（DB + Redis + 磁盘） | 人工 / 监控 |
| `/health/liveness` | 进程存活 | k8s livenessProbe |
| `/health/readiness` | 就绪 | k8s readinessProbe |

### Prometheus 指标

**默认关闭**。开启：

```bash
PROMETHEUS_ENABLED=true
```

访问 `http://server:3000/metrics`，包含 4 个 collector：

- `http_*` —— HTTP 请求耗时 / 计数
- `graphql_*` —— GraphQL operation 耗时 / 错误数
- `db_*` —— 数据库连接池 / 查询耗时
- `business_*` —— 业务指标（登录次数、上传次数等）

> **必须配 `MetricsIpGuard`**：默认拒绝外网访问 `/metrics`，只允许内网 IP。详见 `apps/server/src/common/guards/metrics-ip.guard.ts`。

### 推荐监控方案

| 监控项 | 工具 |
|---|---|
| 进程存活 | Docker / k8s healthcheck |
| HTTP / GraphQL 指标 | Prometheus + Grafana |
| 错误日志 | Loki / ELK |
| 告警 | Alertmanager |
| APM（可选） | Sentry / OpenTelemetry |

## HTTPS / 反向代理

> **生产环境必须 HTTPS**——COOKIE_SECURE / CSRF_COOKIE_SECURE 都依赖 HTTPS。

### 方案 A：CDN 终止 TLS（推荐）

```
Cloudflare / CloudFront
  ├─→ https://admin.example.com → admin:80
  ├─→ https://app.example.com   → web:80
  └─→ https://api.example.com   → server:3000
```

CDN 帮你终止 TLS、处理 DDoS、配 WAF。

### 方案 B：Nginx 终止 TLS

```nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    location / {
        proxy_pass http://server:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

记得设 `TRUSTED_PROXIES=1`（信任前 1 层代理）。

## 升级与回滚

### 升级流程

```bash
# 1. 备份数据库（强制）
docker exec monorepo-postgres pg_dump -U $POSTGRES_USER -d $POSTGRES_DB > backup-pre-upgrade.sql

# 2. 拉新镜像
docker compose -f docker-compose.prod.yml pull

# 3. 滚动重启
docker compose -f docker-compose.prod.yml up -d

# 4. 跑迁移（如果有）
docker compose -f docker-compose.prod.yml exec server pnpm db:migrate

# 5. 验证
curl http://localhost:3000/health
# 登录管理后台、点几个核心页面
```

### 回滚

```bash
# 1. 改 .env.production 的 IMAGE_TAG 到上一个版本
sed -i 's/IMAGE_TAG=v1.2.3/IMAGE_TAG=v1.2.2/' .env.production

# 2. 重启
docker compose -f docker-compose.prod.yml --env-file .env.production up -d

# 3. 如果新版本已经应用了迁移、回滚有破坏，参考上面的「数据库回滚」
```

## K8s 部署

> 仓库根 [`docs/部署-K8s.md`](../docs/部署-K8s.md) 有完整的 K8s manifest 示例（位于开发文档，仅供参考）。

**最小部署清单**：

```text
├── namespace.yaml
├── secrets.yaml            # JWT_SECRET / AES_ENCRYPTION_KEY / DB password
├── configmap.yaml          # 非敏感环境变量
├── postgres-statefulset.yaml
├── redis-statefulset.yaml
├── server-deployment.yaml
├── admin-deployment.yaml
├── web-deployment.yaml
├── server-service.yaml     # ClusterIP
├── admin-service.yaml      # ClusterIP
├── web-service.yaml        # ClusterIP
├── ingress.yaml            # TLS via cert-manager
└── hpa.yaml                # 自动扩缩容（可选）
```

**关键配置**：

- `server` 容器必须设 `securityContext.runAsNonRoot: true` + `readOnlyRootFilesystem: true` + 挂 `emptyDir` 到 `/tmp`
- `livenessProbe`：`httpGet /health/liveness`，`initialDelaySeconds: 30`
- `readinessProbe`：`httpGet /health/readiness`，`initialDelaySeconds: 10`
- `resources.requests` / `limits`：参考 `docker-compose.prod.yml` 的 `deploy.resources`
