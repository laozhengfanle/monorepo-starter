## 安装 / 启动类

### `pnpm install` 报 `ERR_PNPM_PEER_DEP_ISSUES`

**症状**：

```text
× Could not resolve dependencies
  peer graphql@"^16.0.0" from @nestjs/graphql@13.x
```

**原因**：pnpm 11 默认严格 peer 解析，某个 workspace 包版本对不上。

**解法**：

```bash
# 1. 先删 lockfile 和 node_modules
rm -rf node_modules pnpm-lock.yaml
rm -rf apps/*/node_modules packages/*/node_modules

# 2. 重装
pnpm install

# 如果还是不行，检查 pnpm 版本
pnpm -v   # 应该是 11.x

# 极端情况：用 corepack 强制装 11
corepack prepare pnpm@11.4.0 --activate
```

---

### `pnpm dev` 报 `Cannot find module '@packages/shared'`

**症状**：

```text
Error: Cannot find module '@packages/shared'
    at ... apps/server/dist/main.js
```

**原因**：`@packages/shared` 没编译。`predev:server` 钩子没触发。

**解法**：

```bash
# 手动编译 shared
pnpm -F @packages/shared build

# 再启 dev
pnpm dev
```

---

### `docker compose up -d` 端口冲突

**症状**：

```text
Error: bind: address already in use 0.0.0.0:5432
```

**原因**：本机已有 Postgres / Redis 占用了 5432 / 6379。

**解法**：

仓库根的 `docker-compose.yml` 已经把端口映射成 `127.0.0.1:5433:5432` 和 `127.0.0.1:6379:6379`，**不应该**冲突。如果还是冲突：

```bash
# 找占用端口的进程
lsof -i :5433
lsof -i :6379

# 停本机的 Postgres / Redis（macOS 启动台里的 Postgres.app / `brew services stop redis`）
```

## 数据库类

### `pnpm db:migrate` 报 `P1001: Can't reach database server`

**症状**：

```text
Error: P1001: Can't reach database server at `localhost:5433`
```

**原因**：

1. Postgres 容器没起来
2. `DATABASE_URL` 端口写错

**解法**：

```bash
# 1. 确认容器在跑
docker compose ps

# 2. 看日志
docker compose logs postgres

# 3. 手动测连通
docker exec monorepo-postgres pg_isready -U root -d mono_dev

# 4. 确认 .env 的 DATABASE_URL 端口
grep DATABASE_URL apps/server/.env
# 应该是 5433（容器映射的端口）
```

---

### `pnpm db:seed` 报 `Unique constraint failed`

**症状**：

```text
Unique constraint failed on the fields: (`code`)
```

**原因**：seed 不是幂等的，或者某条数据已经存在。

**解法**：

```bash
# 重置数据库（**会丢数据**）
pnpm -F @apps/server db:migrate:reset
# 提示时输入 yes

# 再 seed
pnpm -F @apps/server db:seed
```

> `apps/server/prisma/seed.ts` 应该写成幂等的（先查再插）。如果你 fork 后改了 seed 破坏了幂等性，回退到上一次能跑的版本再改。

---

### Prisma Client 没生成

**症状**：

```text
Cannot find module '@prisma/client' or its corresponding type declarations.
```

**原因**：`apps/server/prisma/generated/` 是 git 忽略的，clone 后没生成。

**解法**：

```bash
pnpm -F @apps/server db:generate
```

## 认证 / 登录类

### 登录返回 401

**症状**：登录页填了正确账号密码，提示「账号或密码错误」。

**可能原因**：

| 原因 | 检查方式 | 解法 |
|---|---|---|
| seed 没跑 | `psql ... -c "SELECT id, user_type FROM account"` 看有没有 `root` | `pnpm -F @apps/server db:seed` |
| 密码哈希不一致 | 默认是 `Root!123`，有没有改过 | 看 `seed.ts` 里默认值 |
| 账号被禁用 | 看 `account.enabled` 字段 | 改成 `true` |
| 账号被软删 | 看 `account.deleted_at` 字段 | 改成 `NULL` |
| `JWT_SECRET` 改了 | 重启后旧 token 全失效 | 用新 secret 重新登录 |

---

### 登录返回 `LOGIN_LOCKED` (21001)

**症状**：

```json
{ "code": 21001, "message": "账号已被锁定，请 15 分钟后再试" }
```

**原因**：`LoginLockService` 计数失败次数达上限（默认 5 次/15 分钟）。

**解法**：

```bash
# 等 15 分钟（默认）
# 或者清 Redis 里的锁定 key
docker exec monorepo-redis redis-cli
> KEYS login-lock:*
> DEL login-lock:<accountId>
```

---

### 登录后立刻跳回登录页

**症状**：登录接口返回 200，但前端跳回登录页。

**可能原因**：

| 原因 | 检查方式 | 解法 |
|---|---|---|
| Cookie 没下发 | DevTools → Application → Cookies | 确认 `Set-Cookie` header 存在 |
| Cookie `Secure=true` 但 HTTP | 看 `apps/server/.env` 的 `COOKIE_SECURE` | dev 改 `false` |
| `SameSite` 跨域问题 | DevTools Console 看 CORS 错误 | 确认 `CORS_ORIGINS` 包含前端 origin |
| 401 触发 refresh 也失败 | Network 看 `/api/auth/refresh` | 看下面 CSRF 章节 |

## CSRF / CORS 类

### POST 请求报 `403 Forbidden` (CSRF token mismatch)

**症状**：

```json
{ "code": 10001, "message": "Invalid CSRF token" }
```

**原因**：写请求没带 `X-CSRF-Token` 头，或带的值和 cookie 不一致。

**解法**：

```bash
# 1. 确认前端代码用 graphql-client.ts（已自动处理 CSRF）
# 2. 确认登录后有 __Host-csrf-token / csrf-token cookie
# 3. 手动测：
curl -c /tmp/cookies.txt -X POST http://localhost:3000/api/auth/csrf-token
# 应该返回 { token: "xxx" }
curl -b /tmp/cookies.txt -X POST http://localhost:3000/api/auth/login \
    -H "Content-Type: application/json" \
    -H "x-csrf-token: <token-from-cookie>" \
    -d '{"username":"root","password":"Root!123"}'
```

> 如果是裸 fetch 没走 `graphql-client.ts` / `request.ts`，需要手动加 `credentials: 'include'` + 从 cookie 读 token + 加 header。

---

### 浏览器报 CORS 错误

**症状**：

```text
Access to fetch at '...' from origin 'http://localhost:5173' has been blocked by CORS policy
```

**原因**：`CORS_ORIGINS` 没包含前端 origin。

**解法**：

```bash
# dev: 改 .env
CORS_ORIGINS=http://localhost:5173,http://localhost:5174

# prod: 改 .env.production
CORS_ORIGINS=https://admin.example.com,https://app.example.com
```

---

### `credentials: 'include'` 跨域 cookie 带不上

**症状**：跨域请求没带 cookie。

**原因**：CORS 配置缺失 `Access-Control-Allow-Credentials: true` 或 origin 不是显式白名单。

**解法**：

- `CORS_ORIGINS` 必须**显式列出** origin（不能用通配符 `*`）
- 前端 fetch 必须 `credentials: 'include'`
- `COOKIE_DOMAIN` 设成共享父域（例：`.example.com`）

## GraphQL 类

### `pnpm -F @apps/server schema:check` 失败

**症状**：

```text
Schema artifact 与 HEAD 不一致
```

**原因**：改了 resolver / type，但没重新生成 `dist/schema.gql`。

**解法**：

```bash
# 重新生成
pnpm -F @apps/server build
pnpm -F @apps/server generate:schema

# 提交新的 artifact
git add apps/server/dist/schema.gql
git commit -m "chore: regenerate schema artifact"
```

---

### `Query is too deep` 错误

**症状**：

```json
{ "errors": [{ "message": "Query is too deep: 11" }] }
```

**原因**：GraphQL depth limit 触发（默认 10 层）。

**解法**：

```bash
# 在 apps/server/src/modules/graphql/graphql.module.ts 调大
depthLimit: 15   # 改大
```

> 警告：depth limit 是防 DoS 的，调大前确认前端查询有边界。

---

### 复杂查询超时

**症状**：

```json
{ "errors": [{ "message": "Query timeout" }] }
```

**原因**：GraphQL timeout 触发。

**解法**：

```ts
// apps/server/src/modules/graphql/graphql.module.ts
{
    timeout: 30000,  // 30 秒
}
```

> 如果某个查询确实需要更长时间，先用 DataLoader 优化 N+1，再考虑调大 timeout。

## 前端类

### Vite dev 启动后访问 `localhost:5173` 白屏

**可能原因**：

| 原因 | 排查 |
|---|---|
| 后端没起来 | `curl http://localhost:3000/health` |
| Vite proxy 配错 | 看 `apps/admin/vite.config.ts` 的 `server.proxy` |
| 端口被占 | `lsof -i :5173`，杀进程 |
| 浏览器缓存 | 强刷（Cmd+Shift+R） |

---

### Tailwind class 不生效

**症状**：写 `class="text-red-500"` 但页面没样式。

**原因**：`apps/admin/vite.config.ts` 没装 `@tailwindcss/vite` 插件。

**解法**：

```ts
// vite.config.ts
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
    plugins: [vue(), tailwindcss()],  // tailwindcss 必须在最后
});
```

---

### 改了 `@packages/shared` 但前端不生效

**症状**：改了 `packages/shared/src/schemas/...`，前端 TS 类型没变。

**原因**：`@packages/shared` 是 workspace 包，前端通过 dist 引用。

**解法**：

```bash
# 重新构建 shared
pnpm -F @packages/shared build

# Vite dev 会自动 HMR，Vue 项目可能需要重启
```

## 部署类

### 容器启动后立刻退出

**症状**：

```text
monorepo-server  | Error: connect ECONNREFUSED 127.0.0.1:5432
monorepo-server exited with code 1
```

**原因**：容器内访问的是 `127.0.0.1:5432`（宿主机），但生产编排应该走 docker network 用 `postgres:5432`。

**解法**：

```bash
# 确认 .env.production 的 DATABASE_URL 用服务名
DATABASE_URL=postgresql://...@postgres:5432/...

# 不是 localhost
DATABASE_URL=postgresql://...@localhost:5432/...  # 错误示例
```

---

### 容器启动后健康检查一直失败

**症状**：`docker ps` 显示 `unhealthy`。

**解法**：

```bash
# 1. 看健康检查日志
docker inspect --format='{{json .State.Health}}' monorepo-server

# 2. 手动测健康端点
docker exec monorepo-server wget -q --spider http://localhost:3000/health/liveness

# 3. 看应用日志
docker logs monorepo-server
```

---

### 生产环境 502 Bad Gateway

**症状**：访问 `https://api.example.com` 返回 502。

**可能原因**：

| 原因 | 排查 |
|---|---|
| server 容器没起来 | `docker ps` |
| Nginx / 反代配错 | 看 nginx error log |
| 端口没暴露 | 检查 `docker-compose.prod.yml` 的 `ports` |
| `server` 健康检查失败 | `/health/liveness` 返回非 200 |

## 性能 / 缓存类

### GraphQL 查询慢

**症状**：列表查询要 3-5 秒。

**解法**：

1. **看 SQL**：用 `pnpm -F @apps/server db:studio` 看实际 SQL
2. **N+1**：用 DataLoader 批量化（参考 `apps/server/src/common/dataloader/`）
3. **加索引**：在 Prisma schema 加 `@@index([<field>])`，跑迁移
4. **Redis 缓存**：参考 `apps/server/src/common/cache/`

---

### Redis 连接失败

**症状**：

```text
Error: connect ECONNREFUSED 127.0.0.1:6379
```

**解法**：

```bash
# 1. 确认 Redis 容器在跑
docker compose ps redis

# 2. 手动 ping
docker exec monorepo-redis redis-cli ping
# 应该返回 PONG

# 3. 确认 REDIS_URL
grep REDIS_URL apps/server/.env
```

> Redis 挂了**不会**让应用崩，会自动降级到内存缓存（见 `RedisDegradationService`）。所以这种错只影响性能，不影响功能。

---

### 内存占用持续上涨

**症状**：`docker stats` 显示 server 内存从 500MB 涨到 1.5GB 不下降。

**可能原因**：

- 内存缓存无界增长
- Prisma 连接池泄漏
- 某个长任务持有大对象

**排查**：

```bash
# 1. 看 Node 堆
docker exec monorepo-server node -e "console.log(process.memoryUsage())"

# 2. 抓堆快照
docker exec monorepo-server wget http://localhost:3000/debug/heap  # 如果开启了
```

> 应用层 `enableShutdownHooks` 已经处理正常退出。**`docker kill` 不会触发优雅关闭**——用 `docker stop` 或 `docker compose down`。

## 仍然解决不了？

2. 翻仓库根 `docs/`（开发文档）—— 里面可能有设计原理
