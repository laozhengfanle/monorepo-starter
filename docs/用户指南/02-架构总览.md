# 02 架构总览

> **目标**：5 分钟看懂 MonoKit 的模块切分、调用关系、扩展点。
> 本文不讲"为什么这样设计"，只讲"它是什么"。

---

## 1. 顶层目录

```text
monorepo/
├── apps/
│   ├── server/      NestJS 11 后端（GraphQL + REST）
│   ├── admin/       Vue 3 管理后台 SPA
│   └── web/         Vue 3 C 端 SPA
├── packages/
│   ├── shared/      共享 Zod schema + 工具函数（前后端共用）
│   └── config/      共享 ESLint / TS 配置
├── docker-compose.yml          开发环境：Postgres + Redis
├── docker-compose.prod.yml     生产环境编排
└── apps/*/Dockerfile           每个应用独立的多阶段构建
```

pnpm workspace 统一管理依赖，**所有 apps 共享 `@packages/shared`**（Zod schema 与工具函数）。

---

## 2. 三端调用关系

```text
┌────────────┐   GraphQL     ┌────────────────────┐
│  Admin SPA │ ────REST────▶ │                    │
│  :5173     │               │   NestJS Server    │   PostgreSQL 16
└────────────┘               │     :3000          │ ────────────▶
                             │                    │   Redis 7
┌────────────┐   GraphQL     │  - GraphQL（95%）  │ ────────────▶
│  Web SPA   │ ────REST────▶ │  - REST（5% 认证） │
│  :5174     │               │                    │
└────────────┘               └────────────────────┘
```

| 端 | 主要职责 |
|---|---|
| `apps/server` | 业务逻辑、数据库、缓存、认证、权限、上传、监控 |
| `apps/admin` | 给运营/管理员用的后台：仪表盘、IAM、系统设置、审计 |
| `apps/web` | 给最终用户用的 C 端：登录、个人中心、会员内容 |

## 3. 后端模块切分

`apps/server/src/` 下的目录按角色划分：

```text
src/
├── main.ts                 启动入口（Helmet / CORS / CSRF / 压缩 / RequestID）
├── app.module.ts           根模块，imports 全部基础模块
│
├── common/                 跨业务共享的横切能力
│   ├── auth/               JWT 守卫、@Public() 装饰器、@Permission() 装饰器
│   ├── cache/              Redis 缓存服务（含内存降级）
│   ├── audit/              审计日志批量写入
│   ├── config/             配置域（database / auth / redis / storage）
│   ├── cors/               CORS origin callback
│   ├── dataloader/         N+1 防护：role / menu / permission
│   ├── decorators/         @Public() / @Permission() / @LoginOnly() / @RequireAuth()
│   ├── email/              邮件发送（mock / 第三方 provider 可扩展）
│   ├── errors/             业务错误码常量
│   ├── filters/            全局异常过滤器（REST + GraphQL 统一）
│   ├── guards/             JwtAuthGuard / AdminPermissionGuard / ThrottlerGuard
│   ├── health/             /health 探活
│   ├── interceptors/       LoggingInterceptor / DateFormatInterceptor
│   ├── logger/             nestjs-pino 配置（dev 彩色，prod JSON）
│   ├── menu/               菜单路径校验
│   ├── middleware/         CSRF（Double Submit Cookie）
│   ├── oauth/              微信 / Apple 第三方登录
│   ├── pipes/              Zod 校验管道
│   ├── prisma/             Prisma 客户端 + 扩展
│   ├── redis/              Redis 分布式锁
│   ├── scalars/            GraphQL JSON scalar
│   ├── services/           TokenBlacklist / RedisDegradation
│   ├── sms/                短信发送（阿里云 / mock）
│   ├── storage/            文件存储（本地 / S3 可扩展）
│   ├── utils/              crypto / 错误格式化 / 菜单树构建
│   └── websocket/          WS 鉴权
│
├── modules/                业务模块（按领域切分）
│   ├── auth/               登录 / 刷新 / 登出 / 改密
│   ├── account/            账户基础 CRUD
│   ├── admin/              管理端 IAM：管理员 / 角色 / 菜单 / 权限
│   ├── member/             C 端 IAM：C 端用户 / 等级 / 菜单
│   ├── dashboard/          仪表盘数据
│   ├── upload/             文件上传
│   ├── docs/               内置文档展示（/api/docs）
│   └── graphql/            GraphQL 公共类型（分页 / cursor）
│
├── tasks/                  定时任务（@nestjs/schedule）
│   ├── cleanup.task.ts     清理过期 token / 上传文件 / 验证码
│   └── monitor.task.ts     健康巡检
│
├── metrics/                Prometheus 指标（HTTP / GraphQL / DB / 业务）
└── bff/                    BFF 层（admin / public REST 控制器）
```

**约定**：

- `common/` 里的东西 **是基础设施**，fork 后基本不动
- `modules/<domain>/` 是 **业务模块**，fork 后你可以加、删、改
- `bff/` 是 **REST 入口**，通常配合 `modules/<domain>/` 一起用

## 4. 前端模块切分

以 `apps/admin/src/` 为例（C 端 `apps/web/src/` 结构相似）：

```text
src/
├── main.ts                 启动入口（Pinia / Naive UI / Router）
├── App.vue
│
├── app/                    框架层（不要改）
│   ├── layouts/            MainLayout / GuestLayout / DocLayout
│   ├── components/         AppRouterView / AppSettings
│   ├── composables/        useMenuTree
│   ├── exceptions/         403 / 404 / 500 页
│   ├── redirect/           登录后重定向
│   └── router/             路由 + 守卫 + 菜单→路由转换
│
├── features/               业务页面（fork 后主要改这里）
│   ├── account/            个人中心 / 账户设置
│   ├── dashboard/          仪表盘
│   ├── docs/               内置文档
│   ├── iam/                IAM：admins / roles / menus
│   ├── login/              登录页
│   └── system/             系统设置：缓存 / 日志 / 短信 / 邮件 / 存储 / Turnstile
│
├── shared/                 跨页面复用（fork 后基本不动）
│   ├── components/         EmptyState / ErrorBoundary / FieldRulePopover / SearchGrid / Skeleton
│   ├── composables/        useFormSubmit / useMessage / usePermission / useTurnstile
│   ├── directives/         v-permission
│   ├── request/            graphql-client / csrf / auth-refresh
│   ├── stores/             admin / permission / settings / tabBar
│   └── utils/              color / icon-resolver / zod-form-rules
│
└── api/                    GraphQL / REST 客户端封装
    ├── auth.ts
    ├── accounts.ts
    ├── menus.ts
    ├── roles.ts
    └── ...
```

**约定**：

- `app/` 和 `shared/` 是 **框架层**，不要改（升级基座时会同步覆盖）
- `features/<domain>/` 是 **业务层**，可以加、删、改
- `api/` 是 **API 客户端层**，按业务模块一个文件

## 5. 数据流（一次典型请求）

以"管理员列表查询"为例：

```text
1. 浏览器打开 /admins
   └─→ Vue Router 守卫（permission.ts）→ 检查 meta.permissions
2. AdminsPage.vue 挂载
   └─→ useQuery 调 GraphQL
3. graphql-client.ts
   └─→ fetch /graphql（带 accessToken cookie + CSRF 头）
4. apps/server 收到请求
   └─→ GqlAwareThrottlerGuard（限流检查）
   └─→ JwtAuthGuard（校验 accessToken）
   └─→ AdminPermissionGuard（检查 'admin:list' 权限码）
   └─→ AdminAccountResolver（路由到方法）
   └─→ AdminAccountService.list(input)
       └─→ Prisma 查询 PostgreSQL
       └─→ AdminPermissionCacheService（Redis 缓存读）
   └─→ DateFormatInterceptor（Date 转字符串）
5. 响应回到浏览器
   └─→ DateFormatInterceptor / Date 反序列化
   └─→ AdminsPage 渲染
```

## 6. 扩展点（fork 后要改的位置）

按 **改动频率** 排序：

| 改动类型 | 改哪里 | 注意事项 |
|---|---|---|
| 加一个业务模块 | `apps/server/src/modules/<新领域>/` + `apps/admin/src/features/<新领域>/` | 见 [05-扩展指南.md](./05-扩展指南.md) |
| 改数据库表 | `apps/server/prisma/schema.prisma` | 必跑 `pnpm -F @apps/server db:migrate:create` |
| 加环境变量 | `apps/server/src/common/config/<新域>.config.ts` + `.env.example` | 见 [04-配置参考.md](./04-配置参考.md) |
| 改认证逻辑 | `apps/server/src/common/auth/` | 高频改动区，先读 [03-模块参考.md#auth](./03-模块参考.md) |
| 改权限码 | `apps/server/prisma/seed.ts`（菜单 permissionCode）+ 装饰器 | 注意 seed 是幂等的，可重复跑 |
| 加第三方登录 | `apps/server/src/common/oauth/providers/` | 复制 wechat-web.provider.ts 改 |
| 加短信 provider | `apps/server/src/common/sms/providers/` | 复制 mock.provider.ts 改 |
| 加邮件 provider | `apps/server/src/common/email/providers/` | 复制 mock.provider.ts 改 |
| 加文件存储 | `apps/server/src/common/storage/` | 实现 `StorageProvider` 接口 |
| 升级基座 | 整体 rebase / merge | 见 [05-扩展指南.md#升级基座](./05-扩展指南.md) |

## 7. 不应改的位置（"基础设施"区）

| 区域 | 为什么不能改 |
|---|---|
| `apps/server/src/main.ts` 启动序列 | 改了就破坏 Helmet / CSRF / CORS / 优雅关闭 |
| `apps/server/src/app.module.ts` 的 imports 顺序 | 改了就破坏 Logger / Config / Throttler 的初始化顺序 |
| `apps/server/src/common/middleware/csrf.middleware.ts` | 改了就破坏 CSRF 防护 |
| `apps/admin/src/app/router/guard/` 三个守卫文件 | 改了就破坏权限 / 登录 / 动态路由 |
| `apps/admin/src/shared/request/graphql-client.ts` | 改了就破坏 CSRF / 401 自动刷新 / 5xx 重试 |
| `apps/server/prisma/migrations/` 历史迁移 | 改了就破坏已有数据库，**永远不要改** |

> **升级基座时怎么改基础设施** —— 见 [05-扩展指南.md](./05-扩展指南.md)。
